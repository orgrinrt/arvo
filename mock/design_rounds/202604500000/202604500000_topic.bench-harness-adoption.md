# Current state

The mockspace v2 bench harness shipped via PR #2 and PR #3 in
`hiisi-digital/mockspace`, with self-bench validation completed at
mockspace `dev` commit `2d76e63`. Three crates form the harness:

- `mockspace-bench-core` carries the `Routine` trait, `RoutineBridge`,
  the `routine_bridge!` macro, and the `timed!` measurement primitive.
- `mockspace-bench-harness` carries the orchestrator, worker
  subprocess loop, dlopen layer, CSV cache, validation, analysis,
  report generation, and `meta_report` aggregation.
- `mockspace-bench-macro` carries `#[bench_variant(Algo, "name",
  sizes = [...])]` which expands the variant cdylib's FFI surface
  (`bench_entry`, `bench_name`, `bench_abi_hash`).

The mockspace self-bench at `mockspace/benches/` validated the entire
pipeline against real consumer code: orchestrator, worker re-exec,
dlopen, CSV cache, findings.md, and `--report-only` regen. No bugs
surfaced in the harness during validation. Reviewer findings (cross-
platform `DLL_PREFIX` / `DLL_SUFFIX` path shaping, `current_dir`
unwrap, profile-comment hygiene) all addressed before merge.

arvo's `mock/Cargo.toml` currently pins mockspace at
`branch = "main"`. The bench harness is on `dev` and has not yet
been promoted to `main`. arvo has no bench scaffolding today; the
nearest reference is the polka-dots `mock/benches/` from which the
mockspace harness was originally ported.

arvo-hash currently exports both algorithms needed for the bench:

- `Fnv1a<const N: u16>` (file `mock/crates/arvo-hash/src/fnv1a.rs`)
- `XxHash3<const N: u16>` (file `mock/crates/arvo-hash/src/xxhash3.rs`)

Both implement `Hasher<N>` (file `algo.rs`) where `N` is the *output*
bit-width, with the canonical `pub type ContentHash = Bits<64, Hot>`
defined in `aliases.rs`. Output is `Bits<N, Hot>`. Round D
(landed at `1abc0e0`) widened ContentHash and stabilised the hash
trait family, so the surface is settled.

# Design decisions

## Routine shape: const-generic input size

`Fnv1aVsXxHash3<const N: usize>` parameterised by input byte length.
`Input = [u8; N]`, `Output = u64` (the hash digest). The bench-macro
requires the routine to have a const generic parameter, and matches
the size dispatch on it. This shape lets one Routine express all
four input sizes via the macro's `sizes = [...]` arg.

`build_input(seed)` fills the byte array with seed-driven content.
A simple SplitMix64 mixer over chunks of 8 bytes gives reproducible
inputs that hash to non-trivial digests at each input size.

`ops_per_call(input) = N as u64`, exposing input bytes as the work
unit so the harness can normalise to bytes-per-second style metrics
in addition to wall time.

## outputs_may_differ = true

FNV1a and xxHash3 are different hash algorithms. For the same input
they produce different `u64` digests. The harness's cross-variant
validation has two modes:

- `outputs_may_differ = false`: every variant is expected to produce
  the same byte-exact output for the same input. Used when variants
  are alternative implementations of the *same* function (e.g. SIMD
  vs scalar of the same algorithm).
- `outputs_may_differ = true`: variants are independent algorithms
  evaluated independently. Validation only confirms each variant is
  internally deterministic across runs and stable per-seed; it does
  not compare digests across variants.

The arvo bench measures perf delta between two distinct algorithms,
not implementation parity. Set `outputs_may_differ = true`. This is
the framework's intended pattern for cross-algorithm comparison and
matches the mockspace self-bench (which compares two distinct mixers
under the same flag).

## ContentHash output width = 64 bits

`pub type ContentHash = Bits<64, Hot>` per arvo-hash's aliases. The
bench fixes the output at u64 (the unwrapped representation of
`Bits<64, Hot>`). This matches arvo-hash's headline use case and
keeps the bench focused on the hot path consumers will exercise.
Other widths (128, 256) can be added later as additional benches if
a workload reveals a tradeoff worth measuring.

## Workspace member expansion

`mock/benches/` becomes a workspace member alongside the existing
arvo crates, with the two variant cdylibs as nested members. This
matches the mockspace self-bench layout. Cargo workspace members are
the only place `[profile.release]` actually applies; arvo's existing
profile block at the workspace root continues to govern.

## Mockspace dep branch bump (main → dev)

Required because the bench-harness crates only exist on `dev`.
Periodic `dev` → `main` releases will eventually retire this need;
until that release lands, every consumer that adopts the bench
harness has to use `branch = "dev"`. Tracked separately if a release
is needed sooner.

## #[bench_variant] over manual extern "C"

The mockspace self-bench's `HashMix` Routine has no const generic
and uses manual extern "C" exports. The arvo bench has a const
generic input-size N and uses the `#[bench_variant(...)]` macro,
which is the framework's intended ergonomic surface. This validates
the macro path against a real consumer (parallel benefit to Phase C
which validated the manual path).

## Routine lives in the bench bin, not in arvo-hash

The Routine type is bench-bin-internal. It does not become part of
arvo-hash's public API. arvo-hash exports the algorithms; the bench
binary defines a Routine that uses those algorithms. Keeps arvo-hash
free of bench-framework dependencies.

# Out of scope

- Multiple output widths (128, 256). Initial bench fixes at 64 bits.
- Hash variants beyond FNV1a and xxHash3. arvo-hash only ships these
  two today; future algorithms get added benches as they land.
- Sized routine over the *output* bit-width. The bench parameterises
  on input length, not output length.
- Inline-bench / disasm / perf / history sensors from the v3 follow-
  up list (#281). Initial adoption only validates the orchestrator +
  worker + dlopen + CSV + findings.md path.
- `cargo bench` integration as a Cargo command. Bench runs via
  `cargo run --release --bin arvo-benches --manifest-path mock/
  benches/Cargo.toml`. A future iteration may wire `cargo bench`
  through the harness; not this round.
- A bench README under `mock/benches/`. The mockspace self-bench
  has none either; the bin's module doc comment carries the usage
  context.

# Risks

- mockspace `dev` may move under arvo's feet. Mitigation: pin to a
  specific `rev` if churn becomes a problem; for now `branch = "dev"`
  is the right granularity given how recently the harness shipped.
- The bench-macro's const-generic dispatch may surface dispatch
  arms the framework treats differently than the mockspace self-
  bench's manual exports. Mitigation: the macro is shipped + tested
  in mockspace; if the dispatch reveals a divergence, capture in
  v3 follow-up (#281).
- Variant validation may fail if `build_input` produces inputs that
  pathologically alias one algorithm's hash distribution. Mitigation:
  SplitMix64-style input synthesis is content-uniform; both
  algorithms are well-distributed per their published behaviour.
- Path-shaping (DLL_PREFIX/DLL_SUFFIX) is currently per-consumer
  boilerplate. Tracked as a v3 follow-up under #281.
