## Purpose

Three small corrections before arvo-spectral impl:

1. **Matrix format.** DESIGN says "Depends on arvo-sparse for matrix representations" but the function signatures use dense `&[[W; N]; N]` directly. Spectral methods operate on small dense matrices (typical N ≤ 32) — dense is the natural format. Clarify that arvo-sparse is NOT a runtime dependency for this crate; the `Cargo.toml` drops it. Const-generic dense arrays are the input/output shape.

2. **Const-generic parameter type.** Use `const N: usize` (aligning with arvo-graph / arvo-sparse / arvo-comb choices).

3. **k-way partitioning shape.** Current DESIGN hand-waves "recursive bisection". Concretize: `k_way_partition<const N: usize, const K: usize, F>(weights, ...) -> [USize; N]` returning partition-id per node, with `K` ≤ N as the caller's desired partition count. Actual partition count may be less than K if the graph doesn't admit that many clean cuts.

## Per-crate doc changes

### arvo-spectral — `mock/crates/arvo-spectral/DESIGN.md.tmpl`

- Update the opening to say "operates on dense weighted matrices; does NOT depend on arvo-sparse at runtime (small-N spectral methods fit in `[[W; N]; N]`)".

- Add a concrete `k_way_partition<const N: usize, const K: usize, F>` signature (see impl CL for the exact shape).
- Add `spectral_bisection<const N: usize, F>(fiedler: &[F; N]) -> (Mask64, Mask64)` — returns (positive_side, negative_side) masks. Requires N ≤ 64 this round.
