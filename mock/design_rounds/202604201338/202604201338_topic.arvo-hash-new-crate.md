# Topic: Bits<N> in arvo-bits + ContentHash alias in new arvo-hash crate

**Date:** 2026-04-20
**Status:** TOPIC phase; key scope adjustment recorded 2026-04-20 (see Decisions)
**Task:** #116
**Round scope:** Two crates this round:
1. **arvo-bits** — add generic opaque `Bits<const N: u8>` as a new bit-primitive type. Extends arvo-bits' identity from "contracts only" to "contracts + opaque-bit concretes (non-arithmetic)".
2. **arvo-hash** — new L2 crate. Ships `ContentHash = arvo_bits::Bits<28>` alias + scaffolding. Algorithm surface (FNV / xxHash / CRC32) deferred.

**Blocks:** #72 (hilavitkutin persistence sweep imports `arvo_hash::ContentHash`).

## Purpose

Hashes — content hashes, name hashes, identity hashes — are numeric algorithms on opaque bit-patterns. The consumer side (`hilavitkutin-persistence`, `hilavitkutin-str`, future `clause-*`) currently uses bare `u32` for these values, which the stack's primitive-vocabulary discipline rejects.

This round creates arvo-hash as the home for those types. Arvo is the numeric substrate; a hash-bits container is a numeric primitive in the same sense `Bool` is. Belonging in arvo keeps the types cross-cutting (every downstream consumer can depend on them) without privileging any one consumer.

## Current state

### What arvo ships today (L0–L3)

- **L0 `arvo`** — `UFixed<I, F, S>`, `IFixed<I, F, S>`, `FastFloat` / `StrictFloat`, strategy markers (`Hot` / `Warm` / `Cold` / `Precise`), newtypes (`IBits`, `FBits`, `USize`, `Cap`, `Bool`), `traits::{Abs, FromConstant, Recip, Sqrt, TotalOrd}`. No hash types.
- **L1 `arvo-bits`** — contracts only: `BitWidth`, `BitAccess`, `BitSequence`, `BitPrim` / `IBitPrim` / `UBitContainer` / `IBitContainer`, semantic aliases `Bit`, `Nibble`, `Byte`, `Word`, `DWord`, `QWord`. No fielded structs.
- **L2 `arvo-bitmask`** — `Mask64`, `Mask256`, `BitMatrix`, bit scanning. L2 concrete storage over arvo-bits contracts. Stylistic precedent for where an L2 bit-family sits.
- **L2 `arvo-graph`, `arvo-sparse`, `arvo-comb`** — algorithm crates, no shared bit-identity types.
- **L3 `arvo-spectral`** — depends on L0 + L2 (bitmask).

### What consumers need

From the #72 hilavitkutin-persistence audit (file parked at `hilavitkutin/mock/design_rounds/202604201319/...`):

- `StringTableEntry::content_hash: u32` — 28-bit FNV content hash.
- `ColumnMeta::name_hash: u32`, `TableMeta::name_hash: u32` — same 28-bit FNV domain.
- `archive_str::evict_str(...) -> u32`, `archive_str::inject_str(content_hash: u32, ...)` — transit the hash bit-form.
- `StringTable::lookup(&self, content_hash: u32) -> Option<&[u8]>`.
- `hilavitkutin-str` internally owns `Str::ID_MASK = 0x0FFF_FFFF` (28 bits) and `const_fnv1a(s: &str) -> u64` → cast to u32. The FNV algorithm is already implemented there; arvo-hash does NOT need to re-implement it this round (see Q3).

Downstream (clause) will want the same type family for string interning and, likely, pass-key identity in codegen. Living in arvo means all consumers share one vocabulary.

### Arvo constraints that bind the new crate

From `arvo/.claude/CLAUDE.md` and the `arvo-*` forbidden-imports rules:

- `#![no_std]`, no alloc, no platform dep.
- Sizes are const.
- No `dyn`, `TypeId`, `std::any`.
- Dependency direction: L0 → L1 → L2 → L3. Never back. arvo-hash may depend on `arvo` and `arvo-bits`; may NOT depend on other L2 crates (graph, sparse, comb, bitmask).
- `arvo-bits` is contracts only. arvo-hash consumes arvo-bits contracts; it does not redeclare them.
- Strategy rule: `S: Strategy` required when the type has precision/throughput tradeoff. Hash bits DO NOT — see Q2.
- `strategy-marker-required` lint (#109 shipped) — requires Strategy on pub numeric types. Interpretation check required during doc CL: does a non-arithmetic hash container count? Precedent: `Bool` in L0 is pub, non-arithmetic, no Strategy — no lint fire. Same shape expected here.

## Open questions (provisional decisions recorded below; override via AskUserQuestion response)

### Q1 — Generic `Bits<const N: u8>` OR just concrete `ContentHash(u32)` this round?

**Decision (user, 2026-04-20):** `Bits<const N: u8>` lives in **arvo-bits** (not in arvo-hash); `arvo-hash` defines `pub type ContentHash = arvo_bits::Bits<28>` and depends on arvo-bits. This means:

- arvo-bits gains a fielded struct (`Bits<N>`). Its current mandate ("contracts only, no fielded structs") adjusts to **"bit-level contracts + opaque-bit concrete containers (non-arithmetic)"**. Arithmetic fielded types (UFixed/IFixed) still belong in L0; masks (Mask64, Mask256) still belong in L2 arvo-bitmask. The new concrete is narrow: opaque non-arithmetic N-bit patterns.
- arvo-hash becomes thin: just the `ContentHash` alias (and its home for future algos). Scope still justifies the crate because of the future algo surface (FNV, xxHash, CRC32) and the semantic domain it carves ("hashes").

Side effects to plan for:

- arvo-bits' `CLAUDE.md.tmpl` + `DESIGN.md.tmpl` rule wording needs updating (from "no fielded structs" to the narrower "arithmetic fielded structs belong elsewhere; opaque-bit concretes are welcome").
- arvo-bits' L1 role summary in root `DESIGN.md.tmpl` adjusts.
- `strategy-marker-required` lint — confirmed non-fire case needs documentation (see Q2).

### Q2 — Strategy marker?

- **A (recommended provisional)** — no Strategy. Hash bits are opaque identities; there is no precision/throughput tradeoff to tune. Precedent: `Bool`, `USize`, `Cap`, `IBits`, `FBits` in L0 are pub non-arithmetic newtypes with no Strategy.
- **B** — carry `S: Strategy` despite the semantic mismatch, for lint conformance. Default to `Warm`; treat as a nominal parameter.
- **C** — other.

**Provisional decision:** A — no Strategy. Check `strategy-marker-required` lint interpretation during doc CL; if the lint fires on non-arithmetic types, the correct move is to refine the lint, not to add a meaningless Strategy parameter.

### Q3 — Hash algorithm scope this round?

**Decision (user, 2026-04-20):** types + **generic algorithm contracts** (no concrete impls). Ship the traits so future algo impls are just "implement a trait"; downstream code parameterises over the algo (generic or associated type) so swapping algos is a type-level change. Concrete algorithms (FNV-1a, xxHash, CRC32) ship in follow-up rounds as trait implementors.

Contracts proposed:

```rust
/// One-shot hash: take bytes, produce an N-bit opaque pattern.
pub trait HashAlgo<const N: u8> {
    fn hash(bytes: &[u8]) -> Bits<N>;
}

/// Streaming hasher: feed bytes incrementally, finalise to an N-bit pattern.
///
/// Useful for inputs that arrive in chunks (file reads, network frames,
/// construction from pieces). Algorithms that naturally support streaming
/// implement this; one-shot-only algorithms implement only `HashAlgo`.
pub trait Hasher<const N: u8> {
    fn update(&mut self, bytes: &[u8]);
    fn finalize(self) -> Bits<N>;
}
```

Why two traits rather than one: `HashAlgo<N>` is const-friendly and matches the hilavitkutin-str `const_fnv1a` shape; `Hasher<N>` handles streaming inputs without forcing buffering. A future algo can implement one or both depending on its nature. A blanket `HashAlgo<N>` impl for `Hasher<N>: Default` is *possible* but we're not shipping it this round — explicit impls per algo avoid the `Default` bound friction.

Out of scope this round: concrete algos, blanket impls, const trait methods (feature unstable at writing), derive macros.

### Q4 — Which arvo-bits contracts does `Bits<N>` implement?

`arvo-bits` offers `BitWidth`, `BitAccess`, `BitSequence` as the standard bit-contract set.

- **A (recommended provisional)** — implement `BitWidth` (returns `N`), `BitAccess` (read individual bits), but NOT `BitSequence` (stream-iteration semantics intended for arithmetic contexts).
- **B** — implement all three. Maximum interop.
- **C** — implement none. Hash bits are opaque; downstream doesn't benefit.
- **D** — other.

**Provisional decision:** A — BitWidth + BitAccess. BitSequence can land in a follow-up if consumers surface need.

### Q5 — Backing storage for `Bits<N>`?

- **A (recommended provisional)** — `u64` backing. Covers 1..=64-bit hashes. Overhead for 28-bit hashes is 4 bytes of zeroes; tolerable for runtime. Disk / rkyv narrowing is a consumer concern.
- **B** — width-adaptive backing via a helper trait (`Bits<28>` backs with `u32`, `Bits<64>` with `u64`). More complex generics.
- **C** — `u128` backing. Overkill for current consumers.
- **D** — other.

**Provisional decision:** A — uniform `u64`. Simplicity wins.

### Q7 — `Bits<N>` bit-width constraint enforcement?

Invalid states: `Bits<0>` (zero bits) and `Bits<65..>` (exceeds u64 backing).

- **A (recommended provisional)** — compile-time guard: `const { assert!(N > 0 && N <= 64) }` inside the constructor. Rust's `generic_const_exprs` (already enabled in arvo) makes this clean.
- **B** — unbounded N with a runtime mask; `Bits<0>` degenerate but allowed.
- **C** — split into concrete `Bits8 / Bits16 / Bits32 / Bits64`; no generic.
- **D** — other.

**Provisional decision:** A — const-asserted.

### Q8 — Round shape: new crate scaffolding + types in ONE round, or two rounds (scaffold first, impl follow-up)?

arvo-bits used "contracts round" then "impl round". arvo-hash is smaller.

- **A (recommended provisional)** — one combined round. Scaffold + types + arvo-bits trait impls together.
- **B** — two rounds.
- **C** — other.

**Provisional decision:** A — one combined round.

### `arvo-bits/src/bits.rs` (new file)

```rust
use crate::traits::{BitAccess, BitWidth};

/// Opaque N-bit pattern. Non-arithmetic; compared by identity only.
///
/// Backing storage is `u64`; only the low N bits are addressable.
/// Invalid at compile time: `N == 0` or `N > 64`.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, PartialOrd, Ord)]
pub struct Bits<const N: u8>(u64);

impl<const N: u8> Bits<N> {
    const _BOUNDS: () = {
        assert!(N > 0, "Bits<0> is invalid; use Bool for 1-bit values");
        assert!(N <= 64, "Bits<N> requires N <= 64");
    };

    const MASK: u64 = if N == 64 { u64::MAX } else { (1u64 << N) - 1 };

    pub const fn new(bits: u64) -> Self {
        let _ = Self::_BOUNDS;
        Self(bits & Self::MASK)
    }

    pub const fn bits(self) -> u64 { self.0 }
}

impl<const N: u8> BitWidth for Bits<N> {
    const BITS: usize = N as usize;
}

impl<const N: u8> BitAccess for Bits<N> {
    fn bit(&self, i: usize) -> bool {
        (self.0 >> i) & 1 != 0
    }
}
```

### `arvo-hash/src/algo.rs`

```rust
use arvo_bits::Bits;

/// One-shot hash: consume bytes, produce an N-bit opaque pattern.
///
/// Algorithms that can compute a hash from a whole byte slice in
/// one pass implement this. Downstream code that wants to be
/// algorithm-agnostic takes `<H: HashAlgo<N>>`.
pub trait HashAlgo<const N: u8> {
    fn hash(bytes: &[u8]) -> Bits<N>;
}

/// Streaming hasher: feed bytes incrementally, finalise to an
/// N-bit pattern. For inputs that arrive in chunks or that cannot
/// be materialised as a single slice.
///
/// Algorithms that support streaming implement this; one-shot-only
/// algorithms implement only `HashAlgo<N>`.
pub trait Hasher<const N: u8> {
    fn update(&mut self, bytes: &[u8]);
    fn finalize(self) -> Bits<N>;
}
```
