## Deliverables

Rust source under `mock/crates/arvo-bitmask/src/` implementing:

- `NodeId(pub USize)` — `#[repr(transparent)]`, derives `Copy`/`Clone`/`Eq`/`Ord`/`Debug`/`Hash`.
- `Mask<W>` — generic bitmask over `W: BitSequence + BitAccess + Copy + Default`. Const-fn where the trait methods allow.
- Concrete type aliases: `Mask64 = Mask<QWord<Hot>>`, `Mask256 = Mask256` (distinct struct wrapping `[QWord<Hot>; 4]` — can't be a `Mask<[_; 4]>` because arrays don't impl arvo-bits contracts).
- Set ops on `Mask64` and `Mask256`: `union`, `intersection`, `difference`, `complement`, `is_empty`, `intersects`, `contains`, `insert`, `remove`, `count`.
- Bit scanning on both: `lowest_set`, `highest_set`, `iter_set_bits` (impl Iterator<Item = USize>, loop-free for Mask64, unrolled over 4 words for Mask256).
- `BitMatrix64<const N: usize>` and `BitMatrix256<const N: usize>` — adjacency matrices: `[Mask64; N]` and `[Mask256; N]`. Methods `edge`, `set_edge`, `clear_edge`, `successors`, `predecessors`, `transitive_closure` (Warshall on bitmasks).
- `propagate_dirty` free function generic over matrix + mask pair for dirty propagation.
- Unit tests: set-op correctness, scan ordering, BitMatrix adjacency semantics, transitive closure on known-shape DAGs, propagate_dirty correctness.

Out of scope:

- `Mask<W>` over types other than `QWord<Hot>` (consumer can instantiate; we test only the two shipping aliases).
- Non-hardcoded BitMatrix dispatch (no runtime choice between Mask64/Mask256 — pick the type at build time).
- Algorithm crates.

## Impl-time decisions (pre-committed)

1. **Mask256 is its own struct, not `Mask<[QWord<Hot>; 4]>`.** Arrays don't impl arvo-bits contracts, so a generic-over-container approach breaks for multi-word masks. `Mask256(pub [QWord<Hot>; 4])` with inherent impl methods mirrors the Mask64 surface. If future needs push a clean unification, promote to a trait-based abstraction in a follow-up round.

2. **`NodeId::new(u: USize)`** as a `const fn` constructor. `NodeId(pub USize)` stays public-field for ergonomics. No `Deref<Target = usize>` — node IDs should not silently decay to plain indices.

3. **Set ops are inherent methods on `Mask64` / `Mask256`, not blanket impls on `Mask<W>`.** Rationale: same const-expr-cycle issue that forced the bridge-trait pattern in arvo-bits. Keep it flat.

4. **`iter_set_bits`** returns an `IntoIterator` adapter struct (not `impl Iterator<Item = USize>` return-position) so the iterator is nameable for tests. Struct: `SetBitsIter<M> { remaining: M }`. For Mask64: advance via `trailing_zeros` + clear lowest bit. For Mask256: advance word-by-word, `trailing_zeros` inside each non-zero word.

5. **`BitMatrix64<const N: usize>`** stores as `[Mask64; N]`. Const-generic `N: usize` (not `Cap`) for simplicity — `Cap` is an arvo wrapper; const-generic newtype parameters on `[_; N]` arrays require `N: usize`. If the design needs the `Cap` type in signatures, a wrapper method can convert at the boundary.

6. **`transitive_closure`** via Warshall. Nested `for k`, `for i` loops. For each `k`, for each `i` where `row[i].contains(k)`, `row[i] |= row[k]`. Bitmask union is one instruction per `u64` word.

7. **`propagate_dirty`** accepts `&BitMatrix64<N>` + `&mut Mask64` (and a mirror for `BitMatrix256` + `Mask256`). Algorithm: for each set bit `i` in `dirty`, `dirty |= matrix.successors(i)`. Fix-point loop until no change.
