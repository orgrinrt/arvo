## Why this round

PR #35 carries Rounds A through D consolidated for the audit-driven follow-up
chain (#259). Pre-merge local review surfaced four small follow-ups that
sit above the threshold for "fold into the next routine round" and below the
threshold for "block the PR". The reviewer's report says merge after fixes; the
local-pr-review-flow rule says fixes land in the same PR.

The findings, in priority order:

1. `mock/crates/arvo/DESIGN.md.tmpl` carries a `Bits<const N: u8>` claim and a
   "0..=255 substrate cap" framing left over from before Pass F widened the
   const-generic to u16. Pass F shipped in Round D and updated source; the
   canonical DESIGN doc was not refreshed. Drift between the design source of
   truth and shipped source is exactly what the design-doc-source-mismatch
   lint guards against, and the assertion is plainly false post-Pass-F.

2. `mock/crates/arvo-storage/src/layout_assertions.rs` contains five em-dash
   tokens in section-divider comments inside the Tier 2 Bits projection
   block. Workspace `writing-style.md` is hard-zero on em-dashes in any
   written content including inline comments. The dividers slipped past the
   pre-commit gate when Round D-aux landed; clean them up here.

3. `mock/crates/arvo-storage/Cargo.toml` does not declare the
   `arvo_full_layout_audit` cfg name, so every `cargo check` emits an
   `unexpected_cfgs` warning at `layout_assertions.rs` line 292. The cfg name
   is the runtime-shape "audit-only" framing introduced by Pass A.2 and is
   kept (not converted to a feature flag) deliberately. Register the cfg via
   `[lints.rust]` `unexpected_cfgs = { check-cfg = ... }` so the warning
   stops firing without losing the runtime-shape design.

4. `mock/crates/arvo-narrow/src/lib.rs` declares
   `#![feature(adt_const_params)]` / `#![feature(const_trait_impl)]` /
   `#![feature(const_param_ty_trait)]`. Pre-Round-D Pass B.4 these were
   load-bearing for the cross-primitive Narrow impls hosted in the crate.
   Post-Pass-B.4 the Narrow impls relocated to `arvo-bits-contracts` (orphan
   rules forced the move), and arvo-narrow is now a pure re-export facade.
   The nightly-feature declarations are dead. Remove them; cargo's
   `unused_features` warning has flagged them on every workspace check
   since Round D close.

## Decisions

- **Decision 1**: keep `cfg(arvo_full_layout_audit)` as a cfg, not a cargo
  feature. Reason: cfg matches the "build-shape" semantics (this audit only
  runs in CI nightly with `RUSTFLAGS=--cfg arvo_full_layout_audit`); a cargo
  feature would surface in Cargo.toml feature dependency graphs and imply a
  semver-affecting flag, which it is not. The check-cfg declaration silences
  the warning while preserving the cfg's semantic.

- **Decision 2**: do not promote the dead `arvo-narrow` empty-crate state
  here. The `[no-empty-crate]` lint is firing as warn; whether to delete the
  crate, repurpose it, or keep it as a thin re-export facade is a design
  decision that needs its own round (the orphan-rules history matters and
  consumers may currently `use arvo_narrow::Narrow;`). Leave the empty-crate
  warning standing; remove only the unused nightly-feature decls.

- **Decision 3**: the BACKLOG.md.tmpl Round 202604280500 forward-looking
  section's "stays at u8" line (NIT 4 in the report) gets a clarifying
  parenthetical so future readers don't read it as current state. Same
  source-of-drift class as Finding 1, smaller scale, fits this round.
