## Decisions captured before the round opens

### Sign axis on Bits (approach iii)

Approach iii from the prior session's three-option discussion: a const-generic Sign parameter on Bits with ZST markers, defaulting to `Unsigned` so existing call sites keep their current behaviour. Distinct from approach i (separate `Bits` and `IBits` types, code duplication) and approach ii (Sign as a phantom marker only, no const-generic, with separate trait paths).

Concrete shape:

```rust
pub trait Signedness: sealed::Sealed + Copy + Clone + Default + 'static {}

#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug, Default)]
pub struct Unsigned;

#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug, Default)]
pub struct Signed;

impl sealed::Sealed for Unsigned {}
impl sealed::Sealed for Signed {}
impl Signedness for Unsigned {}
impl Signedness for Signed {}

#[repr(transparent)]
pub struct Bits<const N: u8, S: Strategy = Hot, Sign: Signedness = Unsigned>(
    <S as BitsContainerFor<N, Sign>>::T,
)
where
    S: BitsContainerFor<N, Sign>;
```

The container projection is sign-aware. Two paths considered, one chosen:

1. Single trait `ContainerFor<N, Sign>` parameterised by Sign internally with cases for (S, Unsigned) and (S, Signed). This collapses the public trait surface to one entry but requires per-(S, Sign, N) impls.
2. Keep `UContainerFor<N>` and `IContainerFor<N>` as today; introduce a `BitsContainerFor<N, Sign>` indirection trait that projects to one or the other based on Sign. This preserves the existing UFixed / IFixed call sites verbatim and keeps the U/I tables independently extensible.

**Decision: option 2.** The user's choice was "split UContainerFor + IContainerFor stay separate; Bits internally selects per Sign axis" (less churn; preserves existing impl tables; UFixed and IFixed continue to bind on UContainerFor / IContainerFor directly). The Sign-axis selection lives inside `Bits`'s where-clause projection.

The Bits where-clause shape:

```rust
pub trait BitsContainerFor<const N: u8, Sign: Signedness>: Strategy {
    type T: Copy + Clone + PartialEq + Eq + ...;
}

impl<S, const N: u8> const BitsContainerFor<N, Unsigned> for S
where S: UContainerFor<N>
{
    type T = <S as UContainerFor<N>>::T;
}

impl<S, const N: u8> const BitsContainerFor<N, Signed> for S
where S: IContainerFor<N>
{
    type T = <S as IContainerFor<N>>::T;
}
```

UFixed and IFixed continue to bind on `UContainerFor<N>` / `IContainerFor<N>` directly (unchanged). The `Bits<N, S, Sign>` declaration uses `BitsContainerFor<N, Sign>` to dispatch to one of the two underlying tables.

### Meta-newtype carrier widening

The current `IBits`, `FBits`, `Width` newtypes wrap bare `u8` (range 0..=255). To support `N = 256`, the carrier widens. Two options considered:

1. Bias offset (range 0..=255 holds value 1..=256, value 0 reserved): saves the bit but adds offset arithmetic at every accessor.
2. Honest carrier widening: inner field becomes `Bits<9, Hot>` (native range 0..=511, value 0..=256 used). 1 bit wasted at the const-generic-position layer.

**Decision: option 2.** The user's framing: "Bits<9, Hot> with honest 0..=511 native range. No offset trickery; 1 wasted const-generic bit costs nothing."

Concrete shape post-Round-B:

```rust
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug, Default)]
#[repr(transparent)]
pub struct IBits(pub Bits<9, Hot>);

pub const fn ibits(n: u16) -> IBits {
    IBits(Bits::from_raw(n))
}
```

`u16` at the helper-fn boundary because `<Hot as UContainerFor<9>>::T = u16`. Consumer call sites continue to write `ibits(13)` (the literal `13` infers as `u16` in the helper-fn argument position).

The `From<u8>` impl on `IBits` / `FBits` / `Width` widens to the new u16-backed shape; existing call sites that pass `u8` still work (`From<u8> for IBits` constructs via `IBits(Bits::from_raw(n as u16))`).

The `meta_bits_wrapper!` macro DRYs all three over the new carrier in one update.

### Multi-value container abstraction (`MultiContainer<HiT, LoT>`)

The 65..=256 bit-width range needs storage. Three sub-decisions:

1. **Single primitive vs multi-primitive split.** `u128` exists natively in Rust; on most targets, codegen lowers `u128` ops to two-`u64`-with-carries sequences anyway. A `MultiContainer<u32, u64>` (96 storage bits) is tighter than `u128` (128 storage bits) for 92-bit values; `MultiContainer<u64, u64>` (128 storage bits) is the same as `u128` but with consumer-controlled half access. Per the user's framing: "what if we want 92bit? 64+32 would be the optimal smallest combo, not 64+64. Needs proper flexibility. Though strategy should hold here too, since two same length will be better for most cases, while smallest footprint bitwise might be better for others."

   **Decision: strategy-driven.** Each (Strategy, N) cell picks the (HiT, LoT) shape per the strategy's footprint policy (smallest-footprint vs same-length). The cell-by-cell tabulation lives in DOC phase.

2. **Strategy projection rules.**
   - Hot: smallest-footprint pair. The (HiT, LoT) split that uses the fewest storage bits >= N. e.g. 92-bit → `MultiContainer<u32, u64>` (96 bits, 4 bytes saved over u128 / u64+u64).
   - Warm: same-length doubled pair. Both halves are the same primitive width, doubled to give a single-op safe-add overhead budget. e.g. 92-bit → `MultiContainer<u64, u64>` (128 bits).
   - Cold: smallest-footprint pair, with bitpacked access on column-store layouts. e.g. 92-bit → `MultiContainer<u32, u64>` (96 bits).
   - Precise: same-length pair, saturating arithmetic. e.g. 92-bit → `MultiContainer<u64, u64>` (128 bits).

   The cell-by-cell tabulation lands in DOC phase. The four projection rules are the framing, not a per-cell list.

3. **Native `u128` vs MultiContainer for 65..=128.** Both are valid containers for 65..=128. The decision per cell:
   - Hot at 65..=96: smallest-footprint dispatches to `MultiContainer<smaller, larger>` (e.g. 65..=80 → `MultiContainer<u16, u64>`; 81..=96 → `MultiContainer<u32, u64>`).
   - Hot at 97..=128: smallest-footprint can dispatch to `MultiContainer<u64, u64>` or `u128` natively. Same storage. Default to `MultiContainer<u64, u64>` for codegen consistency with the >128 path; the option to add `u128` cells if benches show codegen wins lives in BACKLOG.
   - Warm at 65..=128: same-length doubled. The same-length rule projects to `MultiContainer<u64, u64>` for 65..=128 (2x of u64 is u128-equivalent storage, single-half ops fit in u64).
   - Cold at 65..=128: Hot's shape, with column-store bitpacking handled at access sites (separate concern, not container-type).
   - Precise at 65..=128: Warm's shape with saturating ops.

   Per-cell tabulation in DOC CL. Initial default per the always-optimal-internals rule: keep the dispatch consistent across the 65..=256 range; bench-driven u128-native cells in 65..=128 land in a follow-up via BACKLOG.

### IFixed reshape (closes pass 4d)

Today IFixed is `repr(transparent)` directly over `<S as IContainerFor<{ifixed_bits(I, F)}>>::T` with a `PhantomData<fn() -> S>` companion field. Round B reshapes:

```rust
#[repr(transparent)]
pub struct IFixed<const I: IBits, const F: FBits, S: Strategy = Warm>(
    Bits<{ ifixed_bits(I, F) }, S, Signed>,
)
where
    S: IContainerFor<{ ifixed_bits(I, F) }>;
```

Mirrors UFixed's wrap. The `PhantomData<fn() -> S>` companion goes away (`Bits` carries S directly via its own where-clause projection, identical to how UFixed already inherits S through the wrapped Bits today).

`from_raw` / `to_raw` accessors flow through Bits's existing accessors. Marker trait impls (`BitPresentation`, `IntegerLike`, `FractionLike`) keep their current shape; only the field-storage layer changes.

### UContainerFor / IContainerFor table extension

Current tables cover 1..=64 across Hot / Warm / Cold / Precise (with Warm intentionally absent for 33..=64 per doc CL D2). Round B extends the tables to cover 1..=256 across all strategies. The Warm-caps-at-32 design rule from doc CL D2 was the right call when 64-bit was the substrate's max; with 256-bit support landing, the same "2x logical, single-op safe-add" Warm rule extends naturally to 33..=256 via multi-container shapes.

Final per-strategy rules across the full 1..=256 range:

- Hot 1..=64: smallest aligned native primitive (existing).
- Hot 65..=128: smallest-footprint MultiContainer (e.g. 65..=80 → `MultiContainer<u16, u64>`; 81..=96 → `MultiContainer<u32, u64>`; 97..=128 → `MultiContainer<u64, u64>` default).
- Hot 129..=256: smallest-footprint MultiContainer (e.g. 129..=192 → `MultiContainer<u64, u128>`; 193..=256 → `MultiContainer<u128, u128>`).
- Warm 1..=32: 2x logical native primitive (existing).
- Warm 33..=64: 2x logical, single primitive (`u128` for 33..=64). Round B promotes Warm into 33..=64 via `u128`. The previous compile-error-on-Warm-at-33+ shape becomes a compile-success with explicit u128 backing.
- Warm 65..=128: 2x logical doubled MultiContainer (`MultiContainer<u128, u128>`).
- Warm 129..=256: 2x logical via doubled MultiContainer. Ceiling at 256 because `MultiContainer<u128, u128>` is the largest workable double (no native u256).
- Cold 1..=64: same widths as Hot, bitpacking is column-store concern.
- Cold 65..=256: same shape as Hot.
- Precise 1..=32: 2x logical native, saturating (existing).
- Precise 33..=64: 2x logical, single primitive (`u128`), saturating.
- Precise 65..=256: same as Warm with saturating arithmetic.

The Warm extension into 33..=256 is a substrate scope expansion. Per the workspace `arvo-toolbox-not-policer` rule, exposing Warm at wider widths gives consumers the development-friendly default at every width; the previous "Warm caps at 32" diagnostic shifts to a more nuanced "Warm at 65..=128 uses MultiContainer<u128, u128>; consider Hot for tighter footprint" advisory note (still warn-not-error).

### Sign axis ergonomics (consumer-facing)

The default `Sign = Unsigned` keeps every existing UFixed / `Bits<N, S>` / consumer call site unchanged. Domain code that wants signed Bits directly writes `Bits<N, S, Signed>`. IFixed wraps `Bits<{1+I+F}, S, Signed>` internally; consumers continue to write `IFixed<I, F, S>` and the Sign axis is invisible to them.

`Bits<N, S, Sign>` exposes the Sign at the Bits surface to allow signed-bit-pattern domain types (rare; `Bits` is opaque-bit-pattern, not arithmetic, so signed Bits is mostly a placeholder for IFixed's wrap). Consumers normally don't write `Bits<N, S, Signed>` directly; they write IFixed instead.

## Open during DOC phase (will be decided in DOC CL)

Three trait-family-shape and per-cell-tabulation items resolve during DOC:

1. **MultiContainer struct vs Wide<HiT, LoT> alias**: ship `MultiContainer<HiT, LoT>` as a single concrete struct, or define a generic `Wide<HiT, LoT>` and reuse it across strategies. Both are equivalent shape; the choice is naming + visibility. Lean: `MultiContainer<HiT, LoT>` as single struct in `arvo-storage`, alongside the existing `Bits` primitive. Reuse across strategies happens through the type alias / projection, not a generic-over-strategy shape.

2. **UMultiArith trait family vs UArith extension**: arithmetic on multi-value containers is non-trivial (multi-precision add with carry, multi-precision mul with double-width intermediate). Two options:
   - Separate `UMultiArith<N>` / `IMultiArith<N>` trait family for multi-value container arithmetic, kept independent from the existing `UArith<N>` / `IArith<N>`.
   - Extend `UArith<N>` / `IArith<N>` to cover the multi-container case with the same surface.

   Lean: extend UArith / IArith. Same trait surface; the impl bodies for multi-value cases are different but the consumer-facing contract is unchanged. Less surface fragmentation; per the workspace `arvo-toolbox-not-policer` and `arvo-always-optimal-internals` rules, the implementation choice (multi-precision intrinsics, asm microkernel, LLVM intrinsic chain) is internal.

3. **Per-cell (Strategy, N) tabulation for 65..=256**: the precise (HiT, LoT) shape for each of the 4 strategies × 192 widths = 768 cells. Macro-generated per the four strategy projection rules. Tabulation lives in DOC CL `arvo-strategy/DESIGN.md.tmpl` extension; per-cell impl rows generated by macro in SRC.
