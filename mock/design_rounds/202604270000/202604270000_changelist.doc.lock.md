**Date:** 2026-04-20

**Topic:** `202604270000_topic.arvo-bitmask-sweep.md`
**Type:** additive for arvo-bits (new `BitLogic` trait + four-method surface);

## Motivation

Lint scope widened in 2026-04-20 surfaced 41 arvo-bitmask drift sites. Audit
reduced them to 5 cause classes; 34 of the 41 share one root cause — arvo-bits
exposes no whole-word bitwise ops on `QWord<Hot>` (or any `UFixed<I, F, Hot>`
container), so arvo-bitmask reaches through `to_raw`/`from_raw` to core `u64`
to do OR / AND / NOT / XOR, then pays the drift back in `as usize` casts and
raw-typed iterator fields. The topic file resolves: add a `BitLogic` trait to
arvo-bits, concrete-impl it on each container, implement `core::ops::BitOr` /
`BitAnd` / `BitNot` / `BitXor` on the same containers. Classes C/D/E collapse
to zero drift. Classes A (3 core-trait sites) + B (1 nightly-const-eval site)
keep `lint:allow` with task IDs (#115, #121).

Doc surface must reflect this before the src CL describes the rewrite.

## Changes proposed

### 1. `mock/crates/arvo-bits/DESIGN.md.tmpl`

**Add a fourth trait section** after the current `BitSequence` block
(around line 108). Shape:

````markdown
### BitLogic

Whole-word bitwise operations. Single-instruction on the backing
primitive, loop-free:

```rust
pub trait BitLogic: BitWidth + Copy {
    fn bitor(self, other: Self) -> Self;
    fn bitand(self, other: Self) -> Self;
    fn bitnot(self) -> Self;
    fn bitxor(self, other: Self) -> Self;
    fn clear_lowest_set_bit(self) -> Self
    where
        Self: BitAccess + BitSequence,
    { /* default impl using BitSequence::trailing_zeros + BitAccess::with_bit_cleared */ }
}
```

`BitLogic` covers the four bitwise operations that map directly to
hardware instructions (`or` / `and` / `not` / `xor` on x86; identical
on aarch64). `BitSequence` covers SCAN / POPCOUNT; `BitAccess` covers
SINGLE-BIT read/write; `BitLogic` covers WHOLE-WORD logic. The three
are independent contracts that together describe "a bit-bearing
container".

Concrete impls live on each `UFixed<I, F, Hot>` container family
(`Bit`, `Nibble`, `Byte`, `Word`, `DWord`, `QWord`) — each delegates
once via `UFixed::to_raw` / `from_raw` to the backing primitive's
core operator. Other strategies (`Warm`, `Precise`, `Cold`) do not
impl `BitLogic` because their wider containers make whole-word ops
meaningless at the logical-width level — `BitLogic` is a Hot-only
surface.

The `clear_lowest_set_bit` default method is a derived convenience
built on `BitSequence::trailing_zeros` + `BitAccess::with_bit_cleared`.
Concrete types (the `UFixed<I, F, Hot>` containers) override the
default with `self.to_raw() & (self.to_raw() - 1)` (one instruction
via x86 `BLSR`) for the fast path.

In addition to the trait methods, arvo-bits ships `impl core::ops::{BitOr,
BitAnd, BitNot, BitXor} for UFixed<I, F, Hot>` — blanket on `T: BitLogic`
if the orphan rules permit; otherwise per-type — so consumers can use
`a | b`, `a & b`, `!a`, `a ^ b` syntax. Implementing core TRAITS
(contracts) on arvo's own types is distinct from using core CONCRETE
types in signatures — the #115 "freedom from core" direction concerns
the latter, not the former.
````

## Out of scope

- Concrete `BitLogic` impls for `Warm` / `Precise` / `Cold` strategies.
  Whole-word bitwise ops are meaningless above the logical width; Hot-only
  is the contract.
- `#115` redesign of `Iterator` / `PartialOrd` to arvo-native forms.
  Remains a #115 concern, its own design round. This round's `lint:allow`
  annotations track under #115.
- `#121` revisit of `cap_size`. Language-limitation; no action until
  rustc stabilises relevant const-eval features.
- Per-crate scoped lint severity overrides in mockspace.toml. Global
  ramp stays; drops in one commit once all L2/L3 sweeps land.
- arvo-graph / arvo-sparse / arvo-comb / arvo-spectral sweeps — sibling
  rounds under the #120 umbrella. Each is independent; each gets its
  own topic + doc CL + src CL. The BitLogic contract shipped here is
  a force multiplier: those crates may or may not need it (most of
  their drift is per-algorithm), but if they do, it's already there.
