# Topic: arvo-bitmask sweep — eliminate bare primitives from every line

**Date:** 2026-04-20
**Status:** TOPIC phase; decisions pending
**Task:** #120 (first crate of the arvo L2/L3 algorithm-crate sweep)
**Round scope:** `arvo-bitmask` source files (`ops.rs`, `matrix.rs`, `node.rs`,
  `mask.rs`, `dirty.rs`). Possibly touches `arvo-bits` trait surface (see Q1).

**Blocks:** nothing downstream (arvo is the bottom of the stack), but sets the
  pattern every subsequent algorithm-crate sweep follows: arvo-graph, arvo-sparse,
  arvo-comb, arvo-spectral. The treatment of "compiler-mandated bare-primitive
  boundary" cases chosen here is the precedent for all 4 remaining crates.

## Purpose

Lint scope was widened in session 2026-04-20 to scan every `src/**/*.rs`. Prior
scope was `lib.rs` only. The widening surfaced 41 real drift sites in
arvo-bitmask that CI had never caught. The directive is unambiguous: bare `u8..
u128` / `i8..i128` / `f32` / `f64` / `usize` / `isize` / `bool` / `Option` /
`Result` / `String` do not exist in this stack, anywhere, as if they don't
exist. Not in public API, not in private fields, not in internal impl.

This round remediates every arvo-bitmask violation. Sets the precedent for
downstream algorithm-crate sweeps.

detour alive). If Class C stays, these stay.

as `QWord<Hot>`. `word_idx: usize` → `USize` is trivial regardless.

### What arvo-bits already exposes

From `arvo-bits/src/traits.rs`:

- `BitWidth::WIDTH: USize` — const logical bit count.
- `BitAccess::{bit, with_bit_set, with_bit_cleared, with_bit_toggled}` — single-bit ops taking `USize`, returning `Self` / `Bool`.
- `BitSequence::{trailing_zeros, leading_zeros, count_ones, count_zeros, is_zero}` — whole-word scans returning `USize` / `Bool`.

What's MISSING for arvo-bitmask to be fully trait-native:

- Whole-word bitwise ops: `bitor`, `bitand`, `bitnot`, `bitxor`, `difference`
  (or equivalently, `core::ops::BitOr`/etc. impls on `QWord<Hot>`). 
- Optional: `clear_lowest_set_bit(self) -> Self` convenience for the iterator
  advance pattern. Builds on `trailing_zeros` + `with_bit_cleared`.

## Open questions

### Q1 — Scope of the fix for Classes C + D + E

This is the load-bearing decision for the round. All three classes share the
same root cause: arvo-bits doesn't expose whole-word bitwise ops on its
`QWord<Hot>` type, so arvo-bitmask reaches through `to_raw` / `from_raw` to
get at `u64`, and once it's there it also calls core intrinsics and stores
the raw bits in iterator fields.

**Option A — Extend arvo-bits trait surface.** Add whole-word bitwise ops
(`bitor` / `bitand` / `bitnot` / `bitxor`) to `BitSequence` (or a new
`BitLogic` sub-trait) with default impls delegating to the existing
`BitAccess` / `BitSequence` primitives where possible, or as required
methods with concrete impls on each `UFixed<I, F, Hot>` container. Then
`arvo-bitmask::ops` uses trait-native ops throughout. Classes C / D / E
collapse to zero drift — the `qword_to_u64` helpers delete; the `as usize`
casts delete; the iterator fields become `remaining: QWord<Hot>` and
`words: [QWord<Hot>; 4]`. Scope expands to touch arvo-bits + arvo-bitmask
in one round. Sets a strong precedent: arvo-bits IS the place where every
bit-level op contract lives. **This is the consistent, no-shortcut approach.**

**Option B — Narrow sweep: annotate Classes C + D + E with `lint:allow`.** Keep
the `qword_to_u64` bridge and the `as usize` casts, add `lint:allow(...)
reason: ...; tracked: #<new-task>` on each line. Iterator fields stay raw.
New task captures the arvo-bits extension as follow-up. Smaller diff; defers
the real fix. Previous session feedback (`#50` deferred → this whole retrofit)
explicitly flagged this as the wrong move — deferral lets drift compound.

**Option C — Split the scope deliberately.** This round fixes Class C (the
bridge helpers) by extending arvo-bits. Classes D (cast sites) and E
(iterator fields) follow mechanically from C. Classes A + B stay as
`lint:allow` with task IDs. No real difference from A in effect, but
explicit about what's minimum-sweep and what's allow-with-tracking.

**Provisional answer:** A. The session's working rule is "when a rule is
articulated, write the lint/hook in the same round" — converse also holds:
when drift is articulated, fix the root cause in the same round, not the
next one. Arvo-bits is the right home for whole-word bit ops. The scope
expansion is bounded (trait method additions + concrete impls) and stays
within arvo-bits' stated mandate. **Requires user confirmation.**

### Q4 — Final shape of SetBitsIter64 / SetBitsIter256 after Option 1A

Assuming Q1 = Option A, the iterators become:

```rust
pub struct SetBitsIter64 { remaining: QWord<Hot> }

impl Iterator for SetBitsIter64 {
    type Item = USize;
    fn next(&mut self) -> Option<USize> {  // lint:allow, Q2 = A
        if <QWord<Hot> as BitSequence>::is_zero(self.remaining).into() {
            return None;
        }
        let idx = <QWord<Hot> as BitSequence>::trailing_zeros(self.remaining);
        self.remaining = <QWord<Hot> as BitAccess>::with_bit_cleared(self.remaining, idx);
        Some(idx)
    }
}
```

For Mask256: `words: [QWord<Hot>; 4]` + `word_idx: USize`.

Two questions nested:
- **Q4a — `word_idx: USize` internally OK?** `USize(0)`, `USize(1)` during advance. Yes, direct replacement.
- **Q4b — `word_idx` could be compile-time bounded.** Since it's 0..4 only, could be `Cap<4>` or similar. But this is an iterator advance index, natural as `USize`.

**Provisional answer:** Q4a YES, Q4b no — keep as `USize` (simpler).

### Q5 — Naming of the new bit-logic methods/trait

**Option A — Extend existing `BitSequence` with ops.** Traits already named
by concern (width / access / sequence). Bitwise logic is a sequence-level
operation on the bits → fits.

**Option B — New `BitLogic` trait with `BitOr` / `BitAnd` / `BitNot` / `BitXor`.** Separate concern, clean split. Concerns stay minimal.

**Option C — Implement `core::ops::BitOr` / `BitAnd` / etc. on `QWord<Hot>`
and the other `UFixed<I, F, Hot>` containers.** Uses core's operator syntax
directly (`a | b`, `a & b`), most ergonomic at the call site, but re-opens
the "strive free from core" debate. `core::ops` impls are the same class as
`core::Iterator` — tracked under #115 as a broad concern.

**Provisional answer:** B — new `BitLogic` trait. Reason: keeps concerns
explicit (`BitLogic::bitor` reads as "logical OR on bit sequences", vs.
`BitSequence::bitor` which conflates "sequence operations" with "logical
ops"). Opens the door for later `core::ops::BitOr` blanket impls (Option C)
as an ergonomic convenience layer without mixing them at the trait level.
**Requires user confirmation.**

### Q6 — BitLogic bounds and impl strategy

If Q5 = B, where do concrete `BitLogic` impls live?

**Option A — Blanket impl on any `T: BitSequence + BitAccess` using bit-by-bit
iteration.** Correct but slow per-bit. Defeats the purpose.

**Option B — Per-type concrete impl in `ufixed_impl.rs`.** One impl per
container (QWord/DWord/Word/…). Each delegates to the underlying `UFixed`
`to_raw()` / `from_raw()` once, then calls core operators. Identical
performance to the current arvo-bitmask bridge but the bridge now lives
inside arvo-bits (one level deeper, one abstraction up). Scales: adding a
new container means implementing BitLogic for it.

**Option C — Hybrid default-method pattern.** Trait carries default methods
that construct the result via `BitAccess::with_bit_*` calls; concrete types
override with `to_raw` / `from_raw` fast-path. Best of both worlds at the
cost of one extra trait-level knob.

**Provisional answer:** B. Consistent with how `BitSequence::trailing_zeros`
and friends are concrete-impl'd on `UFixed` today. Follows the existing
arvo-bits pattern.

**Q3 (user → agent clarification, 2026-04-20):** Option A — `lint:allow(...)` on
`cap_size(c: Cap) -> usize` + new long-horizon task.

User asked whether a trait could make `Cap` directly compatible in
const-generic array-length position. Agent clarification: no — Rust arrays
`[T; N]` hardcode `N: usize` at the language grammar level. `adt_const_params`
lets `Cap` be a const generic, and `generic_const_exprs` lets the length be a
computed expression like `cap_size(C)`, but the final expression TYPE must be
exactly `usize`. No trait can bridge that. Alternatives considered and rejected:

- Custom `FixedArray<T, Cap>` container (own indexing / iter impls): replaces
  built-in arrays everywhere they cross the `Cap` boundary. Full-stack scope
  creep, breaks the shipping `BitMatrix64` / `BitMatrix256` public surface.
- `pub type ArrLen = usize;` alias: evades the text-match lint but adds no
  semantic weight beyond `USize`. Renames without meaning.

Conclusion: `cap_size` is a language-limitation workaround; the lint is right
to flag it; the correct annotation is `lint:allow(arvo-types-only, no-bare-numeric)
reason: nightly generic_const_exprs requires raw usize in const-generic array-length
position (language grammar constraint); tracked: #121` (the new task captures "revisit when rustc stabilises
const-eval / array-length behavior"). Low priority, no ETA.

Final shape:
- `arvo-bits` gains `pub trait BitLogic: BitWidth + Copy` with four required
  methods: `bitor(self, other: Self) -> Self`, `bitand(...)`, `bitnot(self) -> Self`,
  `bitxor(...)`. Optional: `clear_lowest_set_bit(self) -> Self` convenience
  method (default impl on `BitAccess + BitSequence` bound, or required on
  `BitLogic`, TBD by Q6).
- `BitLogic` is impl'd concretely on each `UFixed<I, F, Hot>` container family
  that matters for the bitmask use case: `QWord<Hot>` is the immediate
  requirement; `DWord<Hot>`, `Word<Hot>`, `Byte<Hot>`, `Nibble<Hot>` follow the
  same pattern and round out the surface.
- `arvo-bits` ALSO adds `impl core::ops::{BitOr, BitAnd, BitNot, BitXor}` on
  the same containers (blanket on `T: BitLogic` if the orphan rules allow;
  otherwise per-type). Enables `a | b`, `a & b`, `!a`, `a ^ b` natively.

Impact on arvo-bitmask ops:
```rust

    let b = qword_to_u64(other.word);
    Self::from_word(u64_to_qword(a | b))

// after (either form; identical semantics)

    Self::from_word(<QWord<Hot> as BitLogic>::bitor(self.word, other.word))

// or, via core::ops impl:

    Self::from_word(self.word | other.word)
```

**Q6 (user, 2026-04-20):** Option A — per-type concrete impls + separate
`clear_lowest_set_bit` method.

Structure:
- `arvo-bits/src/traits.rs` — define `pub trait BitLogic: BitWidth + Copy`
  with four required methods. No default implementations (Option B rejected
  as defeating the single-instruction semantics).
- `arvo-bits/src/ufixed_impl.rs` — concrete `impl BitLogic for UFixed<I, F, Hot>`
  per container family, each delegating once via `UFixed::to_raw`/`from_raw` to
  the core operator on the backing primitive. Pattern identical to how
  `BitSequence::trailing_zeros` / `count_ones` are concrete-impl'd today.
- `clear_lowest_set_bit(self) -> Self` — default method on `BitLogic`, guarded
  by a `Self: BitAccess + BitSequence` bound, computed as:
  ```rust
  fn clear_lowest_set_bit(self) -> Self where Self: BitAccess + BitSequence {
      if <Self as BitSequence>::is_zero(self).into() { return self; }
      let idx = <Self as BitSequence>::trailing_zeros(self);
      <Self as BitAccess>::with_bit_cleared(self, idx)
  }
  ```
  Concrete types can override with the `self.to_raw() & (self.to_raw() - 1)`
  fast path via `to_raw`/`from_raw`. (One instruction on x86 via BLSR.) Keep
  the default as a correctness baseline; override on `UFixed<I, F, Hot>` as
  part of the `BitLogic` impl block for the fast path.
- `arvo-bits/src/ufixed_impl.rs` — also add `impl core::ops::{BitOr, BitAnd,
  BitNot, BitXor} for UFixed<I, F, Hot>` per Q5, delegating to `BitLogic`.
  Orphan rules permitting, make this a blanket `impl<T: BitLogic> core::ops::BitOr for T` in arvo-bits; otherwise per-type.
