## Topic 1: Pass 2 carrier-widening workaround

### Round B's deferred plan

Round B's locked doc CL specifies the meta-newtype design destination:

```rust
pub struct IBits(pub Bits<9, Hot>);
pub struct FBits(pub Bits<9, Hot>);
pub struct Width(pub Bits<9, Hot>);
```

Carrier rationale: 9-bit native covers 0..=511; substrate uses 0..=255. Bits at the meta layer keeps the framing "the substrate's primitives are built on substrate primitives" — meta-newtypes wrap `Bits`, not bare `u8`.

### The cycle (re-stated for Round C)

Implementation attempt during Round B SRC phase:

```
error[E0391]: cycle detected when evaluating trait selection obligation
              `arvo_storage::meta_bits::IBits: Sized`
note: ...which requires building an abstract representation for
      `ufixed::<impl ...>::{constant#0}` ...
note: ...which requires type-checking
      `Hot: UContainerFor<{ ufixed_bits(I, F) }>` ...
note: ...which again requires evaluating
      `IBits: Sized`, completing the cycle
```

The cycle is structural: writing `pub Bits<9, Hot>` (with default `Sign = Unsigned`) as the meta-newtype field forces rustc to resolve `Hot: BitsContainerFor<9, Unsigned>` at IBits's well-formedness check. That resolution interleaves with UFixed's const-eval where-clause `Hot: UContainerFor<{ ufixed_bits(I, F) }>` (which itself requires `IBits: Sized`), producing a cycle the detector trips on.

Original `pub u8` field bypassed this because u8's Sizedness is primitive — no trait-solver hop. The Bits-projection field adds a hop the const-eval consumer cannot wait for.

### Workaround design: parallel layout-stable carrier

Introduce a concrete (non-generic, non-projected) companion type that is `repr(transparent)` over u16. Layout-identical to `Bits<9, Hot, Unsigned>`. Free zero-cost conversion to the Bits-typed view via an `as_bits()` accessor or `From` impl.

The meta-newtype field type is the concrete companion (not the generic Bits projection), so well-formedness check at the meta-newtype layer never enters the trait-solver path that cycles.

**Naming**: `MetaCarrier` (singular, hardcoded for the 9-bit / u16 / Hot / Unsigned shape). Lives in `arvo-storage/src/meta_bits.rs` (alongside the meta-newtype macro that uses it). Re-exported from `arvo_storage`.

**Layout invariant** (load-bearing): MetaCarrier and `Bits<9, Hot, Unsigned>` are byte-identical. Both are `repr(transparent)` over u16. Free conversion is sound.

```rust
#[repr(transparent)]
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug, Default, Hash)]
pub struct MetaCarrier(pub u16);

impl MetaCarrier {
    pub const fn from_raw(n: u16) -> Self { Self(n) }
    pub const fn to_raw(self) -> u16 { self.0 }
    /// Zero-cost view as `Bits<9, Hot, Unsigned>`.
    ///
    /// `MetaCarrier` and `Bits<9, Hot, Unsigned>` are both
    /// `repr(transparent)` over u16; this transmute is sound.
    pub const fn as_bits(self) -> Bits<9, Hot, Unsigned> {
        // SAFETY: layout-identical (repr(transparent) over u16).
        unsafe { core::mem::transmute(self) }
    }
}
```

Then meta-newtypes:

```rust
pub struct IBits(pub MetaCarrier);
pub struct FBits(pub MetaCarrier);
pub struct Width(pub MetaCarrier);

impl IBits {
    pub const fn raw(self) -> u16 { self.0.to_raw() }
    pub const fn as_bits(self) -> Bits<9, Hot, Unsigned> { self.0.as_bits() }
}
```

### Concept retention check

The doc CL claim: "meta-newtypes carry `Bits<9, Hot>`". Verbatim implementation literal would be `pub struct IBits(pub Bits<9, Hot>)`. The workaround uses `pub struct IBits(pub MetaCarrier)` where MetaCarrier is layout-equivalent to `Bits<9, Hot, Unsigned>`.

Concept retained:
- 9-bit native carrier: yes (u16 underlying primitive matches `Bits<9, Hot, Unsigned>::T`).
- Hot strategy: yes (no other Strategy is involved; physically the same as Bits<9, Hot, Unsigned>).
- Unsigned: yes (no signed bit, plain u16).
- Bits-typed view available: yes (via `as_bits()`, zero-cost).

What changed: the field type's textual form. `MetaCarrier` instead of `Bits<9, Hot>`. Doc framing updates to: "meta-newtypes carry a layout-stable companion `MetaCarrier` to `Bits<9, Hot, Unsigned>`; the Bits-typed view is reachable via `MetaCarrier::as_bits()` at zero cost".

### Decision on Topic 1

Adopt the parallel-carrier workaround. Doc CL adjusted from Round B's design to mention `MetaCarrier` alongside the `Bits<9, Hot, Unsigned>` framing. Source ships in this round.

### Alternatives considered (rejected)

- **Spell `Bits<9, Hot, Unsigned>` explicitly** (drop default Sign inference): same fundamental projection still applies; cycle-prone.
- **Reorder UFixed where-clause to extract via `.0.0` field access**: depends on implementation visibility of Bits's inner field (currently private to arvo-storage); cross-crate access not available.
- **Restructure `ufixed_bits` to take u16 directly**: `I.raw()` still returns through a method call on IBits, requires `IBits: Sized` at the same const-eval boundary; cycle-prone.
- **Specialization on (N=9, Hot, Unsigned) shape of Bits**: rustc specialization not stable.

The parallel-carrier approach is the only path that demonstrably bypasses the trait-solver cycle while preserving layout equivalence and the conceptual framing.

## Topic 2: Internal strategy axes decomposition

The `Strategy` axis (Hot / Warm / Cold / Precise) couples three orthogonal concerns:

| Axis | Hot | Warm | Cold | Precise |
|---|---|---|---|---|
| Container width | min-aligned | 2x logical | min-aligned | 2x logical |
| Overflow policy | wrapping | wrapping | wrapping | saturating |
| Storage layout | dense | dense | bitpacked | dense |

Coupling means a consumer choosing "I want bitpacked storage" (Cold) also gets "wrapping arith with widen-op-narrow" — they cannot mix bitpacking with saturating arith without writing an entirely new strategy marker.

### Proposed: orthogonal sub-axes

Internally decompose `Strategy` into three axis traits the strategy markers implement:

```rust
pub trait OverflowPolicy { /* wrapping / saturating */ }
pub trait ContainerWidth { /* min / 2x */ }
pub trait StorageLayout { /* dense / bitpacked */ }
```

The four markers implement specific axis combinations:

| Marker | OverflowPolicy | ContainerWidth | StorageLayout |
|---|---|---|---|
| Hot | Wrapping | Min | Dense |
| Warm | Wrapping | DoubleLogical | Dense |
| Cold | Wrapping | Min | Bitpacked |
| Precise | Saturating | DoubleLogical | Dense |

Public surface unchanged — `Hot` / `Warm` / `Cold` / `Precise` remain the only strategy markers consumers see. Internal arithmetic / widen / narrow impls dispatch via the axis traits, allowing future experimentation with mixed axes (e.g. saturating bitpacked storage) without breaking the public surface.

### Decision on Topic 2

Land the axis traits in this round; existing markers implement the appropriate axis combinations; arithmetic / widen / narrow impls are NOT refactored to dispatch via axes in this round (substantial cascade; defer to a follow-up). The axis traits exist as documentation + future-proofing primitives.

## Topic 3: Bitfield S parameterisation

Today `bitfield!` macro and field-level Bits primitives are implicitly `Hot`-only at the internal API. Consumers building column-store layouts with Cold storage cannot reach for bitfield-packed columns through the existing macro.

### Proposed

Add explicit `S: Strategy` parameter to the bitfield macro's expansion and to relevant internal primitives. Default `S = Hot` to preserve existing call sites. Field accesses dispatch through the strategy's storage layout (dense load+mask for Hot/Warm/Precise; bitpacked load+mask for Cold).

### Decision on Topic 3

Defer the bitfield S parameterisation to a follow-up round. Reason: the bitfield macro is in arvo-bits / arvo-bitmask, not in arvo-storage / arvo-strategy where Round C centers; the cascade across consumers is non-trivial and warrants its own dedicated round. Note in BACKLOG.

## Topic 4: Cross-strategy warning

When a binary op `Hot * Warm` resolves via `Resolve<Hot, Warm> = Warm`, the resulting expression silently adopts Warm's overflow policy (wrapping at 2x-logical width). When `Hot * Precise` resolves via `Resolve<Hot, Precise> = Precise`, the expression silently adopts Precise's saturating semantics. These are non-obvious semantic shifts.

### Proposed

Compile-time `#[deprecated]` or `#[diagnostic]` warning at the cross-strategy op site when the resolved strategy differs from both operands' strategy. Implementation via a marker trait `CrossStrategyOp<S1, S2>` that the user can opt out of via `lint:allow`.

### Decision on Topic 4

Defer to a follow-up round. Reason: requires designing the diagnostic trait + lint-pack interaction; substantial enough to warrant a dedicated round. Note in BACKLOG.
