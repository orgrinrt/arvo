## Purpose

Four corrections before arvo-spectral impl — all around preserving
arvo's newtype surface on the public API.

1. **`const N: Cap`, not `const N: usize`.** Every public function
   signature uses the arvo `Cap` newtype (wraps `USize` wraps `usize`)
   as its const-generic size parameter. Array-sizing inside struct
   fields / method bodies extracts to `usize` via inline `.0.0 as
   usize` in const contexts. Mirrors the L0 pattern where the public
   surface is `const I: IBits` and the storage path extracts via
   `ufixed_bits(I, F) as usize`.

2. **`USize` over `usize`** in all count / index positions on the
   public surface.

3. **`Matrix<W, const N: Cap>`** wrapper type — a dense `N × N` matrix
   of `W`. Ships local to arvo-spectral this round (can be promoted to
   arvo-sparse or arvo proper later if other crates need it). Avoids
   the bare `[[W; N]; N]` on the public API and gives room for
   blanket-impl extension later (identity, diagonal, transpose, etc.).

4. **`Mask64`-backed partition outputs** stay, but with the `N ≤ 64`
   constraint documented explicitly (`Mask256` variants in
   `BACKLOG.md.tmpl`).

These are surface/typing corrections. Algorithm choices (Laplacian →
Fiedler → spectral bisection → recursive k-way) are unchanged.

## Per-crate doc changes

### arvo-spectral — `mock/crates/arvo-spectral/DESIGN.md.tmpl`

- Introduce a `Matrix<W, const N: Cap>` dense-matrix wrapper with a
  private `[[W; _]; _]` storage. Public methods: `new`, `get(i:
  USize, j: USize) -> W`, `set(&mut, i, j, W)`, `from_fn(f)`, plus
  iteration helpers as impl-time discovers a need.
- Replace every `const N: usize` in DESIGN signatures with `const N: Cap`.
- Replace every bare `usize` in count / index positions with `USize`.
- Replace every `&[[W; N]; N]` with `&Matrix<W, N>`.
- Keep `Mask64` output for partitioning; note `N ≤ 64` constraint
  explicitly.

### `mock/crates/arvo-spectral/BACKLOG.md.tmpl`

Add an entry: "Matrix-in-arvo-sparse promotion" — note that the
`Matrix<W, N>` wrapper currently lives in arvo-spectral, and if other
consumers need a dense-matrix surface, promote it into arvo-sparse
(the "matrix representations" crate) as part of a future round.
