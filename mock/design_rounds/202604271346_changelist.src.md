### FromConstant signature: trait-level → method-level

Deprecated src CL specified the trait declaration in `arvo-numeric-contracts/src/lib.rs` as:

```rust
pub const trait FromConstant<const C: USize> {
    fn from_constant() -> Self;
}
```

with consumer impls of the shape `impl<const C: USize> FromConstant<C> for UFixed<{IBits($i)}, {FBits::ZERO}, $strategy>`. This shape is blocked by `feature(generic_const_exprs)` instability: rustc treats `UFixed<{IBits($i)}, {FBits::ZERO}, $strategy>` in any impl with an outer generic parameter as still abstract over `I, F`, and rejects the `Hot: UContainerFor<{ ufixed_bits(I, F) }>` bound on the (concrete) UFixed instantiation. Adding `where [(); { ufixed_bits(IBits($i), FBits::ZERO) } as usize]:` and explicit `$strategy: UContainerFor<{ ufixed_bits(IBits($i), FBits::ZERO) }>` clauses do not help; rustc still reasons over abstract `I, F`.

This src CL revises the declaration to method-level const generic per the re-locked doc CL's promoted-canonical paragraph (`### FromConstant signature clarification`):

```rust
pub const trait FromConstant {
    fn from_constant<const C: USize>() -> Self;
}
```

A single non-generic impl per concrete numeric type covers every `C`:

```rust
impl FromConstant for UFixed<{ IBits($i) }, { FBits::ZERO }, $strategy> {
    fn from_constant<const C: USize>() -> Self {
        Self::from_raw(C.0 as $ctype)
    }
}
```

This shape is rustc-friendly (no outer generic on the impl, no `generic_const_exprs` interaction) and reduces consumer-side bound noise (`F: FromConstant` instead of `F: FromConstant<{USize(0)}> + FromConstant<{USize(1)}> + ...` per used constant).

### Consumer ripple: FromConstant call sites

Method-level const generic consumer call-site form: `<F as FromConstant>::from_constant::<{USize(0)}>()` or `F::from_constant::<{USize(0)}>()` (turbofish on the method, not the trait). The consumer where-clause stays plain `F: FromConstant`.

Existing runtime FromConstant call sites in algorithm crates (deprecated src CL pass 6 ripple) require additional rewrites because no runtime `from_constant(USize(n))` shape exists in the new design:

- `mock/crates/arvo-spectral/src/fiedler.rs:82,83,95`: compile-time `F::from_constant(USize(0))`, `F::from_constant(USize(1))`, `F::from_constant(USize(2))` rewrite to `F::from_constant::<{USize(0)}>()`, `F::from_constant::<{USize(1)}>()`, `F::from_constant::<{USize(2)}>()`. Trivial mechanical edit.
- `mock/crates/arvo-spectral/src/fiedler.rs:89`: runtime `F::from_constant(USize(n))` where `n` is a runtime fn arg. No const-trait equivalent. Rewrite as fold-counted construction:

  ```rust
  let one = F::from_constant::<{USize(1)}>();
  let zero = F::from_constant::<{USize(0)}>();
  let mut n_f = zero;
  let mut k = 0usize;
  while k < n { n_f = n_f + one; k += 1; }
  ```

  O(n) overhead with n ≤ 64 (Fiedler's documented bound); negligible against the matrix-multiply core loop.

- `mock/crates/arvo-comb/tests/dp.rs:24`, `binpack.rs:26`; `mock/crates/arvo-graph/tests/rank.rs:30`, `path.rs:29`, `spanning.rs:28`: test helpers with concrete `type W = UFixed<{IBits(8)}, FBits::ZERO, Hot>;` and `fn w(n: usize) -> W { W::from_constant(USize(n)) }`. Rewrite as direct construction `W::from_raw(n as u8)` with `lint:allow(no-bare-numeric) reason: test helper; runtime usize→u8 cast for typed weight in concrete-W test scope; no runtime-FromConstant by design (round 202604271346, FromConstant is compile-time only); tracked: #256` annotations.

- `mock/crates/arvo-spectral/tests/common/mod.rs:89`: custom `impl FromConstant for TF` rewrites to method-level shape `impl FromConstant for TF { fn from_constant<const C: USize>() -> Self { TF(C.0 as f32) } }`.

### Pass 5+6 notes (helper deletion + bitfield macro rewrite)

The two passes land together because the `bitfield!` macro in `arvo/src/bitfield.rs` emits the per-size helpers (`Bits::<N, Hot>::new`, `bits()`, `from_raw_uN`, `to_raw_uN`) that pass 5 deletes from `arvo-storage/src/bits.rs`. Deleting the helpers without rewriting the macro body would leave the macro emitting calls to non-existent methods.

Source mutations:

- `arvo-storage/src/bits.rs` lines 107-198 deleted: the entire `// --- Per-size helpers (D-7 deletion target) ---` block, the `impl_bits_u64!` macro family definition, and its four invocations (one per size class).
- `arvo/src/bitfield.rs`: bitfield macro body rewritten to compose `Bits::from_raw(<container>)` + `Bits::to_raw()` + a u64-bridged shift/mask layer. A new `__bitfield_container_ty!` helper macro (size-class lookup table for N=1..=64) bridges across the typed-container boundary inside macro expansion. The `bitfield!` DSL captures (`$n`, `$field_bits`, `$lo`) switched from `:literal` to `:tt` so they can pass through the helper macro's literal-pattern matching (per Rust reference: only `:tt`, `:ident`, `:lifetime` metavariables can be compared as tokens in nested macro_rules).
- `arvo-hash/src/fnv1a.rs`: `impl_fnv1a!` macro takes an additional `$ty:ty` arg (per size class). Per-N invocation list mirrors UContainerFor<Hot, N> exactly: `(u8, 1..=8)` `(u16, 9..=16)` `(u32, 17..=32)` `(u64, 33..=64)`. Body of `hash_const` and `finalize` rewrites `Bits::from_raw_u64(state)` to `Bits::from_raw((state & mask) as $ty)` with explicit u64 mask construction.
- `arvo-hash/src/algo.rs`: doc comment for `fnv1a_64` updated to describe the new mask-and-cast path (no method change; doc only).

Test rewrites in the same commit:

- `arvo-bits/tests/bits.rs`: every `Bits::<N>::new(literal)` rewritten to `Bits::<N>::from_raw(literal as <container>)`; every `.bits()` rewritten to `.to_raw()`. Tests that exercised `Bits::new`'s implicit input-masking are dropped or rephrased to test the from_raw round-trip directly. - `arvo-bits/tests/bitfield.rs`: same shape; `Bits::<N, Hot>::new(literal)` → `Bits::from_raw(literal as <container>)`; `.bits()` → `.to_raw()`. The implicit-masking test (`setter_truncates_to_subrange_width`) does the masking explicitly at construction.
