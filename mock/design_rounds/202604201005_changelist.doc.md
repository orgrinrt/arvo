### `mock/PRINCIPLES.md.tmpl`

Insert a new **arvo-specific** section immediately after the shared
cross-ecosystem principles block, before the existing per-arvo
principles. The new section:

```markdown
## arvo: the fixed-point arithmetic stack

arvo is a standalone `#![no_std]` numeric primitive library family,
cross-project — consumed by hilavitkutin, polka-dots, saalis, and
loimu. It is the exclusive numeric substrate for the entire
clause-dev workspace.

Every numeric primitive flows through `UFixed` / `IFixed` /
`FastFloat` / `StrictFloat`. Bare Rust primitives — `u8`, `u16`,
`u32`, `u64`, `u128`, `i8`, `i16`, `i32`, `i64`, `i128`, `f32`,
`f64`, `bool`, `usize`, `isize` — effectively do not exist in
arvo's universe. This is not a preference, not a default, not a
"usually use": it is the identity. Reach for a raw primitive and
you have forgotten what arvo is.

The discipline is enforced by the `arvo-types-only` lint (Task #50,
shipping via #103). The lint rejects raw numerics in data-facing
struct fields and public-API positions across every arvo sub-crate
and every downstream consumer (hilavitkutin, clause). Until the
lint lands, the rule holds by convention and review.

The algorithm crates (arvo-graph, arvo-sparse, arvo-comb,
arvo-spectral) are generic over numeric trait bounds and reflexively
reach for arvo primitives. They do not accept or return bare types
at any boundary.

Strategy markers (`Hot`, `Warm`, `Cold`, `Precise`) are type
parameters, not runtime config — they embed operational intent
into the type system. A `Mass<Hot>` is explicitly lossy-but-fast;
a `Mass<Precise>` is explicitly exact-but-expensive. The caller
picks the tradeoff at the type level.

All arvo types are `repr(transparent)` over their chosen container.
After compilation, arvo primitives incur zero overhead beyond their
strategy's intended cost.
```

```markdown
## Before the rule: what arvo IS

arvo is a fixed-point arithmetic stack. Every numeric primitive —
`u8` through `u128`, `i8` through `i128`, `f32`, `f64`, `bool`,
`usize`, `isize` — is replaced by an arvo primitive
(`UFixed<I, F, S>` / `IFixed<I, F, S>` / `FastFloat` / `StrictFloat`
/ `Bool` / `USize` / `Cap`) or one of the pre-defined semantic
aliases over them (`Byte`, `Word`, `DWord`, `QWord`, `Nibble`, `Bit`
from arvo-bits). When you encounter a raw Rust numeric in a public
signature, the question isn't "which arvo type matches this width";
it's "which arvo type matches this DOMAIN". Public APIs carry
domain aliases (`NodeId`, `Str`, `RecordIndex`, `SieveWeight`,
`ContentHash`) grounded on arvo primitives. Never raw primitives;
never raw newtype-over-primitive either.

Enforced by `arvo-types-only` (#50) + `no-bare-numeric` (#68) +
`no-bare-option/result/string/vec` (#66 / #67 / #69 / #76). Until
those lints land strict, discipline carries it.
```

Existing decision test + DO/DON'T table + red flags section stay
unchanged.
