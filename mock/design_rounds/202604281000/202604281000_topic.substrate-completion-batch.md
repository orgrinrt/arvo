## Decision: Pass A — Soundness extensions on top of Round D-aux

Three sub-passes, all additive to `arvo-storage/src/layout_assertions.rs` and `arvo/src/layout_assertions.rs`.

**Pass A.1 — `HasAxes` correctness check.** `HasAxes` (the projection trait added in Round C) maps each strategy to its axis combination (`OverflowPolicy`, `ContainerWidth`, `StorageLayout`). The trait is documented but never cross-checked. If a future round changes Hot's `ContainerWidth` impl while leaving `HasAxes::Container` pinned to the old value, the documentation drifts silently.

The fix: compile-time `assert_eq!(<Hot as HasAxes>::Container::DISCRIMINANT, <Hot as ContainerWidth>::Output::DISCRIMINANT)` style guards, one per (Strategy, Axis) cell. Each axis surface gets a `const DISCRIMINANT: u8` for compile-time projection.

**Pass A.2 — Full Bits-cell assertion coverage.** Round D-aux pinned 60+ boundary cells. The remaining cells (every (Strategy, N, Sign) tuple in 1..=255 × 4 strategies × 2 signs) are unverified. A macro can expand the full table.

Risk: the macro generates ~2000 `const _: () = ...` items. Build-time impact must be bounded. Decision: ship the macro generating boundary-cell coverage by default (already in Round D-aux); add a `cfg(arvo_full_layout_audit)` gate that opens up the full expansion. Consumers running the audit on a periodic CI nightly get the deep check; default builds keep the boundary cells.

**Pass A.3 — `ConstParamTy` soundness audit.** Every newtype in const-generic position (`IBits`, `FBits`, `Width`, `MetaCarrier`, the future `Bits<const N: u16>` parameter cell, etc.) requires `StructuralEq + Copy + Eq + ConstParamTy`. The audit pass adds compile-time `assert_impl!` lines for each, ensuring the trait obligations are met.

## Decision: Pass B — Original Round D content

Four sub-passes. Each landed as a separate commit.

**Pass B.1 — Hash family adoption.** `arvo-hash` ships with FNV-1a as the default since Round B. xxhash3 (or wyhash; pick one in this round per micro-benches against the substrate's representative payloads) becomes the new default. FNV-1a stays as an explicit option for consumers with known-good fits. `ContentHasher` is the trait surface; concrete strategies implement it.

Decision input: xxhash3 wins on bandwidth for inputs >= 32 bytes; FNV-1a stays competitive for 8-16 byte inputs (typical id-hashing workload). The substrate's primary consumer (loimu-style entity content addressing) sees inputs 16-128 bytes. xxhash3 ships as default; the BACKLOG entry to bench wyhash stays open.

**Pass B.2 — `ContentHash` widens to `Bits<64, Hot>`.** Currently `ContentHash` is a `Bits<32, Hot>` newtype. 64 bits matches xxhash3's natural output and reduces collision probability for the consumer's content-addressing workload (millions of entities). The widen is a public-API change; the type alias keeps its name and consumers re-monomorphise.

**Pass B.3 — Narrow surface expansion.** The `Narrow` trait in `arvo` currently masks unconditionally. Many hot paths know the high bits are already zero (chained narrow, extract-then-narrow). Adding `narrow_to_unmasked(self) -> Self::Out` skips the mask. Soundness depends on the precondition; document it.

**Pass B.4 — arvo-narrow population.** The `arvo-narrow` crate (a contract crate currently empty per the lint warning) gains concrete impls: per-source-target primitive narrowing (u16 → u8, u32 → u16, etc.) and typed `Bits<M, S, Sign> → Bits<N, S, Sign>` where M > N. The empty-crate lint then clears.

## Decision: Pass C — Bitfield `S` parameterisation

The `bitfield!` macro currently emits Hot-strategy storage. Consumers wanting Cold-bitpacked column-store bitfields have no recourse. The macro accepts an explicit `S: Strategy` parameter (default `Hot` for backwards compat). Cascade across consumers: `arvo-bitmask`, `arvo-graph`, downstream lint configs.

Per the toolbox-not-policer rule: do not switch the default to Cold. Cold-bitpacked column stores are exactly the workload arvo exists for, but the substrate does not pre-judge whether a given bitfield consumer is one of those. The default stays Hot; consumers who know they want Cold turn the knob.

## Decision: Pass D — Precise / Warm 33..=64 inconsistency fix

Layout assertions captured the inconsistency: Precise at 33..=64 uses u64 (size 8 bytes); Warm at 33..=64 uses u128 (size 16 bytes). Both are documented as "2x logical width" strategies. The 2x rule says Warm and Precise should match at every cell. They do not at 33..=64.

The fix: promote Precise to u128 at 33..=64. Cascades into `traits.rs` macro-generated impls (`impl_abs_ifixed_integer_sat`, `impl_from_constant_ufixed`) that hard-coded the Precise=u64 assumption across 33..=63. Each macro arm gets a u128 variant.

The layout assertions in `arvo-storage/src/layout_assertions.rs` flag the change automatically once the projection table updates.

## Decision: Pass E — Strategy-axis-driven arithmetic dispatch

Currently `UArith` / `IArith` / `UWidenFrom` / `UNarrowFrom` dispatch via the bundled `Strategy` marker. The Round C decomposition (`OverflowPolicy` / `ContainerWidth` / `StorageLayout`) added the axis traits but the arith dispatch never adopted them. This pass refactors the dispatch to project through the axes.

Why it matters: `Hot + Saturating` (the consumer wants Hot's wide container plus saturating overflow) currently has no path. Axis-driven dispatch lets the trait solver pick `PADDUSB` / `SQADD` codegen for that combination. Same for `Cold + Saturating`, `Warm + Wrapping`, etc.

Risk: substantial cascade. Every `impl UArith for X` in the substrate gets rewritten to bound on axes instead of the bundled marker. Tests stay green only if the axis projections are correct. The layout-soundness gates pin the cells; the axis-correctness check from Pass A.1 pins the trait projections.

The dispatch refactor is the largest pass. Land it after Pass C (so the bitfield macro's S param has shipped and the cascade has one fewer concurrent change) and after Pass D (so the Precise/Warm fix is captured before the dispatch surface widens).

## Decision: Pass F — `Bits<const N: u16>` widening

`Bits` storage primitive's const-generic param is currently `u8`. That caps reachable bit-widths at 255. The substrate dispatches up to that today via `MultiContainer<u128, u128>` at 193..=255.

Lift the const-generic param to `u16` (or wider; `u16` covers up to 65535, far beyond any reachable consumer width). Used range stays 1..=256 in this round; the actual cell projections only define impls up to N=256. The const-generic widening is the parameter-space change; growing the cell table is a future round.

Cascade: every `Bits<N, S, Sign>` reference in the workspace recomp from `u8` to `u16`. `IBits` / `FBits` / `Width` already widened to MetaCarrier (which is `repr(transparent)` over u16) in Round B; this pass closes the loop.

The `ConstParamTy` audit from Pass A.3 must complete first so the new `u16`-shaped const generic position is verified to satisfy the trait obligations.

## Decision: Pass G — Cross-strategy resolution warning

Cross-strategy ops resolve via `Resolve<S1, S2>::Out`. Today the resolution is silent: `Hot + Wrapping` with `Precise + Saturating` projects to `Precise + Saturating` per the resolution table, and the consumer's wrapping site quietly becomes saturating. The substrate cannot refuse the op (the toolbox-not-policer rule forbids that), but it can warn.

The fix: `CrossStrategyOp<S1, S2>` marker trait with `#[diagnostic::on_unimplemented]` on the unresolved projection cells. Consumer opt-out via `lint:allow` annotation. The warning fires at compile time, points the consumer at the resolution path, and lets them decide whether to add the explicit cast.

Design-heavy because the warning text matters. The diagnostic must explain WHICH semantic shifted and HOW (overflow policy, container width, storage layout). Each axis surface contributes a piece. The actual warning text gets drafted in this round; iterations welcome via consumer feedback.

This pass lands last because it bounds on the axis traits (added by Round C; refactored to be the dispatch backbone in Pass E). Writing the warning before Pass E refactors the surface would force a rewrite.
