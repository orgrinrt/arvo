# Doc Changelist: arvo L0 float arithmetic ops

**Date:** 2026-04-26
**Phase:** DOC
**Scope:** `arvo` crate only.
**Source topics:**
- 202603241400_topic.arvo-crate-dependency-revision.md (generic contracts)

## Purpose

Close the L0 float-arith gap surfaced during arvo-spectral impl
(Task #52). `FastFloat<F: Ieee>` and `StrictFloat<F: Ieee>` ship
with `TotalOrd`, `Sqrt`, `Recip`, `Abs`, `FromConstant` — but not
the four `core::ops::{Add, Sub, Mul, Div}` traits. arvo's DESIGN
explicitly calls for consumers to write `T: Add + Mul + Sqrt`
against these types, so missing arithmetic ops is a DESIGN-src
mismatch that forced arvo-spectral tests onto a local `TF(f32)`
workaround.

This round adds the four ops plus `core::ops::Neg` delegating to
the inner `F`. No DESIGN changes — the ops are already implied by
the DESIGN prose at `mock/crates/arvo/DESIGN.md.tmpl:401`
("Consumer writes `T: Add + Mul + Sqrt`"). This round only fulfils
that prose in src.

Semantics:
- `FastFloat<F>`: delegate to `F`'s ops. LLVM fast-math flags are
  set by the consumer's build target (hilavitkutin-build FastMath
  pragma cfg); the Rust-level source is plain delegation.
- `StrictFloat<F>`: delegate to `F`'s ops with IEEE 754
  bit-exact semantics (Rust's default behaviour for `f32`/`f64`).

No changes to `Ieee` supertrait bounds. The op impls use `F: Ieee`
plus whichever `core::ops::_<Output = F>` bound each impl needs;
f32 and f64 satisfy all of these in `core`, so the Ieee seal is
preserved.

## Per-crate doc changes

None — the DESIGN already calls for these ops. No BACKLOG changes;
no SHAME entries. This is a pure src fulfilment round.

- `mock/crates/arvo/BACKLOG.md.tmpl` (does not exist; no need to
  create).
- Any L1 / L2 / L3 crate. arvo-spectral's local `TF(f32)` workaround
  stays for this round; a follow-up L3 round can switch it to
  `StrictFloat<f32>` once these ops land.

## Comparison to deprecated changelist

No deprecated predecessor.

## Execution

Pure src round. Lock the doc CL, author a src CL that adds the five
ops × two wrappers × `F: Ieee` generic parameter = five impl blocks
using a macro to avoid ten hand-written trivial impls. Tests exercise
each op on `FastFloat<f32>`, `StrictFloat<f32>`, `FastFloat<f64>`,
`StrictFloat<f64>`.

## Out of scope

- LLVM fast-math intrinsic wiring (a hilavitkutin-build or codegen
  concern, not src).
- `core::ops::{Rem, Shl, Shr}` — not implied by DESIGN prose. If a
  consumer surfaces a need, add in a follow-up round.
- Saturating / checked arithmetic variants — the floats carry IEEE
  semantics by construction; wrapping isn't meaningful.
- Switching arvo-spectral's `TF(f32)` test workaround to
  `StrictFloat<f32>` — separate round; proves this round's impls
  work but stays out of scope to keep this round small.
