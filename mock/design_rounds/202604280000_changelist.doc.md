# Doc Changelist: arvo-comb sweep + arvo-tensor crate introduction

**Date:** 2026-04-20
**Phase:** DOC
**Scope:** New `arvo-tensor` crate (L2 peer), promoted `Matrix<W, N>` out of
  arvo-spectral, shared `cap_size` centralised. arvo-comb, arvo-bitmask,
  arvo-spectral each updated to reflect the new crate + drop/repoint local
  duplicates. Root DESIGN + agent templates propagate the new crate in
  listings + dep graph + forbidden-imports rules.
**Topic:** `202604280000_topic.arvo-comb-sweep.md`
**Type:** additive (new crate), promotion (Matrix + cap_size relocate),
  rewrite (arvo-comb API sigs route through `Array<T, N>` / `Matrix<Bool, N>`
  + return `Bool` from feasibility callbacks).

## Motivation

Session directive is uncompromising: bare primitives do not exist anywhere
in source. The arvo-comb audit found 46 drift sites — almost all shaped as
`while i < cap_size(N) { ... i += 1; }` loops over raw `&[T; cap_size(N)]`
arrays, plus `[[bool; N]; N]` internal buffers, plus `Fn(...) -> bool`
callback sigs. Patching each loop with `lint:allow` was rejected by the user
as treating symptoms not root cause.

Root cause: the raw `[T; cap_size(N)]` array shape leaks raw `usize` + raw
`bool` through every loop and API position that touches it. The fix is a
structured wrapper — exactly the pattern `arvo_spectral::Matrix<W, N>`
already ships for 2D dense data. Generalise it: promote Matrix out of
arvo-spectral, introduce its 1D sibling `Array<T, N>`, and put both in a
new dedicated crate `arvo-tensor`. Every current and future algorithm crate
takes `Array<T, N>` / `Matrix<W, N>` instead of raw arrays, loops iterate
via a standard `IntoIterator` impl or an arvo-native `Enumerator` trait
yielding `(USize, &T)`.

Cascades cleanly: arvo-graph, arvo-sparse, arvo-spectral each migrate to
arvo-tensor during their own sweep rounds (under the #120 umbrella).
Downstream consumers (hilavitkutin, clause) will also use arvo-tensor
directly where they currently pass raw arrays to arvo algorithm crates.

**`README.md.tmpl`** (3-10 lines, standard template):

```markdown
# arvo-tensor — L2 fixed-shape tensor wrappers

Rank-N containers with type-level capacity: `Array<T, N>` (rank-1),
`Matrix<W, N>` (rank-2). `cap_size(c: Cap)` bridge + `Enumerator` trait
for `(USize, &T)` iteration. No alloc, stack-only. Depends on arvo only.
```

**`DESIGN.md.tmpl`** — new. Full design doc (~150 lines). Sections:

- Identity and scope: fixed-shape typed containers with type-level capacity.
  Each container binds a `Cap` const generic to a homogeneous value type;
  every slot is materialised; no heap.
- Position in the stack: L2, sibling of arvo-bitmask. Depends on arvo (L0)
  only. No dependency on arvo-bits or arvo-bitmask — tensor is
  value-generic, not bit-generic.
- The `cap_size` helper — canonical home. Replaces duplicates in
  arvo-bitmask and arvo-spectral.
- `Array<T, const N: Cap>` — rank-1 wrapper. `#[repr(transparent)]` over
  `[T; cap_size(N)]`. Methods: `new`, `from_fn`, `get(USize) -> &T`,
  `set(USize, T)`, `len() -> USize`, `diagonal`-ish ops where meaningful.
- `Matrix<W, const N: Cap>` — rank-2 wrapper (promoted verbatim from
  arvo-spectral). Methods: `new`, `from_fn`, `get(USize, USize)`, `set`,
  `diagonal`.
- `Enumerator` trait — blanket `impl<T: IntoIterator> Enumerator for T`
  providing `.enumerated() -> impl Iterator<Item = (USize, Self::Item)>`.
  Turns standard `.enumerate()` (raw usize) into typed USize indices.
- `IntoIterator` impls on `&Array<T, N>` and `&Matrix<W, N>` yielding `&T`
  / `&W` (so `for t in &arr { ... }` works standard).
- Scope boundaries (does NOT contain): arithmetic operations on tensors
  (consumer concern), sparse shapes (arvo-sparse), bit-shaped tensors
  (arvo-bitmask's domain), ML-style broadcasting (out of scope, not
  aligned with arvo's substrate role).

**`BACKLOG.md.tmpl`** — empty-ish, list future rank-N shapes (`Grid<T, N, M>`,
`Cube<T, N, M, P>`) as promissory notes.

**`Cargo.toml`** — standard mockspace template. Workspace dep: `arvo`. No
other arvo-family deps.

**`build.rs`** — `mockspace::bootstrap::bootstrap_from_buildscript()`.

### 4. `mock/crates/arvo-comb/DESIGN.md.tmpl`

**API sigs rewrite.** Document the new signatures:

```rust
// NEW
pub fn bin_pack<const N: Cap, const B: Cap, T, W>(
    items: &Array<T, N>,
    capacity: W,
    weight_of: impl Fn(&T) -> W,
    affinity: impl Fn(&T, &T) -> W,
) -> (USize, Array<USize, N>)
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant;

pub fn matrix_chain_dp<const N: Cap, W>(
    cost: impl Fn(USize, USize) -> W,
    feasible: impl Fn(USize, USize) -> Bool,  // was -> bool
) -> (W, Array<USize, N>)
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant;

pub fn greedy_group<const N: Cap, const M: Cap, A, T>(
    items: &Array<T, N>,
    feasible: impl Fn(&A, &T) -> Bool,  // was -> bool
    merge: impl Fn(A, &T) -> A,
    init: impl Fn() -> A,
) -> (USize, Array<Range, M>);
```

**Internal-implementation note.** DP table becomes `Matrix<Bool, N>` for the
feasibility/reachable state; `Matrix<W, N>` for costs; `Matrix<USize, N>` for
the split table.

**Dependencies.** arvo-comb now depends on arvo, arvo-tensor (new),
arvo-bitmask (existing — for cap_size re-export until arvo-tensor is
consumed transitively).

## Out of scope

- `Predicate<X>` trait alias — deferred to #122. This round's callback sigs return `Bool` (raw arvo newtype) but stay as `impl Fn(...) -> Bool` rather than `impl Predicate2<USize, USize>`.
- Migration of arvo-graph, arvo-sparse, arvo-spectral internals to `Array<T, N>` — each is its own sibling sweep round under #120 umbrella.
- `Grid<T, N, M>` / `Cube<T, N, M, P>` ranks-N shapes beyond Array + Matrix — tracked in arvo-tensor BACKLOG as future work.
- Downstream consumer migration (hilavitkutin/clause using arvo-tensor directly) — consumer concern, per downstream sweep rounds.
- `core::ops` / `Index<USize>` impls on Array/Matrix — could land this round as ergonomic convenience, but skipping for round-focus. Get/set methods suffice; can add later.
- Promotion of `cap_size` to a trait — `ArrayLen for Cap` or similar. Not needed this round; `cap_size(c)` function form is clear and matches existing pattern.
