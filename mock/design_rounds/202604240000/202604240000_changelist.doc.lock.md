## Purpose

Two no_std-fitting corrections before arvo-comb impl:

1. The existing DESIGN says `greedy_group` returns `Vec<(usize, usize)>`. arvo-comb is `#![no_std]`, no alloc — no `Vec`. Return a fixed-size `([Range; M], count)` tuple keyed on a caller-supplied maximum-groups const generic `M`.

2. `matrix_chain_dp`'s internal DP table has to be stack-allocated. Fix the signature so `N` (item count) is a const-generic and the DP table is `[[W; N]; N]`.

Both are mechanical no_std adaptations, no change to the algorithmic intent.

## Per-crate doc changes

### arvo-comb — `mock/crates/arvo-comb/DESIGN.md.tmpl`

- Rewrite the `greedy_group` signature block to:
  ```rust
  fn greedy_group<const N: usize, const M: usize, A, T>(
      items: &[T; N],
      feasible: impl Fn(&A, &T) -> bool,
      merge: impl Fn(A, &T) -> A,
      init: impl Fn() -> A,
  ) -> (USize, [Range; M])
  ```
  where `Range { start: USize, end: USize }` is defined in arvo-comb. Returns `(group_count, groups)`. `M` is the caller-supplied max-groups bound.
- Rewrite `matrix_chain_dp` signature to:
  ```rust
  fn matrix_chain_dp<const N: usize, W>(
      cost: impl Fn(USize, USize) -> W,
      feasible: impl Fn(USize, USize) -> bool,
  ) -> (W, [USize; N])
  where W: Add<Output = W> + TotalOrd + Copy + FromConstant
  ```
  Returns `(optimal_cost, split_points)`. Internal DP table is `[[W; N]; N]` stack-allocated.
- Rewrite `bin_packing` signature to:
  ```rust
  fn bin_pack<const N: usize, const B: usize, T, W>(
      items: &[T; N],
      capacity: W,
      weight_of: impl Fn(&T) -> W,
      affinity: impl Fn(&T, &T) -> W,
  ) -> (USize, [USize; N])
  where W: Add<Output = W> + TotalOrd + Copy + FromConstant
  ```
  Returns `(bin_count, bin_id_per_item)`. `B` is the caller's max-bins upper bound (allocation, not iteration count).
