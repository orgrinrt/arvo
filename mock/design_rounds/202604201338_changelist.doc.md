**Date:** 2026-04-20

**Topic:** `202604201338_topic.arvo-hash-new-crate.md`
**Type:** additive for arvo-hash; partial rule-adjustment for arvo-bits (narrows "no fielded structs" to "no arithmetic fielded structs — opaque-bit concretes are welcome").

## Motivation

Hashes (content hashes, name hashes, identity hashes) are numeric algorithms on opaque bit-patterns. The stack currently has no shared home for these; consumers (hilavitkutin-persistence, -str, future clause-*) use bare `u32`. The committed topic file resolves:

- Generic `Bits<const N: u8>` lives in **arvo-bits** (rule narrows: opaque-bit concretes are now welcome alongside traits; arithmetic fielded types still belong elsewhere).
- **arvo-hash** is a new L2 crate providing `ContentHash = arvo_bits::Bits<28>` alias + generic algorithm contracts (`HashAlgo<N>`, `Hasher<N>`).
- Concrete hash algorithms (FNV, xxHash, CRC32) deferred to follow-up rounds.

**Add a new section** documenting the `Bits<N>` type:

```
## Opaque-bit container: Bits<const N: u8>

`Bits<const N: u8>` is a fixed-width opaque bit pattern container.
Backing storage is `u64`; only the low N bits are addressable.
Non-arithmetic — compared by identity, not ordered arithmetically.
Compile-time asserted: N must be in 1..=64.

Implements `BitWidth` (reports N) and `BitAccess` (per-bit read).
Does not implement `BitSequence` — that trait's stream semantics
suit arithmetic contexts; opaque hashes use `bits()` for the full
pattern.

Primary consumer: arvo-hash's `ContentHash = Bits<28>` alias.
General-purpose: any domain that needs an opaque fixed-width
bit-pattern identity (future pass-key identity in clause, for
example).
```

Approx +25 lines in this section. Full text lands in doc execution.

#### 4d. `mock/crates/arvo-hash/BACKLOG.md.tmpl`

```
# arvo-hash — backlog

## Concrete algorithms

Shipping order (as consumers surface need):

- `Fnv1a28` — 28-bit FNV-1a; matches hilavitkutin-str's existing
  `const_fnv1a` + 28-bit mask. First migration target.
- `Fnv1a32`, `Fnv1a64` — wider FNV variants.
- `XxHash64` — when a faster non-crypto hash is wanted.
- `Crc32` — when checksum use cases surface (clause bytecode
  integrity, persistence file checksumming).

Each concrete algorithm lands as a unit struct implementing
`HashAlgo<N>` and (where streaming makes sense) `Hasher<N>`.

## Hilavitkutin-str migration

`hilavitkutin-str::const_fnv1a` is the existing FNV implementation.
Once `Fnv1a28` lands in arvo-hash, migrate hilavitkutin-str to
import + re-export that algo, and adjust downstream use sites.

## Wider bit widths

If `Bits<N>` in arvo-bits grows `u128` backing (its own BACKLOG
item), `arvo-hash` gains `Hash128` semantic aliases.

## Cryptographic hashes

Out of scope. arvo-hash targets non-cryptographic identity hashes.
A sibling `arvo-crypto` crate may appear if cryptographic hash
needs surface.
```
