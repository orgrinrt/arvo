# Topic: Fix Bits<N> container dispatch; add Bitfield<N>

**Date:** 2026-04-21
**Status:** TOPIC phase
**Tasks:** Bits<N> fix (newly discovered drift against original design); #127 Bitfield primitive

## Background — how we got here

The polka-dots origin design for arvo-bits (see
`mock/research/imported-from-polka-dots/arvo-bits/DESIGN.md.tmpl` —
or `~/Dev/polka-dots/mock/crates/arvo-bits/DESIGN.md.tmpl`) defines
the following N-bit value primitives:

```rust
pub type Bit<S: Strategy = Hot>   = UFixed<{IBits(1)},  {FBits(0)}, S>;
pub type Nibble<S: Strategy = Hot> = UFixed<{IBits(4)},  {FBits(0)}, S>;
pub type Byte<S: Strategy = Hot>  = UFixed<{IBits(8)},  {FBits(0)}, S>;
pub type Word<S: Strategy = Hot>  = UFixed<{IBits(16)}, {FBits(0)}, S>;
pub type DWord<S: Strategy = Hot> = UFixed<{IBits(32)}, {FBits(0)}, S>;
pub type QWord<S: Strategy = Hot> = UFixed<{IBits(64)}, {FBits(0)}, S>;
```

Each is a UFixed alias; container dispatch via `UContainerFor<S, BITS>`
picks the smallest native primitive that fits the logical width. Hot
+ 32 bits → u32. Hot + 8 bits → u8. That is the designed substrate.

**The originals have NO `Bits<N>` struct.** The current clause-dev
arvo-bits ships an extra type at `src/bits.rs`:

```rust
pub struct Bits<const N: u8>(u64);
```

Regardless of `N`, the inner container is `u64`. `Bits<32>` is 8
bytes. `Bits<8>` is 8 bytes. `Bits<1>` is 8 bytes. This is drift
against the original design, and it introduces the "weird cliffing"
where every opaque-bit type shares the same fat container.

The user surfaced this while porting `hilavitkutin-str::Str` to
`Bits<32>` — the 4-byte ABI guarantee in the Str DESIGN became 8
bytes. That blocked the hilavitkutin exact-widths round. Fixing
`Bits<N>` here unblocks that round.

The decision is not "rename the type to match reality" — the
**reality should match the original design**. `Bits<N>` is a useful
shape (opaque N-bit pattern, non-arithmetic) and it should use the
smallest container that fits, just as UFixed does.

## Part 1 — fix Bits<N> container dispatch

### Current (drift)

```rust
// src/bits.rs (L17)
pub struct Bits<const N: u8>(u64);

impl<const N: u8> Bits<N> {
    const MASK: u64 = if N == 64 { u64::MAX } else { (1u64 << N) - 1 };
    pub const fn new(raw: u64) -> Self { Self(raw & Self::MASK) }
    pub const fn bits(self) -> u64 { self.0 }
}
```

Always u64, regardless of N.

### Proposed

Introduce `BitContainerFor<const N: u8>` in arvo-bits mirroring
UFixed's `UContainerFor`:

```rust
pub trait BitContainerFor<const N: u8> {
    type T: Copy + PartialEq + Eq + PartialOrd + Ord + Default
          + core::hash::Hash + core::fmt::Debug + 'static;
}

pub struct BitContainer;

// 1..=8  -> u8
// 9..=16 -> u16
// 17..=32 -> u32
// 33..=64 -> u64

impl BitContainerFor<1> for BitContainer { type T = u8; }
impl BitContainerFor<2> for BitContainer { type T = u8; }
// ... etc. through 64, via a per-value macro (same pattern as
// UContainerFor in arvo/src/strategy.rs L166-L199).

#[repr(transparent)]
pub struct Bits<const N: u8>(<BitContainer as BitContainerFor<N>>::T)
where
    BitContainer: BitContainerFor<N>;
```

`Bits<32>` → u32 → 4 bytes. `Bits<28>` → u32 → 4 bytes. `Bits<8>` →
u8 → 1 byte. `Bits<64>` → u64 → 8 bytes. Matches the originals'
container-dispatch principle.

### Ripple to callers

- `Bits::new(...)` keeps accepting `u64` as input (widest representable
  value) and narrows to the container internally. No caller signature
  change.
- `Bits::bits()` keeps returning `u64`. Container is widened on read
  via `as u64`. No caller signature change.
- `size_of::<Bits<N>>()` now reflects the container, not always 8.
  Consumers that asserted 8 bytes (none found) would need updating;
  consumers that want N-bit identity (all current callers) benefit.
- `arvo-hash::ContentHash = Bits<28>` becomes 4 bytes instead of 8 —
  an improvement, no API change.
- `hilavitkutin-str::Str(Bits<32>)` becomes 4 bytes as originally
  documented — unblocks the hilavitkutin exact-widths round.

### Decision D1

Fix `Bits<N>` in place. Do not rename. The originals never had a
separate "opaque bit pattern" type because `UFixed` aliases covered
the named shapes (Bit/Nibble/Byte/...). But `Bits<N>` for arbitrary
`N` (e.g. 28-bit hashes) fills a gap the fixed aliases don't cover,
so it stays — with the correct container.

## Part 2 — add Bitfield<N, Layout>

Task #127 from an earlier session: Str's packed layout (bit 31
origin flag, bits 30-28 reserved, bits 27-0 ID) is a named-sub-range
use case. `Bits<N>` covers "N bits, no structure"; the packed-layout
pattern wants "N bits split into named sub-ranges with accessor
methods".

### Proposed

A declarative macro `bitfield!` that generates a struct with the
right container (via `BitContainer::T`) plus getter / setter methods
for each named sub-range, with compile-time checks that sub-ranges
are non-overlapping and fit within N.

```rust
use arvo_bits::bitfield;

bitfield! {
    /// Interned string handle. 32 bits total.
    pub struct StrHandle: 32 {
        /// 1 = runtime-interned, 0 = compile-time-interned.
        origin: 1 at 31,
        /// Reserved for future use.
        reserved: 3 at 28..=30,
        /// 28-bit interned identity.
        id: 28 at 0..=27,
    }
}

let h = StrHandle::new().with_origin(1).with_id(0x1234);
assert_eq!(h.origin(), 1);
assert_eq!(h.id(), 0x1234);
assert_eq!(core::mem::size_of::<StrHandle>(), 4); // 32 bits → u32 container
```

Generated struct:

```rust
#[repr(transparent)]
pub struct StrHandle(<BitContainer as BitContainerFor<32>>::T)
where BitContainer: BitContainerFor<32>;
```

With inherent methods:

```rust
impl StrHandle {
    pub const BITS: u8 = 32;
    pub const fn new() -> Self;
    pub const fn from_bits(raw: u64) -> Self;
    pub const fn to_bits(self) -> u64;

    pub const fn origin(self) -> u8;
    pub const fn with_origin(self, value: u8) -> Self;
    pub const fn reserved(self) -> u8;
    pub const fn with_reserved(self, value: u8) -> Self;
    pub const fn id(self) -> u32;
    pub const fn with_id(self, value: u32) -> Self;
}
```

Sub-range fields are typed as the smallest primitive that fits the
sub-range width (1 bit → u8, 28 bits → u32). Setters mask the input
to the sub-range's width before writing, so garbage-high-bits callers
don't corrupt other fields.

### Compile-time checks

- Sub-ranges must be disjoint — overlapping bits are a compile error.
- Every sub-range must fit within `N` — bit indices ≥ N are a compile
  error.
- Total N must be ≤ 64 (same as `Bits<N>` bound).
- Gaps (unused bits) are allowed; they read as zero and setters leave
  them alone.

### Why a macro, not a struct with a trait parameter

Named accessor methods per field require code generation; a generic
`Bitfield<L: Layout>` could only expose `get(idx)` / `set(idx, value)`
positional accessors, which defeats the "named sub-ranges at type
level" point. A `macro_rules!` implementation fits arvo's no-proc-
macro constraint; complex const evaluation (overlap detection) lives
in `const fn` helpers inside the macro's generated code, checked via
`const _: () = assert!(...)`.

### Decision D2

`bitfield!` is a declarative macro under `arvo-bits`, generating
`repr(transparent)` structs over the correct `BitContainerFor<N>::T`
container. No runtime dispatch, no trait objects, all bit-shifting
`const fn`.

### Out of scope for this round

- Separate unsigned/signed bitfields. Sub-ranges are unsigned bit
  patterns; if a sub-range needs sign semantics, that's a future
  round.
- Bitfield ↔ UFixed arithmetic interop. Bitfield is opaque identity
  like `Bits<N>`; no arithmetic ops.
- `Copy` / `Eq` derive semantics beyond what the container gives for
  free via `#[derive]` on the generated struct.
