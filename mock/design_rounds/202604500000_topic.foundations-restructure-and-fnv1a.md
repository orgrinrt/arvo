This topic supersedes the prior corrected topic. An independent design review (recorded 2026-04-26 against the deprecated doc CL) surfaced three critical defects: type-level recursion in the `Uint7` / `IBits` / `FBits` chain (Uint7 wraps UFixed which wraps IBits which wraps Uint7), unverified `ConstParamTy` claims on `UFixed` / `IFixed` (the `PhantomData<fn() -> S>` field and associated-type projections both block the trait), and an unbounded `N` on `Fnv1a<const N: BitWidth>` that compiles past 64 bits today by accident. Plus several half-assed migration commitments and one ergonomic trap.

This topic addresses every finding by fix-not-drop. Per workspace convention `.claude/rules/no-bare-primitives.md`, the discipline goal stays absolute: no bare primitives in any non-exception position. The fix is to restructure the dependency graph so the discipline holds AND the type tower bottoms out cleanly without recursion. The round's scope expands to include the restructure; the alternative (defer to a follow-up round) was rejected.

## Decisions

### Restructure: `Bits<const N: u8>` moves from arvo-bits L1 to arvo L0

The reviewer's C1 finding: `Uint7 = UFixed<{IBits(7)}, {FBits(0)}, Hot>` is circular if `IBits(pub Uint7)`. The bottom of the type tower must have one place where const-generic position bottoms out at a bare primitive. That ONE place becomes `Bits<const N: u8>` at L0:

- arvo L0 hosts: `Bits<const N: u8, S: Strategy>`, `UFixed`, `IFixed`, all numeric primitives + meta-types (`IBits`, `FBits`, `Width`), strategy markers, ergonomic helpers.
- arvo-bits L1 stays: trait contracts only (`HasBitWidth` trait, `BitAccess`, `BitSequence`, `BitLogic`) plus blanket impls for arvo's primitive types.

This is the architecturally clean shape. Today's "Bits container + bit traits both at L1" is a layering compromise; the round corrects it. arvo-bits's `forbidden-imports` lint (currently forbidding arvo from `arvo_bits::*`) becomes irrelevant for the moved item, since `Bits` is no longer in arvo-bits to import. The lint stays for genuine arvo-bits-specific items; cross-check at SRC time.

The `Bits<const N: u8>` const-generic parameter remains a bare `u8` because this IS the bottom of the type tower (workspace rule exception #2: storage primitive's width parameter at the very root). All consumers reach `Bits<N>` via the `width_of` ergonomic helper that wraps the bare primitive.

### Meta-types `IBits`, `FBits`, `Width` wrap `Bits<7, Hot>`

```rust
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
pub struct IBits(pub Bits<7, Hot>);

#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
pub struct FBits(pub Bits<7, Hot>);

#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
pub struct Width(pub Bits<7, Hot>);
```

`Bits<7, Hot>` (zero-extended in the 8-bit container per `Hot: UContainerFor<7>`) is the smallest arvo-typed value range that covers 0-127, with one bit of headroom over the realistic 0-64 we need for arvo widths. Distinct newtypes prevent positional swaps on `UFixed` turbofish.

No recursion: `Bits<const N: u8>` does not depend on `IBits` / `FBits` / `Width`. The dependency direction is `Bits -> {IBits, FBits, Width} -> {UFixed, IFixed} -> {Uint*, Int* aliases}`, strictly downward.

`Bits<7, Hot>` derives `ConstParamTy` cleanly because it is `repr(transparent)` over a u8 container (which is `ConstParamTy`), with no projections or lifetime-carrying phantoms.

### Rename: new const-generic-param newtype is `Width`, not `BitWidth`

H4 collision fix. The existing `arvo_bits::BitWidth` *trait* (it asserts "this type carries a bit-width") keeps that name, possibly renamed for clarity to `HasBitWidth` if the SRC phase finds the rename is more orthogonal. The new const-generic parameter newtype is `Width` (it IS a width, expressed as a value in const-generic position). Mechanically free now in DOC; painful retrofit later.

### `ConstParamTy` on `UFixed` / `IFixed` via composition, not projection

C2 fix. The blockers per the reviewer:

1. `PhantomData<fn() -> S>` field. Fn-ptrs are not `ConstParamTy`. Fix: change to `PhantomData<S>`. Strategy markers are unit structs; covariant is fine because S is a marker type, never instantiated as a value. The variance change is a side benefit (fewer unsoundness foot-guns).

2. Associated-type projection `<S as UContainerFor<N>>::T` in storage field. ConstParamTy structural-eq cannot reason through projections. Fix: route the storage type via composition over `Bits<N, S>` rather than via a Strategy associated-type projection. Concretely, post-restructure UFixed/IFixed wrap the appropriate `Bits<N, S>`:

   ```rust
   #[repr(transparent)]
   pub struct UFixed<const I: IBits, const F: FBits, S: Strategy = Warm> {
       bits: Bits<{ ufixed_bits(I, F) }, S>,
       _phantom: PhantomData<S>,
   }
   ```

   The projection moves into Bits. Bits's container choice happens INSIDE Bits via `UContainerFor<const N: u8, S>::T`, but Bits's structural identity is just `Bits<N, S>` over its single u8 const-generic and its zero-sized strategy marker. ConstParamTy on Bits derives cleanly. ConstParamTy on UFixed/IFixed cascades through Bits + IBits + FBits + S (all ConstParamTy).

3. `Strategy` markers (`Hot` / `Warm` / `Cold` / `Precise`) themselves are unit structs that derive `ConstParamTy + PartialEq + Eq + Copy + Clone + Debug`. They already derive most of these; add `ConstParamTy`. Verify on current nightly during SRC.

If the cascade still fails on current nightly after these fixes, the workaround is a manual `unsafe impl ConstParamTy` for UFixed/IFixed with a doc comment establishing the soundness chain (transparent repr cascading through Bits to a u8 container; all type params are ConstParamTy; structural eq is well-defined by transparent layout). Document the soundness explicitly; do not handwave.

### Numeric aliases generic over Strategy with `Warm` default

H5 fix. UFixed-based aliases ship as generic over S with `Warm` default (matching `UFixed::default()`). For widths where Warm is gated (I+F > 32), default-Warm produces a clean compile error pointing the consumer at the strategy choice, better than silent Hot semantics.

```rust
pub type Uint5<S = Warm>  = UFixed<{ ibits(5) },  { fbits(0) }, S>;
pub type Uint6<S = Warm>  = UFixed<{ ibits(6) },  { fbits(0) }, S>;
pub type Uint7<S = Warm>  = UFixed<{ ibits(7) },  { fbits(0) }, S>;
pub type Uint8<S = Warm>  = UFixed<{ ibits(8) },  { fbits(0) }, S>;
pub type Uint16<S = Warm> = UFixed<{ ibits(16) }, { fbits(0) }, S>;
pub type Uint32<S = Warm> = UFixed<{ ibits(32) }, { fbits(0) }, S>;
pub type Uint64<S = Warm> = UFixed<{ ibits(64) }, { fbits(0) }, S>;

pub type Int8<S = Warm>   = IFixed<{ ibits(8) },  { fbits(0) }, S>;
pub type Int16<S = Warm>  = IFixed<{ ibits(16) }, { fbits(0) }, S>;
pub type Int32<S = Warm>  = IFixed<{ ibits(32) }, { fbits(0) }, S>;
pub type Int64<S = Warm>  = IFixed<{ ibits(64) }, { fbits(0) }, S>;
```

Strategy is a storage / operational hint; the same numeric domain (`Uint64`) is roughly equivalent across strategies post-deserialisation. `Uint64<Hot>` and `Uint64<Warm>` (where Warm exists) interchange trivially at the value level; differences are arithmetic semantics and serialisation footprint. The alias names carry the domain; the strategy parameter carries the operational shape.

`Bits<N, S>` aliases (`Bit`, `Nibble`, `Byte`, `Word`, `DWord`, `QWord`) keep `S = Hot` default. Bit-level work is structurally different. Consumers want tight wrapping semantics, not Warm's development-friendly defaults. The two default conventions match the two type-family identities.

### `USize` as newtype `pub struct USize(pub Bits<{cfg-platform-width}, Hot>)`

H3 fix. Today's `pub struct USize(pub usize)` migrates to a newtype over a cfg-gated `Bits` instantiation:

```rust
#[cfg(target_pointer_width = "64")]
pub struct USize(pub Bits<64, Hot>);

#[cfg(target_pointer_width = "32")]
pub struct USize(pub Bits<32, Hot>);
```

Construction syntax preserved (`USize(some_bits_value)`), Deref preserved (now `Deref<Target = Bits<N, Hot>>`), tuple-struct shape preserved, `Cap(pub USize)` works unchanged. The bare `usize` field is gone; the platform-pointer-width difference handled at the cfg boundary inside the newtype rather than at an alias boundary that consumer code would have to deal with.

Cap stays as today: `pub struct Cap(pub USize)`. ConstParamTy chain: USize → Bits<64, Hot> → u8 const-generic + Hot marker. Clean.

### `Fnv1a<const N: Width>` with explicit `where N <= 64` guard + Fnv1a128 BACKLOG

C3 fix. The const-generic guard pattern enforces the algorithmic upper bound:

```rust
pub struct Fnv1a<const N: Width>
where
    Hot: UContainerFor<N>,
    [(); 1 / ((width_le_64::<N>()) as usize)]:; // const-fail when N > 64

const fn width_le_64<const N: Width>() -> bool {
    width_u8(N) <= 64
}
```

Plus a sealed trait alternative if the const-generic guard is too noisy at consumer sites:

```rust
pub trait Fnv1aWidth: sealed::Sealed {}
// impls for Width values 1..=64 only
```

Pick one in SRC iteration; both express the same constraint.

BACKLOG entry: "Fnv1a128 algorithm. Separate state, not a mask of FNV-1a-64. Required when 128-bit hash widths surface; today the 64-bit accumulator caps Fnv1a<N> at N <= 64."

### Const-fn ergonomic helpers + `Bits<7, Hot>::from_raw_u8`

C4 + H7 fix. The helpers explicitly route through Bits's const-fn constructor:

```rust
pub const fn ibits(n: u8) -> IBits { IBits(Bits::<7, Hot>::from_raw_u8(n)) }
pub const fn fbits(n: u8) -> FBits { FBits(Bits::<7, Hot>::from_raw_u8(n)) }
pub const fn width_of(n: u8) -> Width { Width(Bits::<7, Hot>::from_raw_u8(n)) }
```

`Bits<N, S>::from_raw_u8` (and per-container-width siblings `from_raw_u16`, `from_raw_u32`, `from_raw_u64`) ship as inherent const-fn constructors on Bits in this round. The bare-`u8`-in-helper-parameter is workspace rule exception #4 (ergonomic helper-fn parameter); the bare-`u8`-in-`from_raw_u8`-parameter is a definition-site exception (Bits IS the storage primitive accepting raw bytes from the platform).

Mask body in `Fnv1a::<N>::hash_const`:

```rust
pub const fn hash_const(bytes: &[u8]) -> Bits<N, Hot>
where Hot: UContainerFor<N>
{
    let raw = fnv1a_64(bytes);
    let n: u8 = width_u8(N);
    let mask: u64 = if n == 64 { u64::MAX } else { (1u64 << n) - 1 };
    Bits::<N, Hot>::from_raw_u64(raw & mask)
}
```

`width_u8(Width) -> u8` is a const-fn helper next to `ibits_u8` / `fbits_u8` (already in `strategy.rs` for the same reason, currently `lint:allow(no-bare-numeric)` tracked at #71). After the round these helpers route through `Bits<7, Hot>` access.

Note on `width_u8`: the function reads `Width.0`'s inner `Bits<7, Hot>` field and produces a `u8` for the anonymous-const expression context. This is exactly the same pattern as `ibits_u8` / `fbits_u8`. The bare `u8` return is workspace rule exception #4 (ergonomic helper enabling a const-eval position that requires bare-primitive access). Document on the helper.
