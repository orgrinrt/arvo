# Doc Changelist: arvo-bitmask Cap/USize cleanup

**Date:** 2026-04-26
**Phase:** DOC
**Scope:** `arvo-bitmask` crate only.
**Source topics:**
- 202603241400_topic.arvo-crate-dependency-revision.md (generic contracts)

## Purpose

Retroactive cleanup: arvo-bitmask's initial impl round used `const N:
usize` on public surfaces against DESIGN.md.tmpl's `const N: Cap`.
This round aligns the src with the design — mirrors the correction
made in arvo-spectral (round 202604250000) but applied retroactively
to the first L2 crate.

Two corrections:

1. **`const N: Cap`, not `const N: usize`.** `BitMatrix64`,
   `BitMatrix256`, and both `propagate_dirty_*` functions take the
   arvo `Cap` newtype (wraps `USize` wraps `usize`) as const-generic
   size parameter. Array-sizing inside struct fields / method bodies
   extracts via `cap_size(c: Cap) -> usize { c.0.0 }` — inline `.0.0`
   is rejected by nightly `generic_const_exprs`.

2. **`USize` over `usize`** in all count / index positions on the
   public API. `NodeId(pub USize)` is already correct; residual raw
   `usize` inside method bodies stays (it's strictly internal), but
   any public method signature gets the newtype.

No algorithmic changes. Storage layout unchanged
(`[Mask64; cap_size(N)]` equivalent to the previous `[Mask64; N]`
under `const N: usize`).

## Per-crate doc changes

### arvo-bitmask — `mock/crates/arvo-bitmask/DESIGN.md.tmpl`

- Pluralise the bit-matrix type entry from `BitMatrix<const N: Cap>`
  to `BitMatrix64<const N: Cap>` and `BitMatrix256<const N: Cap>` —
  matches the concrete types shipped in src. `BitMatrix` is not a
  real type; keeping it confuses the `design-doc-source-mismatch`
  lint.

### `mock/crates/arvo-bitmask/BACKLOG.md.tmpl` (new file)

- Add "cap_size helper promotion to arvo": the `cap_size(c: Cap) ->
  usize` helper currently lives local to arvo-spectral and will live
  local to arvo-bitmask after this round. Promote to arvo (L0) when a
  third L2 crate needs it; L0 promotion is additive and trivial (one
  const fn).
- Add "`Mask256`-only cleanups" note: the `*_256` variants get the
  same Cap/USize treatment as the 64-variants. No separate deferral.

## Files NOT changed

  promotion tracked in the new BACKLOG entry.

  newtype-at-boundaries lint that would have caught this earlier.
- Mockspace lint-rules.
