**Round:** 202604500000. BitWidth ConstParamTy newtype, plus concrete `Fnv1a` hasher

Introduces a `BitWidth` ConstParamTy newtype as the canonical const-generic parameter type for bit-count positions, paralleling existing `IBits` / `FBits` for fixed-point integer / fractional widths. Migrates `Bits<const N: BitWidth>` and `Hasher<const N: BitWidth>` from bare `u8`. Lands `Fnv1a<const N: BitWidth>` as the first concrete `Hasher` impl, with a const-fn entry point usable from compile-time positions (consumed by the hilavitkutin app-builder formalisation round downstream).

The discipline gap closed by this round: arvo's stated rule is that bare primitives never appear in public positions. The width parameter on `Bits<>` and `Hasher<>` was shipped as bare `u8` because no `BitWidth` newtype existed at the time. Lint allowlisting hid the gap. This round corrects it.

## Decisions

### `BitWidth(u8)` lives in arvo (L0)

Same module as `IBits` and `FBits` (`arvo::newtype`). ConstParamTy + PartialEq + Eq + Copy + Clone + Debug. Tuple struct over `u8`. `pub const fn new(v: u8) -> Self`, `pub const fn raw(self) -> u8`. Standard pattern matching the existing `IBits` / `FBits` shape verbatim.

The bare `u8` inside the newtype struct is the canonical exception: definition-site of the primitive itself. No lint-allow comment needed; this is exactly the position the discipline permits.

### Migrate `Bits<const N: BitWidth>` in arvo-bits

`Bits<const N: u8, S: Strategy = Hot>` becomes `Bits<const N: BitWidth, S: Strategy = Hot>`. Per-width aliases stay where they are; bodies migrate to `BitWidth(N)` form:

```rust
pub type Bit<S = Hot>    = Bits<{ BitWidth(1) }, S>;
pub type Nibble<S = Hot> = Bits<{ BitWidth(4) }, S>;
pub type Byte<S = Hot>   = Bits<{ BitWidth(8) }, S>;
pub type Word<S = Hot>   = Bits<{ BitWidth(16) }, S>;
pub type DWord<S = Hot>  = Bits<{ BitWidth(32) }, S>;
pub type QWord<S = Hot>  = Bits<{ BitWidth(64) }, S>;
```

Every `Bits<8>` / `Bits<16>` / `Bits<32>` / `Bits<64>` site across arvo gets the same `{ BitWidth(N) }` rewrite. `BitWidth` is `pub use`d from `arvo` and re-exported through `arvo-bits` for ergonomic reach.

### Migrate `Hasher<const N: BitWidth>` in arvo-hash

`Hasher<const N: u8>` and `HasherExt<const N: u8>` become `Hasher<const N: BitWidth>` and `HasherExt<const N: BitWidth>`. The `Hot: UContainerFor<N>` bound continues to apply — `UContainerFor` migrates the same way (`UContainerFor<const N: BitWidth>`).

`ContentHash = Bits<{ BitWidth(28) }, Hot>` per the new alias form.

### `UContainerFor<const N: BitWidth>` in arvo-bits

The `UContainerFor` trait (whichever crate defines it; appears to be arvo-bits or arvo) carries the same migration. Bound sites in arvo-bits / arvo-bitmask / arvo-tensor / arvo-graph / arvo-sparse / arvo-comb / arvo-spectral / arvo-hash all update from `Hot: UContainerFor<N>` (with N: u8) to the BitWidth form.

### Concrete `Fnv1a<const N: BitWidth>` in arvo-hash

```rust
pub struct Fnv1a<const N: BitWidth>
where Hot: UContainerFor<N>;

impl<const N: BitWidth> Hasher<N> for Fnv1a<N>
where Hot: UContainerFor<N>
{
    fn update(&mut self, bytes: &[u8]) { /* runtime FNV-1a-64 mask-down to N */ }
    fn finalize(self) -> Bits<N> { /* ... */ }
}

impl<const N: BitWidth> Fnv1a<N>
where Hot: UContainerFor<N>
{
    /// Compile-time entry point: hash a byte slice, mask to N bits.
    pub const fn hash_const(bytes: &[u8]) -> Bits<N> { /* const-fn FNV-1a-64 + mask */ }
}
```

Algorithm: canonical FNV-1a-64 (offset basis `0xcbf29ce484222325`, prime `0x100000001b3`). For N < 64, mask the low-N bits of the 64-bit accumulator (matches `hilavitkutin-str::const_fnv1a` semantics for the 28-bit `Str` case).

`hash_const` is the const-fn form. `Hasher::update` / `finalize` is the runtime form. Both use the same algorithm; the const-fn version is what consumer compile-time-validation code relies on.

### Promote `ContentHash` to point at `Fnv1a` output

`ContentHash` stays `Bits<{ BitWidth(28) }, Hot>` (alias form). Documentation cross-links to `Fnv1a` as the producer.

### Out of scope

- BitWidth-aware lint upgrades. The position-aware lint enhancement is separately tracked and will land with the broader lint overhaul. This round closes the substantive gap (the bare `u8` in width positions) by migrating arvo's source; the lint catches future drift.
- `Fnv1a32`, `Fnv1a28` named aliases. Once `Fnv1a<N>` ships, consumers instantiate at any width directly. Add aliases only when a concrete consumer wants the named form.
- Other hash algorithms (xxHash, CRC32). Stay BACKLOG.
- Migration of downstream consumer crates outside arvo. Each consumer repo handles its own migration in a follow-up round.
