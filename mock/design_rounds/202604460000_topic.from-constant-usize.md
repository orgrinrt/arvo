**Phase:** TOPIC
**Scope:** arvo::traits::FromConstant — take USize instead of u8 (#123)

# Current state

`arvo::traits::FromConstant` was designed around small integer
constants (`from_constant(1)`, `from_constant(0)`):

```rust
pub trait FromConstant {
    fn from_constant(n: u8) -> Self;
}
```

The `u8` parameter is a design-time bandage from when arvo had no
semantic alias for counts. Every consumer passes an integer
literal; nobody needs the 0..=255 range per se. Consumers that
carry counts through call sites have to widen-then-narrow (see
`arvo-spectral/src/fiedler.rs:92` — `n as u8 // lint:allow
tracked: #123`).

USize (wrap around host usize) is the correct semantic type: count /
index that the caller already carries. Lets us drop the boundary
cast in fiedler and keep the trait honest about what the operand is.

# Changes proposed

- Trait signature: `fn from_constant(n: USize) -> Self`.
- All arvo-internal impls: body takes `n.0 as $ctype` instead of
  `n as $ctype`. The `0` for USize is `n.0` (tuple-struct field
  access).
- `arvo::USize` imported in every consumer crate where needed.
- Consumer call-site shape: `F::from_constant(USize(0))` etc.
- `fiedler.rs:92` drops the `n as u8` cast — the value is already
  `USize`-typed (`n` is the node count via `USize` in the plan).
- Drop the `tracked: #123` `lint:allow` annotation on the fiedler
  line.
- Doc table at top of traits.rs updates `u8` → `USize`.

# Non-goals

- No trait-method rename.
- No additional constructors.
- No `from_constant` for values > 255. USize inputs are still
  expected to fit the concrete type's range; out-of-range truncates
  at the container per existing contract.
