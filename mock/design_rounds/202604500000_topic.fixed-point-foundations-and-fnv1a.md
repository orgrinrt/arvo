This topic supersedes the prior topic for round 202604500000. The prior topic landed `BitWidth(u8)` as the corrective newtype. That was wrong: arvo's discipline is fixed-point everything, and "bare u8 inside a newtype" is not the floor — UFixed-based aliases are. Per workspace rule `no-bare-primitives.md`, this topic re-grounds the round in the actual discipline and expands the scope to cover what should have been one coherent reach: introduce the UFixed-based alias layer at L0, migrate the meta-types (`IBits`, `FBits`, new `BitWidth`) to wrap the new aliases, migrate `Bits<>` and `Hasher<>` const-generic positions, ship `Fnv1a` concrete, sweep arvo's internal source for bare primitives in non-definition positions.

## What changed in our understanding

Three points of arvo internal architecture that the prior topic missed:

1. **UFixed is the foundation, not the leaf.** `UFixed<I, F, S>` and `IFixed<I, F, S>` lower to bare primitives via `#[repr(transparent)]`, with the container chosen at codegen by `Strategy` and notko profile. They are codegen-transparent zero-cost wrappers. Every other arvo numeric type should lift above them, not parallel to them or pin specific bare primitives.

2. **Exact widths, not byte buckets.** A meta-type describing UFixed widths (`IBits`, `FBits`, `BitWidth`) needs a value range up to 64ish. `Uint7 = UFixed<{IBits(7)}, {FBits(0)}, Hot>` covers 0-127 with one bit of headroom. Forcing `Uint8` (or worse, bare `u8`) is the byte-bucket habit the substrate exists to break.

3. **`ConstParamTy` is sound on UFixed via the transparent repr.** Stable Rust's structural-equality requirement is satisfiable when the inner container is structurally-eq-compatible (which `u8` / `u16` / etc. are) and the wrapping is transparent. `feature(adt_const_params)` is already enabled. UFixed gets a `ConstParamTy` impl (derived where it cascades cleanly, manual otherwise). Then `const N: Uint7` works as a const-generic parameter. The prior topic's `BitWidth(u8)` choice came from "ConstParamTy needs bare integers" — it doesn't, given the transparent lowering.

## Decisions

### New arvo L0 alias layer: unsigned and signed UFixed aliases

`arvo/src/aliases.rs` (new file or extension of an existing one) ships:

```rust
pub type Uint5  = UFixed<{ IBits(5) },  { FBits(0) }, Hot>;
pub type Uint6  = UFixed<{ IBits(6) },  { FBits(0) }, Hot>;
pub type Uint7  = UFixed<{ IBits(7) },  { FBits(0) }, Hot>;
pub type Uint8  = UFixed<{ IBits(8) },  { FBits(0) }, Hot>;
pub type Uint16 = UFixed<{ IBits(16) }, { FBits(0) }, Hot>;
pub type Uint32 = UFixed<{ IBits(32) }, { FBits(0) }, Hot>;
pub type Uint64 = UFixed<{ IBits(64) }, { FBits(0) }, Hot>;

pub type Int8   = IFixed<{ IBits(8) },  { FBits(0) }, Hot>;
pub type Int16  = IFixed<{ IBits(16) }, { FBits(0) }, Hot>;
pub type Int32  = IFixed<{ IBits(32) }, { FBits(0) }, Hot>;
pub type Int64  = IFixed<{ IBits(64) }, { FBits(0) }, Hot>;
```

Narrow-bit unsigned aliases (`Uint5`, `Uint6`, `Uint7`) are first-class because the substrate's whole point is exact widths, not byte buckets. Strategy `Hot` is the default for these; consumers needing different containers reach for `UFixed<...>` directly with the right `S` parameter.

`USize`: cfg-gated alias `pub type USize = Uint64` on 64-bit targets, `Uint32` on 32-bit. The existing `pub struct USize(pub usize)` newtype is replaced by this alias. `Cap` migrates to wrap the new `USize`.

### `UFixed` / `IFixed` get `ConstParamTy` impls

Derive `ConstParamTy` (with `feature(adt_const_params)`) on `UFixed` and `IFixed`. Required derives on the surrounding types (`IBits`, `FBits`, `Strategy` markers) already exist. The transparent repr makes the structural equality sound at the layout level; the type-level impl is the missing piece.

Where `derive(ConstParamTy)` doesn't cascade cleanly (likely because of generic parameters), a manual `unsafe impl ConstParamTy for UFixed<...>` lands with a doc comment explaining why it's sound (transparent lowering, all type parameters are themselves ConstParamTy).

### Migrate `IBits` / `FBits` / introduce `BitWidth`

```rust
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
pub struct IBits(pub Uint7);  // was: pub u8

#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
pub struct FBits(pub Uint7);  // was: pub u8

#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
pub struct BitWidth(pub Uint7);  // new
```

Each wraps `Uint7` (covers 0-127, one bit of headroom over the realistic 0-64 range). The `pub Uint7` field is the inner arvo type; the `Uint7` itself bottoms out at the UFixed container (#2 in the rule's exception list), not at this newtype. So `IBits` / `FBits` / `BitWidth` no longer carry a bare-primitive field.

Distinct newtypes (not collapsed to a shared alias) so `I` and `F` can't be swapped on a `UFixed` turbofish without a compile error. The newtype tax is one definition each; the win is type-checked positional safety.

### Ergonomic constructor helpers

```rust
pub const fn ibits(n: u8) -> IBits { IBits(Uint7::from_u8(n)) }
pub const fn fbits(n: u8) -> FBits { FBits(Uint7::from_u8(n)) }
pub const fn width_of(n: u8) -> BitWidth { BitWidth(Uint7::from_u8(n)) }
```

The bare `u8` parameter is the call-site exception (#4 in the rule) — without it, every literal at every turbofish requires `Uint7::from_u8(...)` boilerplate. With these helpers, consumer turbofish stays readable:

```rust
Bits<{ width_of(12) }, Hot>
UFixed<{ ibits(8) }, { fbits(0) }, Hot>
Hasher<{ width_of(28) }>
```

`Uint7::from_u8` (or equivalent constructor) is itself the canonical place where a bare `u8` literal becomes a `Uint7`. This is the deepest the bare primitive reaches; everything past the helpers is arvo-typed.

### `Bits<const N: BitWidth>` migration

`arvo-bits::Bits<const N: u8, S: Strategy = Hot>` becomes `Bits<const N: BitWidth, S: Strategy = Hot>`.

Per-width aliases update bodies:

```rust
pub type Bit<S = Hot>    = Bits<{ width_of(1) },  S>;
pub type Nibble<S = Hot> = Bits<{ width_of(4) },  S>;
pub type Byte<S = Hot>   = Bits<{ width_of(8) },  S>;
pub type Word<S = Hot>   = Bits<{ width_of(16) }, S>;
pub type DWord<S = Hot>  = Bits<{ width_of(32) }, S>;
pub type QWord<S = Hot>  = Bits<{ width_of(64) }, S>;
```

`UContainerFor<const N: BitWidth>` migrates the same way. Every per-width impl (`UContainerFor<{ width_of(8) }>`, `UContainerFor<{ width_of(16) }>`, etc.) lands as a substitution of the previous `UContainerFor<8>` form.

### `Hasher<const N: BitWidth>` migration

`arvo-hash::Hasher<const N: u8>` becomes `Hasher<const N: BitWidth>`. `HasherExt<const N: BitWidth>` and the blanket impl follow. `ContentHash = Bits<{ width_of(28) }, Hot>`.

### Ship concrete `Fnv1a<const N: BitWidth>` in arvo-hash

```rust
pub struct Fnv1a<const N: BitWidth>
where Hot: UContainerFor<N>;

impl<const N: BitWidth> Hasher<N> for Fnv1a<N>
where Hot: UContainerFor<N>
{
    fn update(&mut self, bytes: &[u8]) { /* runtime FNV-1a-64, mask to N bits */ }
    fn finalize(self) -> Bits<N> { /* ... */ }
}

impl<const N: BitWidth> Fnv1a<N>
where Hot: UContainerFor<N>
{
    /// Compile-time entry point: hash a byte slice, mask to N bits.
    pub const fn hash_const(bytes: &[u8]) -> Bits<N> { /* const-fn FNV-1a-64 + mask */ }
}
```

Algorithm: canonical FNV-1a-64 (offset basis `0xcbf29ce484222325`, prime `0x100000001b3`). For N < 64, mask the low-N bits of the 64-bit accumulator. Matches `hilavitkutin-str::const_fnv1a` semantics for the 28-bit `Str` case (which `ContentHash = Bits<{ width_of(28) }, Hot>` aliases).

`hash_const` is the const-fn entry point. `Hasher::update` / `finalize` is the runtime form. Both use the same algorithm; the const-fn version is what compile-time-validation code (downstream hilavitkutin app-builder formalisation) consumes.

The `bytes: &[u8]` parameter is acceptable here under exception #4 (ergonomic helper-fn parameter accepting a literal-shaped value): `&[u8]` is the canonical byte-slice shape produced by `core::any::type_name::<T>().as_bytes()` and similar const-fn sources. Wrapping in a `ByteEmitter`-style boundary at this level is for a follow-up round.

### Sweep arvo internal source for bare primitives

Walk every `.rs` file in every arvo crate. Replace bare primitives in non-definition positions with the new aliases:

- function arguments and return types → `Uint*` / `Int*` / `USize` / `Bool` / appropriate alias
- struct fields (unless on the exception list) → arvo types
- type aliases (when the body uses bare primitives) → arvo types
- const declarations → arvo types
- closure types → arvo types
- internal let bindings where the type is named explicitly → arvo types
- where clauses and trait bounds → arvo types

Where a `lint:allow(no-bare-numeric)` or `lint:allow(arvo-types-only)` exists today on a position now covered by the new aliases, remove the allow comment along with the migration.

The exception positions stay bare:
- `UFixed` / `IFixed` container fields (the transparent-repr root, exception #2)
- `Bool` / platform-binding wrappers (exception #3)
- Helper-fn parameters that construct arvo types from literals (exception #4)
- Std-trait method signatures impl'd on our types (exception #5)
- FFI `#[repr(C)]` fields (exception #6)

Everything else gets migrated.

### Out of scope

- Lint upgrades. The position-aware lint refinement (allow primitives only in the six definition-site positions, block elsewhere) belongs to the broader lint overhaul tracked separately. This round closes the substantive gap by sweeping arvo's source clean; the lint catches future drift once it ships.
- `Fnv1a32` / `Fnv1a28` named aliases. With `Fnv1a<const N: BitWidth>`, consumers instantiate at any width. Add named aliases when a concrete consumer surfaces.
- Other hash algorithms (xxHash, CRC32). BACKLOG.
- Migration of downstream consumer crates (hilavitkutin, vehje, viola, notko). Each consumer repo handles its own migration in a follow-up round, tracked in their respective task queues.
