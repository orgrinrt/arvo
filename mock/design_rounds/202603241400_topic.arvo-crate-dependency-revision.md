# Topic: arvo Crate Dependency Revision — arvo-bits and Generic Numeric Contracts

**Date:** 2026-03-24
**Status:** IN PROGRESS — being built interactively

## Purpose

During src planning for the doc CL, we discovered that the arvo
sub-crate dependency structure needs revision. The original structure
had arvo-bitmask as a standalone crate depending directly on arvo.
But bitmasks are not numeric types — they're bit-level types with
fundamentally different operations (OR not Add, AND not Mul). This
topic introduces arvo-bits as the bit-level contract crate and
revises the dependency tree. It also formalises that the mathematical
sub-crates (graph, sparse, spectral, comb) are generic over numeric
trait bounds, not concrete arvo types.

---

## The problem with the original structure

The original arvo crate family (from the consolidation topic):

```
arvo              (no deps)
arvo-bitmask      (deps: arvo)
arvo-sparse       (deps: arvo, arvo-bitmask)
arvo-graph        (deps: arvo, arvo-bitmask)
arvo-spectral     (deps: arvo-sparse)
arvo-comb         (deps: arvo, arvo-bitmask)
```

Issues:

1. **arvo-bitmask depends on arvo for what?** Bitmasks don't use
   UFixed/IFixed. They don't do arithmetic. They do bitwise ops.
   The dependency was for the "arvo-types-only lint" but a bitmask
   is not a number.

2. **No bit-level contract layer.** The traits for "this type is a
   sequence of bits" (bit access, popcount, trailing zeros) don't
   exist. arvo defines numeric traits (Add, Sqrt, TotalOrd). The
   bit traits are a different concern.

3. **Sub-crate numeric generics unclear.** arvo-graph's weight types,
   arvo-sparse's matrix values, arvo-spectral's eigenvector values,
   arvo-comb's cost types — are these concrete arvo types or generic?

---

## Revised crate structure

```
arvo              numerics: UFixed, IFixed, Float, Strategy markers,
                  composable numeric traits (TotalOrd, Sqrt, Recip, Abs,
                  FromConstant). No deps within arvo family.

arvo-bits         bit-level contracts: BitWidth, BitAccess, BitSequence.
                  Semantic aliases (Bit, Byte). Blanket impls for arvo
                  numerics (all UFixed/IFixed with F=0 are bit-accessible).
                  Deps: arvo.

arvo-bitmask      Mask64, Mask256, BitMatrix, set ops, bit scanning.
                  Built on arvo-bits contracts.
                  Deps: arvo-bits.

arvo-graph        Topo sort, rank, waist, components, spanning tree.
                  Generic over numeric traits for weights.
                  Deps: arvo, arvo-bitmask.

arvo-sparse       CSR, RCM, block diagonal, D-M.
                  Generic over numeric traits for matrix values.
                  Deps: arvo, arvo-bitmask.

arvo-comb         DP, greedy grouping, bin-packing.
                  Generic over numeric traits for costs.
                  Deps: arvo, arvo-bitmask.

arvo-spectral     Laplacian, Fiedler, power iteration.
                  Generic over numeric traits for weights and eigenvectors.
                  Deps: arvo, arvo-sparse.
```

Dependency flow:
```
arvo                 (no deps)
arvo-bits            (deps: arvo)
arvo-bitmask         (deps: arvo-bits)
arvo-graph           (deps: arvo, arvo-bitmask)
arvo-sparse          (deps: arvo, arvo-bitmask)
arvo-comb            (deps: arvo, arvo-bitmask)
arvo-spectral        (deps: arvo, arvo-sparse)
```

Note: arvo-graph, arvo-sparse, arvo-comb do NOT depend on arvo-bits
directly. They use arvo-bitmask (which depends on arvo-bits) for DAG
adjacency, and arvo for numeric trait bounds on weights/costs.

## arvo-bits: bit-level contracts

### Traits

**`BitWidth`** — how many bits this type has:
```rust
pub trait BitWidth {
    const WIDTH: usize;
}
```

Blanket impl for all UFixed/IFixed where the width is I + F (for
UFixed) or 1 + I + F (for IFixed). Also impl for standard types
(u8, u16, u32, u64) if we ever need them as bit containers.

**`BitAccess`** — individual bit manipulation:
```rust
pub trait BitAccess: BitWidth + Copy {
    fn bit(self, idx: usize) -> bool;
    fn with_bit_set(self, idx: usize) -> Self;
    fn with_bit_cleared(self, idx: usize) -> Self;
    fn with_bit_toggled(self, idx: usize) -> Self;
}
```

All methods take `self` and return `Self` (functional style, no
mutation). `idx` is from LSB (bit 0 = least significant).

**`BitSequence`** — scanning and counting:
```rust
pub trait BitSequence: BitWidth + Copy {
    fn trailing_zeros(self) -> usize;
    fn leading_zeros(self) -> usize;
    fn count_ones(self) -> usize;
    fn count_zeros(self) -> usize;
    fn is_zero(self) -> bool;
}
```

These map directly to hardware intrinsics: `cttz` (trailing zeros),
`ctlz` (leading zeros), `popcnt` (count ones). On aarch64: `rbit`
+ `clz`, `clz`, and hardware popcount. On x86: `tzcnt`, `lzcnt`,
`popcnt`.

### Blanket impls

Any arvo numeric type where the underlying bits are accessible gets
blanket impls:

```rust
// all integer-like fixed-point (F=0) are trivially bit-accessible
impl<const I: u8, S: Strategy> BitAccess for UFixed<I, 0, S> { ... }
impl<const I: u8, S: Strategy> BitSequence for UFixed<I, 0, S> { ... }

// fractional fixed-point (F>0) is also bit-accessible as a raw
// container — you're operating on the bits representation, not the
// numeric value. This is explicit opt-in.
impl<const I: u8, const F: u8, S: Strategy> BitAccess for UFixed<I, F, S> { ... }
impl<const I: u8, const F: u8, S: Strategy> BitSequence for UFixed<I, F, S> { ... }
```

The F>0 impl means you can use any UFixed as a bit container. This
is useful for packed flag fields: `UFixed<0, 8, Hot>` treated as 8
individual flag bits.

### Semantic aliases

```rust
pub type Bit<S: Strategy = Hot> = UFixed<1, 0, S>;
pub type Byte<S: Strategy = Hot> = UFixed<8, 0, S>;
pub type Word<S: Strategy = Hot> = UFixed<16, 0, S>;
pub type DWord<S: Strategy = Hot> = UFixed<32, 0, S>;
pub type QWord<S: Strategy = Hot> = UFixed<64, 0, S>;
```

Default strategy is Hot for bit-level aliases — these are typically
used in bit manipulation contexts where wrapping is expected and
storage density matters.

## Mathematical sub-crates: generic over trait bounds

The mathematical sub-crates (arvo-graph, arvo-sparse, arvo-spectral,
arvo-comb) are generic over their numeric types. They don't import
UFixed or IFixed directly. They require composable trait bounds.

### arvo-graph

Weights for rank computation, path lengths:
```rust
fn upward_rank<W>(dag: &BitMatrix, weights: &[W]) -> [W; N]
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant
```

The algorithm needs addition (accumulate path weights), ordering
(find maximum), copy (store in arrays), and a zero value (initialise).
It does not need multiplication, division, or square root.

If W is `UFixed<16, 0, Hot>`, rank accumulation wraps on overflow.
If W is `UFixed<16, 0, Precise>`, it saturates. The algorithm is
identical. The consumer chooses the safety level by choosing W.

### arvo-sparse

RCM reordering and block diagonal detection operate on the access
matrix (binary: edge or no edge). Weights are only needed for
spectral methods (which live in arvo-spectral). RCM and D-M are
essentially graph algorithms on the bipartite structure — they don't
need numeric weights at all. They need arvo-bitmask for the bit
matrix representation.

For CSR format with weighted edges:
```rust
struct Csr<W: Copy> {
    row_ptr: [u16; N],
    col_idx: [u16; NNZ],
    values: [W; NNZ],
}
```

Generic over the value type. No numeric trait needed on W for the
CSR structure itself — only for algorithms that compute on the values.

### arvo-spectral

Power iteration needs float-like operations:
```rust
fn fiedler_vector<W, F>(weights: &[[W; N]; N]) -> [F; N]
where
    W: Into<F>,  // weight matrix converts to float for eigenvector computation
    F: Add<Output = F> + Mul<Output = F> + Sqrt + Recip + Copy + FromConstant
```

The Laplacian is built from integer weights (arvo fixed-point). The
eigenvector computation uses float (FastFloat or StrictFloat). The
`Into<F>` bound handles the conversion at the API boundary.

### arvo-comb

DP cost functions:
```rust
fn matrix_chain_dp<W>(costs: impl Fn(usize, usize) -> W) -> W
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant
```

Same bounds as arvo-graph rank computation. Addition, ordering, copy,
zero. The consumer's cost function returns an arvo type; the DP
accumulates and compares via trait bounds.

Greedy grouping:
```rust
fn greedy_group<A, T>(
    items: &[T],
    feasible: impl Fn(&A, &T) -> bool,
    merge: impl Fn(A, &T) -> A,
    init: A,
) -> Vec<(usize, usize)>  // ranges
```

Fully generic. No numeric traits needed — the feasibility predicate
and merge function are closures. The consumer provides the domain
logic.

## Impact on hilavitkutin

hilavitkutin instantiates the generic sub-crates with concrete types:

- DAG adjacency: arvo-bitmask `BitMatrix` / `Mask64`
- Weights for rank/path: `UFixed<32, 0, Warm>` (or similar)
- Eigenvector: `FastFloat` (via the `Float` alias)
- DP costs: `UFixed<32, 0, Warm>`

These concrete instantiations happen in hilavitkutin, not in the
arvo sub-crates. The sub-crates are library code; hilavitkutin is
the application.
