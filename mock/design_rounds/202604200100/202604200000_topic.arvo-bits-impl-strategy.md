# Topic: arvo-bits impl strategy — concrete-type impls, no L0 change

**Date:** 2026-04-20
**Status:** Resolved — captured as design intent for the arvo-bits impl round
**Related:** 202603241400_topic.arvo-crate-dependency-revision.md (arvo-bits layer)

## The gap the prior design left

The arvo-bits `DESIGN.md.tmpl` (shipped in the L0-foundation round)
described the BitAccess / BitSequence impls as blanket over the
`BitPresentation` marker:

```rust
impl<T: BitPresentation> BitAccess for T { ... }
impl<T: BitPresentation> BitSequence for T { ... }
```

`BitPresentation` as shipped in L0 exposes only `LOGICAL_WIDTH: USize`
(see `mock/crates/arvo/src/markers.rs`). There is no method on the
trait that hands arvo-bits a handle to the raw container, so there
is no way to implement bit manipulation from inside a `T:
BitPresentation` bound. The blanket-impl language cannot stand.

## Options considered

1. **Extend `BitPresentation`** in arvo to expose the container
   (`type Bits: BitPrim; fn bits(self) -> Self::Bits; fn
   from_bits(Self::Bits) -> Self;`). Clean, but extends L0 — and L0
   is closed. Would require a new arvo round preceding arvo-bits.

2. **Concrete impls on `UFixed<I, F, S>` and `IFixed<I, F, S>` from
   arvo-bits.** arvo-bits is downstream of arvo; it can impl its
   own traits on arvo's types (orphan rule permits: the trait is
   local to arvo-bits, the type is foreign — local-trait-foreign-
   type is allowed). Uses the already-public `to_raw()` /
   `from_raw()` inherent methods on `UFixed` / `IFixed`. No L0
   change needed.

3. **Free functions** keyed off container types. Least idiomatic;
   loses trait-based dispatch.

## Resolution

Option 2. Keep the arvo-bits scope tight and avoid reopening the
just-closed L0 round. Concrete impls over `UFixed<const I: IBits,
const F: FBits, S: Strategy>` and `IFixed<...>` for `BitWidth`,
`BitAccess`, and `BitSequence`. A sealed `BitPrim` trait in
arvo-bits is implemented for the four concrete container primitives
(`u8`, `u16`, `u32`, `u64`) with the bit-manipulation surface
(`count_ones`, `trailing_zeros`, `leading_zeros`, `bit`,
`with_bit_set`, `with_bit_cleared`, `with_bit_toggled`). Each
`impl<const I: IBits, const F: FBits, S: Strategy> BitAccess for
UFixed<I, F, S> where S: UContainerFor<{ufixed_bits(I, F)}>,
<S as UContainerFor<{ufixed_bits(I, F)}>>::T: BitPrim` delegates
bit-level ops via `self.to_raw()` and `Self::from_raw(...)`.

Signed case mirrors unsigned via a separate `IBitPrim` sealed trait
for `i8` / `i16` / `i32` / `i64` — same operations, different
underlying types.

## Why not extend `BitPresentation`

Small-rounds convention (validated in the L0 close): each round stays
tight. Extending L0 now would force a new arvo round before arvo-bits
and churn the closed archive. The concrete-impls-from-arvo-bits path
reaches the same user-facing API without touching L0. If a later
round surfaces a need to blanket across non-UFixed `BitPresentation`
implementers, that's the right time to propose the extension.

## Consequence for arvo-bits aliases

The semantic aliases `Bit`, `Nibble`, `Byte`, `Word`, `DWord`, `QWord`
remain `pub type X<S: Strategy = Hot> = UFixed<{IBits::N},
{FBits::ZERO}, S>`. Named `IBits::N` constants required because
`generic_const_exprs` on current nightly still forbids struct
construction inside anonymous const args; L0 provides `IBits::ZERO` /
`IBits::ONE` already and this round adds `IBits::FOUR` / `EIGHT` /
`SIXTEEN` / `THIRTYTWO` / `SIXTYFOUR` to arvo-bits (local helpers,
NOT another arvo L0 change — these constants live in arvo-bits as
private helpers used only by its own aliases; consumers who want
different widths spell their own). A minimal alternative: just use
the inline `IBits::new_const::<4>()` if we add one. Final mechanism
picked at src-time.
