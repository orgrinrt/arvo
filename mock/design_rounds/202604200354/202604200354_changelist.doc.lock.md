# Doc Changelist: arvo post-review polish

**Date:** 2026-04-20
**Phase:** DOC
**Scope:** cross-cut — `arvo` L0 + `arvo-spectral` L3.

## Purpose

Post-shipping review surfaced two correctness bugs and three
contract-gaps. This round fixes all five and records follow-ups in
BACKLOG. No new types, no new traits. Shape changes are local to the
functions named below.

## Fixes

### CRITICAL

**F1 — `wrapping` div-by-zero in `impl_u_arith_wrapping!` / `impl_i_arith_wrapping!`**
(`arvo/src/strategy.rs:438` and `:493`). The current guard catches
`b == 0` then recurses into `a.wrapping_div(0)`, which Rust std
panics on. The wrapping contract has no identity for zero, but it
must not panic under the `Hot` / `Warm` discipline — silent
numerical garbage is strictly preferred over an unhandled panic in
LLVM-vectorised loops.

Chosen **Option A**: return `a` (the numerator) unchanged on
divide-by-zero. Documented in DESIGN.md.tmpl as the wrapping
div-by-zero semantic. Symmetry with the saturating variant, which
returns `MAX`, is intentional — wrapping's "no identity" justifies
the sentinel of "identity under the only operation we actually
performed" (no-op).

Applies to both `u_div` (unsigned) and `i_div` (signed). No change
to non-zero behaviour.

**F2 — `k_way_partition` stack overflow-drop**
(`arvo-spectral/src/partition.rs:178`). Two independent
`stack_len < k` guards: the first push succeeds when one slot is
left, the second silently drops its half. Algorithmic invariant
analysis shows current control-flow keeps `stack_len ≤
partition_count ≤ K`, so the bug is not reachable in shipping
traces — but the guard is incorrect in intent (both halves belong
on the stack or neither).

Chosen **unified guard under one capacity check** (a variant of
Option A from the task): promote both pushes under
`stack_len + 2 <= cap_size(K)`. Fallback `stack_len + 1 <=
cap_size(K)` pushes whichever half is larger (best-effort
degradation — documented in the fn doc-comment). This preserves
current behaviour on the reachable paths and closes the silent-drop
hole.

### IMPORTANT

**F3 — `sqrt` precision on `StrictFloat`**
(`arvo/src/traits.rs:188` / `:204`). Current impl is 3-iter (f32) /
4-iter (f64) Newton-Raphson with a bit-manipulated seed; gives ~7
ULP, not correctly-rounded. `FastFloat` contract is fine with 7 ULP
(approximation is its contract). `StrictFloat` promises bit-exact
IEEE 754 — 7-ULP violates that.

Chosen **Option B** (BACKLOG-only): prominent doc-comment warning
on both `sqrt_f32` / `sqrt_f64` and on the `Sqrt` impls for both
`FastFloat` and `StrictFloat`. BACKLOG note: integrate libm /
musl-math for correctly-rounded sqrt in `no_std`. Option A (cfg-gate
on `feature = "std"`) is rejected this round — arvo's `#![no_std]`
discipline is absolute, a `feature = "std"` gate opens a precedent
we don't want in L0.

**F4 — `fiedler.rs` const-N bound**
(`arvo-spectral/src/fiedler.rs:80`). `let n_as_u8 = n as u8;` then
`F::from_constant(n_as_u8)`. `cap_size(N) > 255` truncates silently.

Chosen **Option A** with `const { assert!(...) }`: compile-time
failure at `cap_size(N) > 64`. Matches the Mask64 output ceiling
already documented in the module docs (line 79: "we guarantee
N <= 64 via the Mask64 partitioning surface"). This promotes a
comment-level promise to a compile-time check.

**F5 — `Matrix::get` / `Matrix::set` bounds check**
(`arvo-spectral/src/matrix.rs:67`). Current impl indexes directly.
Valid N guarantee (const-generic) does not imply valid (i, j) —
those come from runtime `USize` values.

Chosen **Option A** (`debug_assert!`): zero release cost, catches
bugs in dev. `set` gets the same treatment for consistency.

## BACKLOG additions

`arvo/BACKLOG.md.tmpl` gains a `## Post-review notes` section with:

- Cap(pub USize) has no upper-bound enforcement. Constructing
  `Cap(USize(usize::MAX))` is silently legal. A future round should
  consider non-pub inner fields with a constructor that enforces
  `N <= MAX_CONST_ARRAY` (~2^24 as a pragmatic stack-array ceiling).

- `cap_size` duplicated across `arvo-bitmask/src/matrix.rs` and
  `arvo-spectral/src/matrix.rs`. Resolve to a single definition when
  `Matrix` promotes into `arvo-sparse` (already tracked in
  `arvo-spectral/BACKLOG.md.tmpl` Matrix-in-arvo-sparse item — this
  note is the cross-reference).

- `StrictFloat::sqrt` is not correctly-rounded in no_std. The 7-ULP
  Newton-Raphson approximation is acceptable for current consumer
  surfaces (spectral partitioning Fiedler-vector sign inspection)
  but violates the bit-exact IEEE contract StrictFloat promises.
  Integrate libm or musl-math when a consumer actually depends on
  correctly-rounded sqrt. Ties to F3 above.

## DESIGN.md additions

`arvo/DESIGN.md.tmpl` gains a short note in the Float wrappers
section clarifying that `StrictFloat::sqrt` is currently a 7-ULP
Newton-Raphson approximation in no_std. This is a correctness note,
not a contract change — the design intent remains correctly-rounded,
the no_std substrate ships an approximation until libm integrates.

A second note in the strategy-comparison wrapping arithmetic row
clarifies wrapping div-by-zero returns the numerator unchanged (no
identity; silent garbage is preferable to panic under the Hot / Warm
single-instruction contract).
