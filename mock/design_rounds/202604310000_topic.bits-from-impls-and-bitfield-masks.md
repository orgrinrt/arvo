# Topic: Bits From impls + per-field bitfield masks

**Date:** 2026-04-21
**Status:** TOPIC phase
**Scope:** arvo-bits ergonomics — two focused changes identified during
hilavitkutin-str consumer migration of the newtype-wrapper Bits.

## Motivation

After round 202604300000 landed `Bits<N, S>` as a container-dispatched
newtype, two ergonomic gaps surfaced in consumer code:

1. `Bits::<28>::new(raw_u64)` repeats at every construction site —
   persistence bit-arithmetic, test setup, interner narrowing.
2. Bitfield layouts expose named-range accessors but no mask
   constants. Consumers (persistence) need to perform bit-arithmetic
   on raw values that haven't been wrapped yet, which requires a
   masked-at-layout-width pattern. Today those constants live
   duplicated on `Str` itself (`CONST_ORIGIN_FLAG`, `RUNTIME_MASK`,
   `ID_MASK`), which redundantly restates `StrLayout`'s structure.

## Part 1 — `From` impls on `Bits<N, S>`

### Proposed

```rust
// Exact-container conversion (generic, single impl).
impl<const N: u8, S: Strategy> From<<S as UContainerFor<N>>::T> for Bits<N, S>
where S: UContainerFor<N>,
{
    fn from(raw: <S as UContainerFor<N>>::T) -> Self {
        Self::from_raw(raw)
    }
}

// u64-from path for the Hot strategy (per-N, via macro; mask-then-narrow).
impl From<u64> for Bits<N, Hot> { ... }  // one per N in 1..=64
```

### Why two

- The exact-container From is a generic blanket — it's the arvo-native
  conversion and carries no width loss.
- The u64 From exists because most consumer call sites already think
  in `u64` (FNV hashes, content digests) and want a single call to
  mask-then-narrow to whatever width Bits needs.

### Const-fn constraint

`From::from` is a trait method; calling it inside `const fn` requires
`const_trait_impl` (unstable, not yet available for `From`/`Into`).
That means:

- `.into()` works at non-const call sites — persistence, tests, the
  interner's `intern()` runtime path.
- `const fn` paths (currently `str_const!()` macro body and
  `StrLayout::with_<field>` setters) keep the explicit
  `Bits::<N>::new(...)` form. Acceptable: these are authored once
  inside macro bodies, not sprawled across callers.

### Decision D1

Ship both From impls. Consumer APIs keep their concrete `Bits<W, S>`
parameter types (no `impl Into` on setters) to preserve const-fn
compatibility. The ergonomics win lands on non-const sites via
`.into()`.

## Part 2 — per-field `MASK` constants on `bitfield!`-generated structs

### Proposed

Extend the `bitfield!` macro to generate, per sub-range, a
`<FIELD_UPPER>_MASK: Bits<N, Hot>` associated constant on the
wrapping struct. The mask is pre-shifted to the sub-range's
position within the parent layout:

```rust
bitfield! {
    pub struct StrLayout: 32 {
        origin: 1 at 31,
        reserved: 3 at 28,
        id: 28 at 0,
    }
}
// Generated:
impl StrLayout {
    pub const ORIGIN_MASK: Bits<32, Hot> = Bits::<32>::new(1u64 << 31);
    pub const RESERVED_MASK: Bits<32, Hot> = Bits::<32>::new(0b111u64 << 28);
    pub const ID_MASK: Bits<32, Hot> = Bits::<32>::new(0x0FFF_FFFFu64);
}
```

### Decision D2

Generate only `MASK`. `SHIFT` and `WIDTH` constants are not currently
used by any consumer; add them lazily when a concrete use case
surfaces. Rust's dead-code elimination for unused `pub const` is
not guaranteed (linker visibility rules), so leaving them out is
cheaper than hoping for it.

## Part 3 — hilavitkutin cascade (separate commit after arvo merges)

Str drops its duplicated mask constants:

```rust
// Before:
impl Str {
    pub const CONST_ORIGIN_FLAG: Bits<32, Hot> = Bits::<32>::new(0);
    pub const RUNTIME_MASK: Bits<32, Hot> = Bits::<32>::new(1u64 << 31);
    pub const ID_MASK: Bits<32, Hot> = Bits::<32>::new(0x0FFF_FFFF);
    ...
}

// After:
impl Str {
    // All layout-level constants come from StrLayout.
    // CONST_ORIGIN_FLAG had no callers (is_const() covers the use);
    // remove entirely.
    // RUNTIME_MASK → StrLayout::ORIGIN_MASK (same value, better-named).
    // ID_MASK → StrLayout::ID_MASK (exact duplicate).
}
```

Persistence switches `Bits::<28>::new(...)` call sites to `.into()`
via the new From impl. `StrLayout::ID_MASK` replaces `Str::ID_MASK`
at read sites.

## Out of scope

- `Bits::SHIFT` / `Bits::WIDTH` per-field constants (D2 deferral).
- `impl Into<Bits<N, S>>` on setter signatures (const-fn constraint).
- `From<u32>` / `From<u8>` / `From<u16>` — u64 is the widest; callers
  that have a narrower type can upcast to u64 at negligible cost, or
  use the exact-container From path.
- `ConstFrom` sealed trait — over-engineering for the ergonomic gain.
