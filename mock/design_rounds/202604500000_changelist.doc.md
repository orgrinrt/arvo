### Add new section: "Numeric aliases (Uint*, Int*)"

Insert after the existing UFixed / IFixed framing (before any algorithm-domain sections). Section content:

- One-paragraph framing: arvo ships exact-width unsigned and signed aliases over `UFixed<I, F, S>` / `IFixed<I, F, S>` so consumer code reaches for an arvo type at every numeric position, never a bare primitive. ### Add new section: "Const-fn ergonomic helpers"

Insert after the new aliases section. Section content:

- One paragraph: `ibits(n)`, `fbits(n)`, `width_of(n)` are const-fn constructors that take a bare-primitive literal and return the matching ConstParamTy newtype. They are the single allowed call-site exception to the bare-primitive rule, by workspace policy `.claude/rules/no-bare-primitives.md`.
- Helper signatures:
  ```rust
  pub const fn ibits(n: u8) -> IBits;
  pub const fn fbits(n: u8) -> FBits;
  pub const fn width_of(n: u8) -> BitWidth;
  ```
- One paragraph on usage: consumer turbofish like `Bits<{ width_of(12) }, Hot>` reads naturally; without these, every literal at every call site would require `Uint7::from_u8(...)` boilerplate, which defeats the substrate's ergonomic case.

### Update existing "ConstParamTy newtypes" framing (`IBits`, `FBits`)

Existing prose describing `IBits(pub u8)` / `FBits(pub u8)` updates to:

- `pub struct IBits(pub Uint7)`. Wraps `Uint7` (covers 0-127, one bit of headroom over the realistic 0-64 range). Distinct newtype from `FBits` so I and F can't be swapped on a `UFixed` turbofish without a compile error.
- `pub struct FBits(pub Uint7)`. Same shape as `IBits`, semantically distinct.
- `pub struct BitWidth(pub Uint7)`. New. Const-generic parameter type for arvo bit-width positions (`Bits<const N: BitWidth>`, `Hasher<const N: BitWidth>`, `UContainerFor<const N: BitWidth>`).

All three remain `ConstParamTy + PartialEq + Eq + Copy + Clone + Debug`. The `pub Uint7` field is the inner arvo type, not a bare primitive. `Uint7` itself is a UFixed alias whose container lifts via `#[repr(transparent)]`.

### Update existing "USize / Cap / Bool" framing

- `USize`: was `pub struct USize(pub usize)`, becomes a cfg-gated alias: `pub type USize = Uint64` on 64-bit targets, `pub type USize = Uint32` on 32-bit. The platform-pointer-width difference handled at the alias level; consumer code reads `USize` everywhere.
- `Cap`: continues to wrap `USize` (the alias).
- `Bool`: unchanged. `pub struct Bool(pub bool)` stays; bool has no fixed-point analogue, this is the irreducible exception.

### Add new section: "ConstParamTy on UFixed / IFixed"

Insert after the ConstParamTy newtypes framing. Section content:

- One paragraph: `UFixed<I, F, S>` and `IFixed<I, F, S>` impl `ConstParamTy`, enabling them to appear as const-generic parameter types (`const X: UFixed<...>`). Soundness: the transparent repr makes the structural-equality requirement satisfiable through the underlying container's structural eq.
- One paragraph on the implementation note: derive where the cascade is clean, manual `unsafe impl` where it's not. Manual impls carry a doc comment explaining the soundness chain: transparent repr, structurally-eq inner container, all type parameters are themselves ConstParamTy.

### Update top-of-file framing paragraph

Add one sentence: "arvo ships UFixed-based aliases (`Uint5`..`Uint64`, `Int8`..`Int64`, `USize`) so every numeric position in consumer code uses an arvo type, never a bare primitive. Bit-count meta-types (`IBits`, `FBits`, `BitWidth`) wrap `Uint7`. Const-fn helpers (`ibits`, `fbits`, `width_of`) provide call-site ergonomics."

### Add new section: "Concrete algorithm: Fnv1a<N>"

Insert after the `Hasher` trait section. Section content:

- One-paragraph framing: `Fnv1a<const N: BitWidth>` is the first concrete `Hasher` impl shipped by arvo-hash. Implements canonical FNV-1a-64 (offset basis `0xcbf29ce484222325`, prime `0x100000001b3`). For `N < 64`, low-N bits of the 64-bit accumulator are returned. Matches `hilavitkutin-str::const_fnv1a` semantics for the 28-bit content-hash case.
- The runtime trait impl (`Hasher<N>::update` / `Hasher<N>::finalize`) and the const-fn entry point (`Fnv1a::<N>::hash_const(bytes: &[u8]) -> Bits<N>`) coexist; both compute the same algorithm. Const-fn variant is what compile-time validation downstream consumes (the hilavitkutin app-builder formalisation round).
- Table summarising the surface:

  | Item | Form |
  |---|---|
  | Type | `pub struct Fnv1a<const N: BitWidth>` where `Hot: UContainerFor<N>` |
  | Runtime trait impl | `impl<const N: BitWidth> Hasher<N> for Fnv1a<N>` |
  | Const-fn entry | `pub const fn hash_const(bytes: &[u8]) -> Bits<N>` |
  | Match for `Str` | `Fnv1a::<{ width_of(28) }>::hash_const(s.as_bytes())` |

- One paragraph on the `bytes: &[u8]` parameter: the canonical byte-slice shape produced by `core::any::type_name::<T>().as_bytes()` and similar const-fn sources. Wrapping in a higher-level boundary is for a follow-up round; for now the slice form matches what const-fn callers can produce.
