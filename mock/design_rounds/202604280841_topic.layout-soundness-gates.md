**Round:** 202604280841. Layout-soundness gates for repr(transparent) chains and the MetaCarrier transmute
**Predecessor:** Round 202604280806 (Round C; closed at `92a7cc4`)

This round adds compile-time layout-equivalence assertions across the substrate's `repr(transparent)` chains and the MetaCarrier transmute that landed in Round C. The substrate now has several invariants the type system can't verify on its own:

1. `MetaCarrier` (concrete `repr(transparent)` over u16) and `Bits<9, Hot, Unsigned>` (generic, projection-resolved to u16) must remain layout-byte-identical for the `MetaCarrier::as_bits()` transmute to stay sound. If a future round changes Hot's container projection at N=9, the transmute silently breaks.
2. Each `unsafe impl Transparent for $W { type Inner = ... }` asserts a layout equivalence the compiler can't cross-check unless the field's type is named in the trait. With multi-level transparency (`IBits` → `MetaCarrier` → `u16`), divergence at any layer can drift.
3. `Bits<N, S, Sign>` is `repr(transparent)` over the projection target. If the projection table for a known cell ever changes shape (e.g., u8 → MultiContainer<u8, u8>), every consumer of `Bits<N, S, Sign>` at that cell must rebuild.
4. `UFixed<I, F, S>` and `IFixed<I, F, S>` are now both `repr(transparent)` over `Bits<{I+F or 1+I+F}, S, Sign>`. The transparency is contract-stable but unverified at the type level past one repr hop.
5. `MultiContainer<HiT, LoT>` is `repr(C)` with two fields. Layout is `(size_of::<HiT>() padded to align_of::<LoT>()) + size_of::<LoT>()` per Rust's `repr(C)` rules, but consumer code (and the Bits projection) implicitly assumes specific HiT/LoT pairs at specific N bands.

All five are runtime-uncatchable: the substrate compiles fine when they hold; if they break, the failure mode is silent UB at the next transmute or memcpy.

The fix: compile-time `const _: () = assert!(...)` guards. Each adds a panic-on-mismatch at the type level. Cheap insurance.

## Decision: scope of guards

Three tiers, all shipping in this round:

**Tier 1 (CORE — load-bearing for the MetaCarrier workaround):**
- `MetaCarrier` size + alignment == `Bits<9, Hot, Unsigned>` size + alignment.
- `MetaCarrier` size + alignment == u16 size + alignment.
- `IBits` / `FBits` / `Width` size + alignment == `MetaCarrier` size + alignment (each).

**Tier 2 (BITS PROJECTION SANITY — load-bearing for any `Bits<N, S, Sign>` usage):**
- For Hot at boundary cells (1, 8, 9, 16, 17, 32, 33, 64, 65, 128, 129, 192, 193, 255): `size_of::<Bits<N, Hot, Unsigned>>() == size_of::<<Hot as UContainerFor<N>>::T>()`.
- Same for Cold (mirrors Hot).
- Same for Warm at its supported cells (1, 8, 9, 16, 17, 32, 33, 64).
- Same for Precise at its supported cells (1, 8, 9, 16, 17, 32, 33, 64).
- Same for the signed counterparts on each strategy where IContainerFor is implemented.

**Tier 3 (CONSUMER WRAPPER SANITY):**
- `size_of::<UFixed<{ibits(8)}, {fbits(0)}, Hot>>() == size_of::<u8>()`.
- `size_of::<UFixed<{ibits(13)}, {fbits(3)}, Hot>>() == size_of::<u16>()` (16 logical bits → u16).
- `size_of::<IFixed<{ibits(7)}, {fbits(0)}, Hot>>() == size_of::<i8>()` (1+7 = 8 → i8).
- `size_of::<IFixed<{ibits(15)}, {fbits(0)}, Warm>>() == size_of::<i64>()` (16 logical bits → Warm 2x → i32; check actual at cell).

**Tier 4 (MULTICONTAINER LAYOUT — load-bearing for 129..=255 storage):**
- `size_of::<MultiContainer<u64, u128>>() >= size_of::<u64>() + size_of::<u128>()` (lower bound; padding may make it larger).
- `align_of::<MultiContainer<HiT, LoT>>() == max(align_of::<HiT>(), align_of::<LoT>())`.
- For each cell at 129..=192 and 193..=255, the projection target's MultiContainer pair size matches the impl table.

## Decision: where the guards live

Two new files:

- `arvo-storage/src/layout_assertions.rs`: Tier 1 + Tier 2 + Tier 4 (layouts that involve types in arvo-storage / arvo-strategy).
- `arvo/src/layout_assertions.rs`: Tier 3 (UFixed / IFixed wrapper sanity).

Both modules are private (`mod layout_assertions;` with no `pub use`). They have no public surface; they exist to break the build at compile time if invariants drift.

## Decision: assertion macro shape

```rust
macro_rules! assert_layout_eq {
    ($a:ty, $b:ty $(,)?) => {
        const _: () = {
            assert!(
                core::mem::size_of::<$a>() == core::mem::size_of::<$b>(),
                concat!(
                    "size_of::<", stringify!($a), ">() != size_of::<", stringify!($b), ">()"
                )
            );
            assert!(
                core::mem::align_of::<$a>() == core::mem::align_of::<$b>(),
                concat!(
                    "align_of::<", stringify!($a), ">() != align_of::<", stringify!($b), ">()"
                )
            );
        };
    };
}
```

Macro lives at module level in each layout_assertions.rs. Failure produces a clear compile-time panic mentioning both types.

## Out of scope (deferred / BACKLOG)

- **Strategy-axis HasAxes correctness check**: cross-checking that the four canonical markers' axis combinations match the doc table at compile time. Possible via const evaluation of axis-marker `RANK`-style discriminants; warrants its own design.
- **Auto-generated assertions for every Bits cell**: would require macro expansion across all (Strategy, N) cells; the boundary-cell coverage in Tier 2 is sufficient for catching projection drift without explosion.
- **ConstParamTy soundness audit**: separate concern; the trait derives + manual impls already encode the relevant invariants.
- **Mockspace lint to auto-generate layout assertions**: a future feature; manual guards are correct insurance for now.
