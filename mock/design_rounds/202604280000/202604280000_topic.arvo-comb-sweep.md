## Purpose

Lint-scope widening surfaced 46 drift sites in arvo-comb, all of one of
three shapes:

- **Loop-index `usize`**: `let mut i: usize = 0; while i < cap_size(N) { ... i += 1; }`. Dominant pattern, repeats ~30 times.
- **Internal 2D `bool` buffers**: `let mut reachable: [[bool; cap_size(N)]; cap_size(N)] = [[false; ...]; ...];` in dp.rs; plus `let mut placed = false;` / `let mut open = false;` local flags.
- **Public callback-return `bool`**: `feasible: impl Fn(USize, USize) -> bool` on `matrix_chain_dp` and `feasible: impl Fn(&A, &T) -> bool` on `greedy_group`. API-visible; callers currently pass closures returning raw `bool`.

Remediation is mechanical; the design question is just which treatment each
class gets.

### Why the loop-index usize exists

`cap_size(c: Cap) -> usize` returns raw `usize` by language grammar (#121 ŌĆö
Rust arrays require `[T; N: usize]` at the language level). Loop indices that
compare against `cap_size(N)` are naturally `usize` to avoid `.0` wrapping on
every comparison and `USize(idx + 1)` wrapping on every increment.

If every loop index becomes `USize`, each iteration looks like:
```rust
let mut i = USize(0);
while i.0 < cap_size(N) {
    // body, with i.0 used as array index
    i = USize(i.0 + 1);
}
```
vs. the current:
```rust
let mut i: usize = 0;
while i < cap_size(N) {
    // body, with i used as array index
    i += 1;
}
```

The wrapping adds visual noise. Once the caller provides `USize`-typed
inputs, the internal index is a bookkeeping detail.

### What arvo-bitmask did (reference ŌĆö sibling round just closed)

arvo-bitmask had 3 lint:allow sites: one on `cap_size` (language grammar,
tracked #121) and two on `core::Iterator::next` + one on
`core::PartialOrd::partial_cmp` (core trait signatures, tracked #115). No
loop-index-over-cap_size pattern occurred there because bitmask set-ops
delegate to trait-native `BitSequence` / `BitLogic`. arvo-comb is algorithm
code without a bitmask-trait-shaped abstraction; the loop-index pattern is
unavoidable.

## Open questions

### Q1 ŌĆö Treatment of loop-index `usize` inside `while i < cap_size(N)` patterns

**Option A ŌĆö Wrap as `USize`, accept `.0` boilerplate.** Every loop index
becomes `USize`; compares use `i.0 < cap_size(N)`; increments use
`i = USize(i.0 + 1)`. No `lint:allow` needed. Most verbose; strictest.

**Option B ŌĆö Keep raw `usize` + `lint:allow(no-bare-numeric) lint:allow(arvo-types-only) reason: loop index bounded by cap_size(N) return (tracked #121); tracked: #121`.**
Treats loop-index-over-raw-cap_size as a legitimate child of #121's language-grammar
exemption. Pattern repeats ~30 times in arvo-comb + similar in graph/sparse/
spectral ŌĆö each site is a trailing inline comment. Preserves algorithm
readability; relies on #121 to eventually resolve.

**Option C ŌĆö Provide a helper `for_cap<const N: Cap>(|i: USize| { ... })`** in
arvo-bitmask (next to `cap_size`) that abstracts the loop. Call sites become:
```rust
for_cap::<N>(|i| {
    // body uses i: USize
});
```
Wraps the raw-usize / USize boundary into one-place abstraction. Downsides:
closure overhead (should monomorphise away); still allocates Coroutine on
paper; adds one more crate-level helper to maintain.

**Option D ŌĆö Wait for arvo L0 to ship `Range<USize>` + `IntoIterator` on
`Cap`.** Not landed, not planned this session. Scope creep.

**Provisional answer:** B. The `cap_size` exemption already frames this as a
language-limitation class, not a design issue. Wrapping loop indices as
`USize` adds visible .0 friction without semantic benefit ŌĆö the raw usize
IS the array index; it's internal bookkeeping. One inline `lint:allow` per
loop-variable declaration (~10 sites after consolidation) vs. per-line
wrapping. Pattern cascades cleanly to graph/sparse/spectral. When #121
resolves (rustc const-eval over newtypes stabilises), the allows + cap_size
disappear together. **Requires user confirmation.**

### Q2 ŌĆö Internal 2D `bool` buffers (dp.rs `reachable` table + local `open` / `placed` flags)

**Option A ŌĆö Wrap as `Bool`.** `reachable: [[Bool; N]; N]` with `[[Bool::FALSE; N]; N]` init. Reads require `.0` or `into()` for `if`. Writes work directly (`reachable[i][i] = Bool::TRUE`).

**Option B ŌĆö `lint:allow(no-bare-numeric)` on each declaration.** Same precedent class as Q1 ŌĆö internal scratch state.

**Option C ŌĆö Mix by cost**: top-level predicate returns and function-local `bool` flags get `Bool`; the multi-dimensional `reachable[][]` table stays raw with `lint:allow` because every read needs `.0` unwrap and the table is scanned in hot loops.

**Provisional answer:** A. `Bool` is cheap (single-byte), conversions are zero-cost after compilation, and `Bool` implements `Try` so `if reachable[lo][hi]? { ... }` is ergonomic. Raw bool here has no justification beyond the cap_size class. **Requires user confirmation, small.**

### Q3 ŌĆö Public callback signatures (`feasible: impl Fn(...) -> bool`)

**Option A ŌĆö Change to `impl Fn(...) -> Bool`.** Callers update their
closures to return `Bool`. API break. But this is a sweep; API clean-up is
the point.

**Option B ŌĆö Keep `bool` return + `lint:allow(no-bare-numeric)` on the
predicate sig.** Treats caller ergonomics as the load-bearing concern.

**Provisional answer:** A. arvo is a library; every consumer is downstream
(hilavitkutin + clause), and both are themselves mid-sweep. Landing the
`Bool` surface here means downstream closures update once during their own
sweep rounds rather than paying a `lint:allow` forever. API-visible bare
`bool` contradicts the "bare primitives don't exist in this stack" directive
ŌĆö the lint is right. **Requires user confirmation.**

## Decisions (record as user answers)

**Q1 (user, 2026-04-20):** None of the three proposed options. User pointed
out the established convention already shipping in arvo-spectral:
`Matrix<W, const N: Cap>` ŌĆö a `#[repr(transparent)]` wrapper over
`[[W; cap_size(N)]; cap_size(N)]` that hides the raw-`usize` array indexing
behind `get(i: USize, j: USize) -> W` / `set(i: USize, j: USize, v: W)` /
`from_fn(|row: USize, col: USize| -> W)`. The `cap_size` and `.0` unwraps
live in ONE place (the wrapper's method bodies), not at every call site.

The correct remediation for arvo-comb is to introduce the 1D sibling of
`Matrix`, then migrate. This unblocks the same pattern in arvo-graph /
arvo-sparse / arvo-spectral ŌĆö every L2/L3 algorithm crate that currently
takes raw `&[T; cap_size(N)]` converts to the wrapper. `cap_size` + #121
exemption become a single-site concern inside the wrapper, not a
per-algo-loop concern.

Scope expands: this round introduces the wrapper(s), promotes shared
scaffolding if applicable, migrates arvo-comb. Downstream rounds
(graph / sparse / spectral) inherit the wrapper cleanly.

**Q1a (user, 2026-04-20):** New crate named `arvo-tensor`. Hosts `Array<T, N>`
(rank-1), promoted `Matrix<W, N>` (rank-2), and the shared `cap_size` helper.
Future rank-N shapes (Grid, Cube, etc.) live here too. Rationale: tensor is
the math term for a fixed-shape typed multi-dimensional quantity; unifies
Array + Matrix + future shapes under one name; no naming collision with
existing arvo crates.

Scope and migration:
- New crate `arvo-tensor` at `mock/crates/arvo-tensor/` (L2 peer of arvo-bitmask).
- `Array<T, const N: Cap>` ŌĆö new rank-1 wrapper.
- `Matrix<W, const N: Cap>` ŌĆö promoted from `arvo-spectral/src/matrix.rs`.
  arvo-spectral re-exports from arvo-tensor or depends-and-uses.
- `cap_size(c: Cap) -> usize` ŌĆö canonical home. arvo-bitmask + arvo-spectral
  drop their local duplicates in favour of `arvo_tensor::cap_size`.
- `arvo-comb` depends on `arvo-tensor`; takes `&Array<T, N>` / `Matrix<W, N>`
  where it currently takes `&[T; cap_size(N)]` / `[[W; N]; N]`.
- Forward dependency: arvo-graph, arvo-sparse, arvo-spectral migrate to
  arvo-tensor as part of their own sweep rounds.

**Q1b (user, 2026-04-20):** Implement `core::iter::IntoIterator` on
arvo-tensor types (standard Rust iteration yielding `&T`), then ship an
arvo-native `Enumerator` trait with a blanket `impl<T: IntoIterator>`
providing `.enumerated()` ŌåÆ `Iterator<Item = (USize, Self::Item)>`. Trait-
first + composable: any `IntoIterator` type (including future wrappers
outside arvo-tensor) picks up `enumerated()` automatically.

Shape:
```rust
// arvo-tensor/src/enumerator.rs
pub trait Enumerator {
    type Item;
    fn enumerated(self) -> impl Iterator<Item = (USize, Self::Item)>;
}

impl<T: IntoIterator> Enumerator for T {
    type Item = T::Item;
    fn enumerated(self) -> impl Iterator<Item = (USize, Self::Item)> {
        self.into_iter()
            .enumerate()
            .map(|(i, item)| (USize(i), item))
    }
}
```

Consumers that want USize-typed indices use `.enumerated()`; consumers that
want raw iteration use standard `for t in &arr { ... }`. Separate concerns,
composable.

`Array<T, N>` / `Matrix<W, N>` themselves impl `IntoIterator for &Array<T, N>`
with `Item = &T` via the inner `[T; cap_size(N)]`'s iterator.

**Q2 (user, 2026-04-20):** Wrap internal `[[bool; N]; N]` buffers in dp.rs as
`Matrix<Bool, N>`. Zero runtime cost; reads through `.0` for `if` or `?` via
`Bool: Try`. Consistent with "bare primitives don't exist anywhere"
directive. Local `let mut open = false;` / `let mut placed = false;` flags
become `let mut open = Bool::FALSE;` etc.

**Q3 (user, 2026-04-20):** Change `feasible: impl Fn(...) -> bool` on both
`matrix_chain_dp` and `greedy_group` to `impl Fn(...) -> Bool`. API break;
downstream consumers update their closures during their own sweep rounds.

**Q4 (user, 2026-04-20):** Defer `arvo::Predicate<X>` trait alias to a
follow-up round (tracked #122). This round's callback sigs stay as
`impl Fn(...) -> Bool`; the Predicate trait refactor across arvo-comb /
graph / sparse / spectral / downstream consumers lands as its own small
round after arvo-tensor ships. Reason: this round already expanded
significantly (new arvo-tensor crate + Enumerator trait); scope control
favours a focused follow-up over bundling.
