## Purpose

The build configuration topic (T4 §Q5a) defined arvo's strategy markers
as a design direction: `Hot` (lane-capped), `Precise` (full-width),
`Warm` (runtime). The consolidation topic carried this forward without
resolving the mechanism. This topic resolves what each strategy marker
does concretely.

## Prior art

loimu-primitives has working `UFixed<I, F>` and `IFixed<I, F>` with
saturating arithmetic and "shaped aliases" (`wide_f32`, `prec_f32`,
`std_f32`) that vary the I/F split. These shaped aliases are a
different concept from strategy markers — they control the precision/
range tradeoff within the type's logical bit layout, not the storage
and operation semantics.

---

## The type

```rust
pub struct UFixed<const I: u8, const F: u8, S: Strategy = Warm> {
    bits: S::Container<I, F>,
}

pub struct IFixed<const I: u8, const F: u8, S: Strategy = Warm> {
    bits: S::SignedContainer<I, F>,
}
```

`I` = integer bits, `F` = fractional bits. `S` = strategy marker
(default: `Warm`). The strategy determines the backing container
type, not just operation semantics.

`I + F` is the logical bit width. The actual storage width depends
on the strategy.

---

## Four strategies

### Hot

Optimised for L1 density and operation throughput. Store small,
operate fast.

**Container:** minimum byte-aligned standard width that fits `I + F`
bits. A 5-bit value stores in u8. A 12-bit value stores in u16. No
bitpacking — byte-aligned always. The 3 wasted bits in a u8 for a
5-bit value are acceptable because the load is a single byte-load
with no shift/mask overhead.

**Same-width arithmetic:** wrapping. Single instruction per op. May
overflow silently. LLVM vectorises freely. Consumer is responsible
for ensuring values stay in range (or accepting wrap-around).

**Cross-width arithmetic:** shift-narrow the wider value to match
the narrower type's layout. Right-shift preserves integer bits,
loses fractional precision. Operate at the narrower width. Wrapping
if the shifted result doesn't fit.

**SIMD:** most lanes per register. u8 in 128-bit NEON = 16 lanes.
u16 = 8 lanes. Highest throughput per SIMD instruction.

**Register pressure:** lowest per value. More values fit in the
register file. Best for the domain 14 holistic feasibility check.

**Use case:** inner morsel loops where range is known safe. Counters,
indices, flags, enum-like values, temperatures.

### Warm (default)

Store big, operate fast. The development-friendly default.

**Container:** 2x the logical bit width, rounded to next standard
width. A 16-bit logical value stores in u32. A 32-bit value stores
in u64. This means wrapping arithmetic cannot overflow on single
operations — the container has room.

**Same-width arithmetic:** wrapping. Single instruction. But because
the container is 2x wide, a single add/sub/mul of two values within
their logical range cannot overflow the container. Chains of
operations (accumulating many values) can eventually overflow.

**Cross-width arithmetic:** widen both operands to 2x the wider
operand's container width. Operate at that width. Wrapping. Single
ops safe, chains may overflow.

**Bitpacking:** no. Byte-aligned, rounded up. Same reasoning as Hot
— avoid shift/mask in the op path.

**SIMD:** fewer lanes than Hot (wider containers), but still
wrapping ops so LLVM vectorises freely.

**Use case:** the default for most columns. Correctness on single
ops without saturating overhead. Morsel loop friendly.

### Cold

Store small, operate carefully. Optimised for storage density.

**Container:** minimum container that fits `I + F` bits. Same as
Hot for byte-multiples. But for sub-byte values: bitpacked. A 5-bit
value stores as 5 bits, not u8. A 3-bit value stores as 3 bits.
Bitpacking requires BitPacked-capable storage (a trait contract the
column storage must satisfy).

**Same-width arithmetic:** widen to 2x before operating, narrow
back after. The widening ensures no precision loss on the op. The
narrowing pays the shift cost. Every operation has load-widen-op-
narrow-store overhead.

**Cross-width arithmetic:** widen both to the wider operand's
container width. Operate at that width. Narrow back to each
operand's original width on store.

**Bitpacking:** yes. Stores exactly the logical bits. Pays
shift/mask cost on every access. Acceptable because Cold values
are accessed infrequently — they ride L2/L3, not L1.

**SIMD:** narrow storage but ops widen, so SIMD applies to the
widened values, not the stored width. Effective lane count matches
the widened width.

**Use case:** columns that are stored in bulk but processed rarely.
Archival data, audit trails, metadata columns, lookup tables.
Also appropriate for resource Seq/Map collections (R5) where
storage density matters more than access speed.

### Precise

Store exactly, operate exactly. Correctness above all.

**Container:** whatever width is needed so that no operation can
lose precision. For same-width multiply, this means 2x the logical
width (same as Warm). For cross-width ops, this means 2x the wider
operand.

**Same-width arithmetic:** saturating. Overflow clamps to max/min
rather than wrapping. The saturating add/sub have SIMD intrinsics
on most platforms (less vectorisation-hostile than checked). Multiply
uses the widened container (2x) so the intermediate result is exact,
then saturate on store if the result exceeds the logical range.

**Cross-width arithmetic:** widen both to 2x the wider operand's
container. Operate. Saturate on store. No precision loss at any
point.

**Bitpacking:** yes. Stores exactly the logical bits. A 5-bit
Precise value is 5 bits. The representation IS the precision — no
rounding, no padding, no wasted bits.

**SIMD:** fewest lanes (widest containers + saturating ops). But
saturating add/sub are SIMD-supported. The main cost is the
saturation clamp per op.

**Use case:** financial calculations, audit-critical values,
values that flow into user-visible reports where silent data loss
is unacceptable. Also the safety-net: if you're unsure about
overflow, use Precise.

## Strategy comparison table

| Aspect | Hot | Warm (default) | Cold | Precise |
|---|---|---|---|---|
| Container | min aligned | 2x logical | min (bitpacked) | 2x logical |
| Bitpacking | no | no | yes | yes |
| Add/Sub | wrapping | wrapping (safe 1-op) | widen-op-narrow | saturating |
| Multiply | wrapping | wrapping (safe 1-op) | widen-op-narrow | widening + saturate |
| Chain safety | overflows | overflows on long chains | overflows on long chains | clamps always |
| Cross-width | shift-narrow wider | widen to 2x wider | widen to wider | widen to 2x wider |
| SIMD lanes | most | moderate | most (stored) / moderate (ops) | fewest |
| L1 density | best | 2x worse | best | 2x worse |
| Access cost | byte load | byte load | shift/mask | shift/mask |
| Register pressure | lowest | moderate | lowest (stored) | moderate |
| Precision loss | yes (wrapping + narrow) | no (single ops) | no (single ops) | never |

## Column layout derivation

The column layout mode (ByteAligned / BitPacked from T5 §Q10) is
derived from the strategy, not set independently:

| Strategy | Layout mode |
|---|---|
| Hot | ByteAligned (minimum aligned container) |
| Warm | ByteAligned (2x container, no packing) |
| Cold | BitPacked (exact logical bits) |
| Precise | BitPacked (exact logical bits) |

This supersedes the scheduling-hint-driven layout from T5 §Q10.
The strategy marker IS the layout decision.

## BIT_WIDTH interaction (R4)

`ColumnValue::BIT_WIDTH` reports the logical bit width (`I + F`).
The actual stored width depends on the strategy:

```rust
impl<const I: u8, const F: u8, S: Strategy> ColumnValue for UFixed<I, F, S> {
    const BIT_WIDTH: usize = (I + F) as usize;
}
```

BIT_WIDTH is always the logical width. The storage engine uses both
BIT_WIDTH and the strategy to determine physical layout:
- BitPacked strategies (Cold, Precise): physical width = BIT_WIDTH
- ByteAligned strategies (Hot, Warm): physical width = container size in bits

## Conversion between strategies

Converting between strategies for the same `<I, F>` is a storage
operation, not a computation:

- **Hot → Warm**: zero-extend into wider container. Free (just a wider store).
- **Warm → Hot**: truncate to minimum container. Free if value fits.
  Wraps if it exceeds the container.
- **Hot/Warm → Cold/Precise**: bitpack from byte-aligned to exact bits.
  Shift/mask cost.
- **Cold/Precise → Hot/Warm**: unpack from bitpacked to byte-aligned.
  Shift/mask cost.
- **Hot → Precise**: zero-extend + bitpack. Cheap.
- **Precise → Hot**: unpack + truncate. May lose precision (shift-narrow).

Converting between different `<I, F>` configurations is a computation:
right-shift or left-shift the fractional bits, clamp or wrap the
integer bits. The strategy determines the clamping/wrapping semantics.

## Cross-width operation detail

When `UFixed<I1, F1, S1>` operates with `UFixed<I2, F2, S2>`:

1. **Strategy resolution:** if S1 != S2, the result strategy is the
   more conservative: Precise > Cold > Warm > Hot. The less
   conservative operand is converted to the more conservative strategy
   before the operation.

2. **Width alignment:** fractional bits are aligned by shifting the
   value with fewer F bits left. Integer bits are aligned by widening.
   The intermediate width depends on the resolved strategy (see table).

3. **Result type:** the result has `I = max(I1, I2)` integer bits and
   `F = max(F1, F2)` fractional bits. The result strategy is the
   resolved strategy from step 1.

## Float types (separate from strategy markers)

`FastFloat` and `StrictFloat` wrap actual `f32`/`f64`. They are NOT
fixed-point. They do not have strategy markers. They are a completely
separate type family.

`Float` is a type alias that resolves to `FastFloat` when
`cfg(arvo_fast_math)` is active (the FastMath pragma from
hilavitkutin-build), and `StrictFloat` otherwise.

The LLVM fast-math flags apply to float operations only. Fixed-point
uses integer arithmetic — no float passes, no fast-math semantics.

## Semantic aliases

Semantic aliases are fixed to specific `<I, F>` configurations AND
a default strategy:

```rust
pub type Ratio = UFixed<0, 8, Warm>;
pub type Factor = UFixed<2, 8, Warm>;
pub type Norm = IFixed<1, 14, Warm>;
pub type Coord = IFixed<16, 16, Warm>;
pub type Angle = UFixed<9, 7, Warm>;
pub type Count = UFixed<32, 0, Warm>;
pub type Average = UFixed<16, 16, Warm>;
pub type Deviation = UFixed<8, 24, Warm>;
pub type Bool = UFixed<1, 0, Hot>;  // always Hot: 1-bit, no precision to lose
```

Consumers can override the strategy: `UFixed<0, 8, Hot>` is a
hot Ratio. The semantic aliases are conveniences, not constraints.

## Open questions

1. **Warm default on release vs debug:** should Warm behave differently
   in debug mode (e.g. panic on chain overflow in debug, wrap in release)?
   Or is the whole point that Warm is the same everywhere?

2. **Strategy on IFixed:** does the signed container follow the same
   widening rules? i16 → i32 for Warm, etc.? Signed saturation has
   different semantics (clamp to i_MIN..i_MAX).

3. **const generic bounds:** `I + F <= 64` for Hot (u64 max container),
   `I + F <= 64` for Warm (u128 container for 64-bit logical width —
   but we said no u128?). Need to decide the maximum logical width and
   whether u128 is acceptable as a Warm/Precise container.

4. **Mixed-strategy columns in one fiber:** if two columns in the same
   fiber have different strategies, how does the holistic feasibility
   check (domain 14) account for the different physical widths?
