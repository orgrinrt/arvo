## Resolutions since the deprecated CL

### D1. No semantic aliases in arvo L0

Previous CL shipped `Ratio`, `Factor`, `Norm`, `Coord`, `Angle`, `Count`, `Average`, `Deviation`, `Bool` as arvo-level type aliases. None had a concrete consumer driving the choice — they were inherited speculation from loimu-primitives. arvo L0 ships raw `UFixed<I, F, S>` / `IFixed<I, F, S>` and the four strategy markers; downstream (hilavitkutin, clause, or anyone else) defines domain aliases as needs surface.

### D2. `Warm` const-bounded to `I + F ≤ 32`

Warm's value proposition is the 2x container safety guarantee on single ops. Above 32 logical bits that guarantee evaporates — no u128 container per Q3, so Warm would degrade silently to u64 (same as Hot). Rather than accept the silent degradation, Warm is a compile-time error at widths it cannot service. At `I + F > 32` the consumer picks Hot (wrapping, accepts overflow risk) or Precise (saturating, carries correctness). This surfaces the trade-off at the type level instead of hiding it.

Encoded as a const-generic bound on the Warm `Strategy` impl (exact mechanism — trait bound, `where`-clause, or `Strategy::Container` associated type with phantom — chosen at src time, whichever is cleanest on nightly).

### D3. Defer `ByteCount` / `Alignment` / `CoreCount` / `Nanoseconds`

Originally grouped as L0 candidates; deferred out of L0 for layer-fit reasons:

- `CoreCount`, `Nanoseconds` — platform-environment concerns. arvo has zero platform dep. These belong adjacent to `ThreadPool` / `Clock` trait definitions, which live in hilavitkutin-api.
- `ByteCount`, `Alignment` — layout / physical-storage concerns. These sit more naturally at the storage boundary (ColumnValue layer in hilavitkutin-api) than in L0 substrate.

arvo's DESIGN.md explicitly lists these as L0 non-members with rationale, so a future reader doesn't mistake their absence for oversight.

### D4. Bit-level aliases in arvo-bits

`Bit`, `Byte`, `Word`, `DWord`, `QWord` — substrate bit-level primitives, not speculative domain types. Default to `Hot` since bit-level ops are typically perf-critical. Ship in arvo-bits.

## Per-crate doc changes

### arvo — `mock/crates/arvo/DESIGN.md.tmpl`

- Add strategy markers: `Hot`, `Warm`, `Cold`, `Precise`, `Strategy` trait
- Document 4-strategy semantics per topic 202603241200:
  - Hot: min aligned container, wrapping arithmetic
  - Warm (default): 2x container, wrapping, const-bounded `I + F ≤ 32` per D2
  - Cold: bitpacked, widen-op-narrow
  - Precise: bitpacked/2x, saturating
- Container width table (1–8 / 9–16 / 17–32 / 33–64 × Hot/Warm/Cold/Precise), noting Warm is unavailable in the 33–64 row
- Cross-width resolution rule: more-conservative strategy wins (Precise > Cold > Warm > Hot)
- `UFixed<const I: u8, const F: u8, S: Strategy = Warm>`, `IFixed<...>` signatures
- `FastFloat`, `StrictFloat`, `Float` alias resolved by `cfg(arvo_fast_math)`
- Numeric traits: `TotalOrd`, `Sqrt`, `Recip`, `Abs`, `FromConstant`; arithmetic via `core::ops`
- Strategy conversion semantics (Hot→Warm zero-extend free, Warm→Hot truncating, etc.)
- Explicit **L0 non-members** section: no semantic aliases at this layer (consumer concern); `ByteCount` / `Alignment` / `CoreCount` / `Nanoseconds` deferred with rationale per D3

### arvo-bits — `mock/crates/arvo-bits/DESIGN.md.tmpl` (stub already exists)

- Traits: `BitWidth`, `BitAccess`, `BitSequence` (hardware-intrinsic-backed: popcnt, cttz, clz)
- Aliases: `Bit<S = Hot>`, `Byte<S = Hot>`, `Word<S = Hot>`, `DWord<S = Hot>`, `QWord<S = Hot>`
- Blanket impls for arvo numerics: F=0 specialisation (trivially bit-accessible) and general F>0 (explicit opt-in for packed-flag treatment)
- Rule: no fielded structs at this layer — contracts only

### arvo-bitmask — `mock/crates/arvo-bitmask/DESIGN.md.tmpl`

- Dependency: arvo → arvo-bits (already reflected in forbidden-imports lint + CLAUDE.md)
- Types: `Mask64`, `Mask256`, `BitMatrix<const R: usize, const C: usize>`
- Set ops (union / intersection / difference / complement) via bit intrinsics
- Built on `BitSequence` / `BitAccess` contracts from arvo-bits
- No direct arvo numeric imports (only arvo-bits)

### arvo-graph — `mock/crates/arvo-graph/DESIGN.md.tmpl`

- Generic over numeric trait bounds; no `UFixed` / `IFixed` imports
- Algorithms: topo sort, upward rank, waist detection, spanning tree, components
- Adjacency via `arvo-bitmask::BitMatrix`
- Example signature: `fn upward_rank<W, const N: usize>(dag: &BitMatrix<N, N>, weights: &[W; N]) -> [W; N] where W: Add<Output = W> + TotalOrd + Copy + FromConstant`
- Consumer chooses safety by choosing W (Hot wraps, Precise saturates, same algorithm)

### arvo-sparse — `mock/crates/arvo-sparse/DESIGN.md.tmpl`

- CSR / COO structures generic over value type V (no numeric bound on V for structure itself)
- RCM reordering, block diagonal, Dulmage-Mendelsohn operate on bit-matrix adjacency — no numeric bound needed for structural algos
- Numeric trait bounds applied only at algorithms that compute on values

### arvo-comb — `mock/crates/arvo-comb/DESIGN.md.tmpl`

- DP cost functions generic: `W: Add<Output = W> + TotalOrd + Copy + FromConstant`
- Greedy grouping fully generic via closures — consumer supplies feasibility predicate + merge function
- Bin-packing variants (first-fit, best-fit, worst-fit) generic over cost type

### arvo-spectral — `mock/crates/arvo-spectral/DESIGN.md.tmpl`

- Power iteration on Laplacian
- Generic over weight W and eigenvector F with `W: Into<F>` boundary conversion
- F bound: `Add<Output = F> + Mul<Output = F> + Sqrt + Recip + Copy + FromConstant`
- Laplacian built from integer weights (arvo fixed-point); eigenvector computation uses Float
