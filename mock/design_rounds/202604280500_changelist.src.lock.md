### Pass 2: Meta-newtype carrier widening (DEFERRED to follow-up round)

**Status: DEFERRED to Round B-2 (or Round C)**.

Implementation attempt during the SRC phase of round 202604280500 hit a `rustc` trait-solver cycle:

```
error[E0391]: cycle detected when evaluating trait selection obligation
              `arvo_storage::meta_bits::IBits: core::marker::Sized`
note: ...which requires building an abstract representation for
      `ufixed::<impl ...>::{constant#0}` ...
note: ...which requires type-checking
      `Hot: UContainerFor<{ ufixed_bits(I, F) }>` ...
note: ...which again requires evaluating
      `IBits: Sized`, completing the cycle
```

Root cause: with `IBits` field widened to `pub Bits<9, Hot>`, `IBits: Sized` evaluation requires resolving `Hot: BitsContainerFor<9, Unsigned>` (which is fine in isolation: u16 is Sized). However, `UFixed`'s where-clause `Hot: UContainerFor<{ ufixed_bits(I, F) }>` calls `ufixed_bits(I: IBits, F: FBits)` at the const-eval position, and the const-eval query for the call-argument's Sizedness recursively re-enters the trait-solver context already busy resolving `IBits`. The cycle detector trips on this.

The original `pub u8` field avoided the cycle because u8's Sizedness is trivially primitive and doesn't go through the trait solver. Substituting `pub Bits<9, Hot>` adds a trait-solver hop the const-eval-position consumer hits during well-formedness checking.

This is a current `rustc` limitation around `feature(generic_const_exprs)` + const-fns over ConstParamTy newtypes whose fields require trait-solver work. Possible resolutions for a follow-up round:

1. Change `Bits` to allow a non-projection field on this specific (N=9, Hot, Unsigned) shape (carrier-cell shortcut bypassing `BitsContainerFor`).
2. Move `meta_bits` types out of the const-eval-call chain (decouple `ufixed_bits` from `IBits` directly; pass raw u8 through helpers).
3. Wait for a `rustc` cycle-detector improvement that disambiguates the const-eval-from-where-clause path.

**Resolution for Round B**: revert the carrier-widening attempt; meta-newtypes stay at `pub u8` for this round. The doc CL claim (`pub Bits<9, Hot>` field) is the design destination per "docs=design"; source achieves the destination in a follow-up round. The lint pack does not fire on this specific doc-source mismatch (verified during Round B SRC validation: `cargo mock --lint-only --commit` clean post-revert).

The bounds-check macro `impl_u_narrow!` uses u128 internally; for the new u128 cells, a `u256`-equivalent would be needed. Workaround: for 33..=64 Warm/Precise narrow, the source is u128 and the target is u64; the comparison `(v as u128) > max_u128` resolves with `max_u128 = (1u128 << bits) - 1` for `bits <= 64`. For 65..=128, the source and target are both u128 and the bounds check uses `bits = 128` → `1u128 << 128` overflows. Special-case 128 to skip the bounds check (the value is already at maximum).

### Pass 4: MultiContainer<HiT, LoT> + 129..=255 storage routing

**Layer placement note (mid-pass discovery)**: the doc CL placed `MultiContainer` in `arvo-storage/DESIGN.md.tmpl`. However, `arvo-storage` depends on `arvo-strategy` (not the reverse), and `arvo-strategy/src/container.rs` needs `MultiContainer<HiT, LoT>` as the `UContainerFor<N>::T` projection target for 129..=255 cells. Resolution: `MultiContainer` is *defined* in `arvo-strategy/src/multi_container.rs` and *re-exported* from `arvo-storage::MultiContainer` so the public surface still reads "storage primitive in arvo-storage" (consumers `use arvo_storage::MultiContainer` per the doc CL). The internal source organization places it one layer down to break the cycle. `BitPrim` follows the same pattern, defined in `arvo-strategy` and re-exported from `arvo-storage`.

`arvo-strategy/src/multi_container.rs` (new file):
- Declare `BitPrim` sealed trait. Impl for `u8` / `u16` / `u32` / `u64` / `u128` (unsigned native primitives) and `i8` / `i16` / `i32` / `i64` / `i128` (signed). Sealed via the existing `sealed::Sealed` trait. Bounds: `Copy + Clone + PartialEq + Eq + PartialOrd + Ord + Default + Hash + Debug + 'static` (matches `UContainerFor<N>::T` minimum surface).
- Declare `pub struct MultiContainer<HiT: BitPrim, LoT: BitPrim> { pub hi: HiT, pub lo: LoT }` with `repr(C)` and standard derives (`Copy`, `Clone`, `PartialEq`, `Eq`, `PartialOrd`, `Ord`, `Default`, `Hash`, `Debug`). The auto-derived bounds propagate via `HiT: BitPrim` / `LoT: BitPrim` ensuring all required traits are present.
- Construct via `pub const fn new(hi: HiT, lo: LoT) -> Self`.
- No `from_raw` / `to_raw` accessors at this layer (no single "raw" representation; consumers reach `.hi` / `.lo` directly).
- No `ConstParamTy_` impl: `MultiContainer` is never in const-generic position; it appears only as the `UContainerFor<N>::T` associated type.

`arvo-strategy/src/lib.rs`:
- Add `mod multi_container;` and `pub use multi_container::{BitPrim, MultiContainer};`.

`arvo-storage/src/lib.rs`:
- Add `pub use arvo_strategy::{BitPrim, MultiContainer};` so the documented public surface (`arvo_storage::MultiContainer`) holds.
