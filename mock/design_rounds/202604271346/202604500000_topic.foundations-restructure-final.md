## Decisions

### CR-1 resolved: ConstParamTy scope narrowed to Bits, IBits, FBits, Width

Prototype against current nightly showed `core::marker::ConstParamTy` is derive-macro-only on this rustc version; using it as a trait bound (e.g., on `UContainerFor::T`) is non-trivial. arvo's existing `UContainerFor::T` bound does not declare ConstParamTy, and adding it creates new soundness questions that this round cannot honestly resolve in scope.

The narrowing: this round derives `ConstParamTy` on `Bits<7, Hot>` (single-u8-field tuple struct, no projections, follows arvo's existing tuple-struct derive precedent on `IBits(pub u8)`), `IBits(pub Bits<7, Hot>)`, `FBits(pub Bits<7, Hot>)`, `Width(pub Bits<7, Hot>)`, and the `Strategy` markers (`Hot`/`Warm`/`Cold`/`Precise` — already partially derived; complete the set).

The round does NOT derive `ConstParamTy` on `UFixed<I, F, S>` or `IFixed<I, F, S>`. No const-generic position in this round's surface takes a `UFixed`; consumers reach UFixed via use-site values, not const-generic args. The aliases (`Uint8`, `Int16`, etc.) are type aliases over UFixed instantiations; aliases do not require their underlying type to be ConstParamTy.

Open follow-up: `ConstParamTy on UFixed / IFixed` lands as its own round once a concrete consumer needs it. BACKLOG entry in arvo/BACKLOG.md.tmpl: "ConstParamTy on UFixed/IFixed. Required when a consumer wants `const X: UFixed<...>` const-generic positions. Today no consumer surface needs this. Lift requires resolving the associated-type-projection cascade in `<S as UContainerFor<N>>::T` field positions; verified-friction on rustc 1.96-nightly."

This narrowing is the reviewer's prior "scope-correct minimum viable" recommendation applied at the ConstParamTy boundary. Everything else the user wanted (alias layer, sweep, Fnv1a, USize newtype, BitWidth rename) stays in scope.

### CR-2 resolved: USize keeps `Deref<Target = usize>`

The escape via Deref to `usize` is required for `array[*idx]` ergonomics, which is USize's primary use case. Replacing with a `Bits<N, Hot>` Deref breaks every consumer call site (Bits has no `Index` impl by deliberate design — it is a non-arithmetic opaque-bit primitive).

Decision: `USize` stays as a newtype. Field type changes from bare `usize` to `Bits<{cfg-platform-width}, Hot>`. `Deref<Target = usize>` is preserved by routing through the inner `Bits`'s `to_raw` method (`*idx` still produces `usize` for `array[*idx]` ergonomics).

```rust
#[cfg(target_pointer_width = "64")]
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct USize(pub Bits<64, Hot>);

#[cfg(target_pointer_width = "32")]
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct USize(pub Bits<32, Hot>);

impl Deref for USize {
    type Target = usize;
    fn deref(&self) -> &usize {
        // SAFETY: Bits is repr(transparent) over its container.
        // For Bits<64, Hot> the container is u64 on 64-bit targets, layout-equivalent to usize.
        // For Bits<32, Hot> the container is u32 on 32-bit targets, layout-equivalent to usize.
        unsafe { &*(&self.0 as *const Bits<_, Hot> as *const usize) }
    }
}
```

The `unsafe` Deref is the workspace-rule-exception #4 boundary (ergonomic helper that returns a bare primitive). Document the soundness: Bits is repr(transparent), the cfg-gated width matches `target_pointer_width`, the underlying container IS `usize`'s layout-equivalent.

This preserves every existing call site. No ripple beyond the field-type change.

### CR-3 resolved: `Fnv1a<N>` upper bound via const-eval guard

The const-eval guard `where [(); 1 / ((width_le_64::<N>()) as usize)]:` is the chosen mechanism. The sealed-trait alternative was rejected because the impl-on-values question (sealed traits go on types, not const-generic values, requiring phantom-typed-by-value indirection that adds mechanism without adding clarity).

The const-eval guard depends on `width_u8(Width) -> u8` being usable in anonymous-const context. arvo already ships analogous helpers (`ibits_u8`, `fbits_u8` in strategy.rs at #71's tracked exception), proving the pattern works. `width_u8` follows the same shape.

Diagnostic-quality message: add `#[diagnostic::on_unimplemented]` (rustc 1.78+, stable in this nightly) on the where-clause failure to produce a readable error. This is the ergonomic cost of choosing const-eval over sealed-trait.

```rust
pub struct Fnv1a<const N: Width>
where
    Hot: UContainerFor<{ width_u8(N) }>,
    [(); 1 / (width_le_64(N) as usize)]: ;

const fn width_le_64(n: Width) -> bool {
    width_u8(n) <= 64
}
```

The error on `Fnv1a::<{ width_of(96) }>` surfaces as a const-eval division-by-zero, attached to the where clause site. Acceptable.

### H-1 resolved: Bits constructor surface

Three constructors exist after this round; their roles are partitioned:

- `Bits<N, S>::from_raw(<S as UContainerFor<N>>::T) -> Bits<N, S>`: the general associated-type-typed constructor. Stays.
- `Bits<N, S>::from_raw_u64(u64) -> Bits<N, S>` where `Hot: UContainerFor<N>`: per-container-width convenience that **masks input to N bits** before storing. Replaces today's `Bits::new(u64)`. Per-N-class siblings (`from_raw_u8`, `from_raw_u16`, `from_raw_u32`) exist for narrower containers; `from_raw_u64` is the universal masking entry.
- `Bits::new(u64)`: deprecated in this round, removed in a follow-up. The surface collapses to `from_raw` + `from_raw_uN`.

`Bits<N, S>::to_raw(self) -> <S as UContainerFor<N>>::T`: the existing accessor. Stays. `to_raw_u8(self) -> u8` is the symmetric sibling for the meta-layer use case (reading `Width.0.0` in `width_u8`).

### H-2 resolved: `from_raw_uN` masks to N bits

Critical for ConstParamTy soundness. `Bits<7, Hot>::from_raw_u8(255)` masks to `255 & 0b0111_1111 = 127`. Two const-generic args constructed via `Width(Bits::<7, Hot>::from_raw_u8(7))` and `Width(Bits::<7, Hot>::from_raw_u8(135))` (which masks to 7) compare structurally equal. Without masking, `135` stays as 135 in the underlying byte and the two values are unequal.

```rust
impl<const N: u8, S: Strategy> Bits<N, S>
where Hot: UContainerFor<N>, S: UContainerFor<N>
{
    pub const fn from_raw_u64(raw: u64) -> Self {
        let mask: u64 = if N == 64 { u64::MAX } else { (1u64 << N) - 1 };
        let masked = raw & mask;
        // ... convert masked to <S as UContainerFor<N>>::T via narrow-cast helpers ...
    }
}
```

The narrow-cast from u64 to the actual container type is the SRC-phase mechanism question. arvo already ships `Bits::new(u64)` with this exact shape; `from_raw_u64` adopts the same body.

`from_raw_u8` / `from_raw_u16` / `from_raw_u32` are width-class siblings: `from_raw_u8` for Bits where the container is u8, `from_raw_u16` for u16-class, etc. Each masks within its width class.

### H-3 resolved: drop struct-level Warm default on UFixed/IFixed

Two sources of Warm-default (struct-level + alias-level) creates drift risk. Drop the struct-level default. Aliases carry the strategy default at the consumer surface.

```rust
// Before:
pub struct UFixed<const I: IBits, const F: FBits, S: Strategy = Warm> { ... }

// After:
pub struct UFixed<const I: IBits, const F: FBits, S: Strategy> { ... }

// Aliases keep the default:
pub type Uint8<S = Warm> = UFixed<{ ibits(8) }, { fbits(0) }, S>;
```

Existing call sites that wrote `UFixed<I, F>` (relying on the struct default) gain an explicit `Warm` or migrate to an alias. The migration is part of this round's sweep.

### H-4 resolved: rename existing `BitWidth` trait to `HasBitWidth`

Mechanical rename of `arvo_bits::BitWidth` → `arvo_bits::HasBitWidth`. Verb-y, captures intent ("this type has a bit width"). New `Width` newtype keeps that name. Ripples through every blanket impl in arvo-bits and every consumer's `T: BitWidth` bound.

### H-5 resolved: `#[diagnostic::on_unimplemented]` on `UContainerFor`

Add to `UContainerFor` (and `IContainerFor`) the diagnostic attribute with a strategy-pointing message:

```rust
#[diagnostic::on_unimplemented(
    message = "strategy `{Self}` does not provide a container for {BITS}-bit width",
    note = "Warm caps at I+F<=32; for wider widths, choose Hot or Precise explicitly: `Uint64<Hot>`, `Uint64<Precise>`, etc.",
)]
pub trait UContainerFor<const BITS: u8>: Strategy { type T: ...; }
```

This makes the Warm-default-Uint64 error point at strategy choice in human-readable terms.

### Bits-to-L0 migration

arvo-bits stays as L1 trait-only crate: `HasBitWidth` (renamed), `BitAccess`, `BitSequence`, `BitLogic` traits + blanket impls on arvo's primitives.

### arvo-bits emptiness acknowledgement

Post-restructure arvo-bits is meaningfully smaller (traits + blanket impls only). The crate stays separate at L1 because:

- The trait surface IS the L1 contract layer; merging into L0 would conflate "concrete primitives" with "traits asserting properties of those primitives."
- arvo-bitmask, arvo-tensor, arvo-graph, etc. depend on arvo-bits for trait surface; flattening would merge L0 + L1 + the algorithm crates' direct trait dependencies.
- The L0/L1 split mirrors arvo's existing layering convention; breaking it requires its own design round.

Future merge consideration: if a follow-up audit shows arvo-bits has degraded to a near-empty crate with low maintenance value, a "merge L0 + L1" round becomes appropriate. Today it doesn't; the trait contracts have real load-bearing weight.

### C2 prose corrections (third topic carried bugs)

The third topic's prose contained three errors flagged by the second-pass reviewer:

1. The proposed `_phantom: PhantomData<S>` field on UFixed is redundant with `bits: Bits<..., S>`. **Drop it.**
2. The PhantomData<fn() -> S> → PhantomData<S> change does NOT change variance. Both are covariant (or invariant in the marker-type case). The actual change is auto-trait inheritance. Reword: "Strategy markers are unit structs and Send/Sync trivially; the change preserves auto-trait inheritance from S without spurious function-pointer phantoms."
3. The cascade prose said "ConstParamTy cascades through Bits + IBits + FBits + S". S is a type parameter, not a const-generic param. Only `I: IBits` and `F: FBits` const-generic positions need ConstParamTy. Reword: "The cascade requirement applies to the const-generic params `I: IBits` and `F: FBits`. The type parameter `S: Strategy` is a marker type, not a const value."

Per CR-1 narrowing, these prose corrections matter only for the BACKLOG entry tracking the eventual ConstParamTy-on-UFixed work; the corrected prose lands there.
