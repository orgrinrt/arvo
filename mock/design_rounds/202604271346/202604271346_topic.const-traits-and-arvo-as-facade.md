**Date:** 2026-04-27
**Phase:** TOPIC (frozen on commit)
**Round:** 202604271346. Const traits, arvo as facade, ergonomic Int/Uint generics
**Supersedes:** None (first topic of new round). Round 202604500000 was deprecated when the unified-API direction outgrew its scope.

This topic captures the structural redesign that fell out of round 202604500000's source execution. Three forces converged:

1. The strategy-bound trilemma (`mock/research/strategy-bound-trilemma.md`) made clear that per-size helpers on `Bits` (`from_raw_u8` / `from_raw_u64` / `to_raw_u64` / `From<u64>` per-N + the `impl_bits_u64` macro) are a sprinkled smell, not a unified surface.

2. RFC 3762 (the const-trait-impl successor to RFC 2632) verified working on current nightly via spike. `pub const trait` declarations, `impl const Trait`, and `T: [const] Trait` bounds compile cleanly. Risk class: comparable to `generic_const_exprs`. This unlocks expressing nearly the entire substrate as const traits.

3. The `Narrow<T>` trait (the unified mask-and-cast bridge replacing per-size helpers) needs Mask primitives. Mask primitives live in `arvo-bitmask` (L2). `Bits` lives in `arvo` (L0). `Bits` wanting to use `Narrow` would invert the dependency.

The redesign: arvo becomes a downstream facade, not L0. Real L0 shrinks to a minimal set of bare-primitive wrappers. Trait contracts split into granular crates per concern. Implementations sit in domain-organised crates. The facade re-exports the public surface and provides default impls connecting contracts.

## Decisions

### D-1: arvo becomes the downstream facade

`arvo` (the crate) is no longer L0. It sits at the bottom of the dependency graph, downstream from every other workspace crate. Its job is to:

1. Re-export the public surface every consumer cares about.
2. Provide default impls that connect trait contracts (so consumers get composed behaviour for free).
3. Define ergonomic aliases (`Int<N>`, `Uint<N>`).
4. Host `UFixed<I, F, S>` and `IFixed<I, F, S>` as compositions over `Bits<N, S>` (see D-3).

`arvo` itself contains no fielded primitive types. Every concrete type in arvo is a newtype, an alias, or a default-impl carrier; every load-bearing primitive lives in a smaller upstream crate.

### D-2: Real L0 — minimal bare-primitive layer

A new minimal L0 crate (working name TBD; candidates: `arvo-storage`, `arvo-prim`, `arvo-bits-storage`) hosts the smallest possible set of types whose existence requires wrapping bare primitives. The expected contents:

| Type | Purpose |
|---|---|
| `Bits<const N: u8, S: Strategy>` | The N-bit storage primitive. Only fielded value type at L0. |
| `Hot` / `Warm` / `Cold` / `Precise` | Strategy markers. ZSTs. |
| `Bool` | Wraps `bool`. Not a fixed-point type; required for predicates. |
| `USize` | Wraps `usize`. Platform-pointer-width primitive. |

`IBits`, `FBits`, `Width` may also live here if D-A (the typed-meta-bits spike, see Open Questions) succeeds and they become real `Bits<7, Hot>` newtypes. If the spike fails they stay as bare-u8 newtypes in this same crate.

That is the entire L0 surface. Everything else — bit ops, narrowing, strategy-container dispatch, masks, fixed-point, algorithms — lives upstream of L0 in trait-contract or implementation crates.

### D-3: UFixed / IFixed become compositions over Bits

UFixed and IFixed are load-bearing for arvo's fixed-point-first identity. They stay in the substrate.

But they are no longer fielded over `<S as UContainerFor<N>>::T` directly. They become `#[repr(transparent)]` newtypes over `Bits<{I+F}, S>` (or `Bits<{1+I+F}, S>` for IFixed). The arithmetic/semantic behaviour comes via const trait impls of `Add`/`Sub`/`Mul`/`Div`/`Recip`/`Sqrt`/etc., not via direct field manipulation.

This means UFixed/IFixed are "Bits with arithmetic semantics" rather than "their own primitive". They live in arvo (the facade), not L0.

### D-4: Trait contracts split into granular crates by concern

Per the workspace principle of optimising transitive dependency *size* (not count), trait contracts split per concern. Each downstream crate pulls in only the contracts it implements or consumes. Initial planned split:

| Crate | Contracts |
|---|---|
| `arvo-transparent` | `Transparent` (const unsafe), `NumericPrimitive` (marker), free `arvo::raw` fn |
| `arvo-strategy` | `Strategy` marker, `UContainerFor<N>` (const), `IContainerFor<N>` (const), `Resolve<S1, S2>` |
| `arvo-bits-contracts` | `HasBitWidth` (const), `BitAccess` (const), `BitSequence` (const), `BitLogic` (const), `BitPrim` family (const) |
| `arvo-mask-contracts` | `Mask<W>` (const) — abstract trait over Mask64/Mask256/etc. |
| `arvo-narrow-contracts` | `Narrow<T>` (const) plus the typed type alias `Narrowed<N, T>` (or whatever name lands) |
| `arvo-numeric-contracts` | `Abs` (const), `Recip` (const), `Sqrt` (const), `TotalOrd` (const), `FromConstant` (const), `Predicate` family |

Crates may merge if the granularity has zero practical impact (e.g., `arvo-transparent` and `arvo-strategy` may end up being one crate if every consumer of one needs the other). The split is a starting structure; the SRC phase consolidates as appropriate.

The crate-conflict-resolution rule applies throughout: when X needs Y and Y needs X, extract the shared contract upstream. Both depend on the contract; neither on each other.

### D-5: Implementation crates stay domain-organised

The existing layout largely survives, with renames and refactors:

| Crate | Implements / hosts |
|---|---|
| `arvo-bits` | `BitAccess`/`BitSequence`/`BitLogic`/`HasBitWidth`/`BitPrim` impls on `Bits<N, S>` and primitive containers; aliases `Bit`/`Nibble`/`Byte`/`Word`/`DWord`/`QWord` |
| `arvo-bitmask` | `Mask<W>` contract impls + `Mask64` / `Mask256` / `BitMatrix` |
| `arvo-narrow` | `Narrow<T>` contract impls (uses `Mask<W>` from contract crate, abstracted). Concrete instances on the relevant primitive types. |
| `arvo-tensor` | `Array<T, N>`, `Matrix<W, N>`, `Cap`, `Enumerator` |
| `arvo-graph` | DAG analysis algorithms |
| `arvo-sparse` | CSR / RCM / block-diagonal / Dulmage-Mendelsohn |
| `arvo-spectral` | Laplacian / Fiedler / power iteration |
| `arvo-comb` | DP / greedy / bin-packing |
| `arvo-hash` | `Hasher<N>` (const trait now), `Fnv1a<N>`, `ContentHash` |

`arvo-narrow`'s placement: downstream of `arvo-mask-contracts` (uses the Mask trait), downstream of `arvo-bits-contracts` (uses BitLogic for the masking operation). Concrete masks come from `arvo-bitmask` impls — `arvo-narrow` does not redefine masks or bypass the contract. Only the Mask trait surface is named at the contract layer.

### D-6: Embrace `feature(const_trait_impl)` + `feature(const_param_ty_trait)` (RFC 3762 line)

Verified compiling on current nightly (rustc 1.96-nightly fda6d37bb 2026-03-27) via two spikes. `pub const trait Foo` / `impl const Foo for T` / `T: [const] Foo` bounds. Same risk class as `generic_const_exprs` — incomplete but actively developed (RFC 3762, tracking issue rust-lang/rust#143874). RFC 2632 line is closed in favour of this.

Every trait in the contract crates is `pub const trait` where its methods can sensibly be called from const context. Default impls in the facade likewise.

Feature flag set for arvo crates after this round:

- `feature(adt_const_params)` — already used
- `feature(generic_const_exprs)` — already used
- `feature(const_trait_impl)` — added
- `feature(const_param_ty_trait)` — added (manual `ConstParamTy_` impls on `Bits<N, S>` per Q-A)
- `feature(macro_metavar_expr_concat)` — already used
- `feature(try_trait_v2)` — already used (notko-derived)

`[const]` syntax constraint: `T: [const] Trait` bounds are only allowed in const fn signatures and `impl const Trait` blocks. Non-const impls and struct definitions use the plain `T: Trait` form for the same bound; the const-callability is established at the use site.

### D-7: Per-size helpers deleted from `Bits`

The entire family is gone:

- `Bits::<N, Hot>::new(u64)` (deprecated)
- `Bits::<N, Hot>::bits()` (deprecated)
- `Bits::<N, Hot>::from_raw_u64`
- `Bits::<N, Hot>::from_raw_u8`
- `Bits::<N, Hot>::to_raw_u64`
- `Bits::<N, Hot>::to_raw_u8`
- `From<u64> for Bits<N, Hot>` per-N
- The `impl_bits_u64` macro itself

Bits keeps only:
- `Bits::from_raw(<S as UContainerFor<N>>::T)` (typed-container in)
- `Bits::to_raw() -> <S as UContainerFor<N>>::T` (typed-container out)
- `Deref<Target = ::T>`, `AsRef<::T>`
- `Transparent` impl (const)

All u64↔container narrowing flows through `Narrow<T>::narrow_to::<const N: u8>(self) -> T` (the const trait, in `arvo-narrow-contracts`). Mask construction goes via the `Mask<W>` contract; `Narrow`'s impl uses Mask + BitLogic + the `as` operator.

### D-8: `Fixed<const I: u8, const F: u8, S>` and `Signed<...>` deleted

The bare-u8 ergonomic aliases for `UFixed`/`IFixed` are dropped entirely. They were a cop-out around the trilemma constraint. `IBits`/`FBits` typed const generics exist precisely to prevent the swap risk that bare-u8-twice reintroduces.

Consumer call sites for non-integer fixed-point use `UFixed<{ibits(I)}, {fbits(F)}, S>` directly. Verbose but correct. The trilemma research note documents this trade.

### D-9: `Int<const N: u8, S = Warm>` and `Uint<const N: u8, S = Warm>` ergonomic aliases

The convenience layer for integer types. Bare `const N: u8` is acceptable here because:

1. Integer types collapse to F=0; only one width param, no swap risk.
2. Parallel to `Bits<const N: u8, S>` itself (foundational storage primitive uses bare u8 in the same position).
3. Replaces 254 individual `Uint2..Uint256` / `Int2..Int256` aliases with one each.

Semantic convention: **N = total storage bit count**. `Int<8>` ↔ `i8` semantics (1 sign + 7 integer bits internally). `Uint<32>` ↔ `u32` semantics (32 magnitude bits). This matches std and reads cleanly.

```rust
// In arvo (the facade):
pub type Uint<const N: u8, S = Warm> = UFixed<{ ibits(N) }, { fbits(0) }, S>;
pub type Int<const N: u8, S = Warm>  = IFixed<{ ibits(N - 1) }, { fbits(0) }, S>;
```

`S` defaults to `Warm` so monomorphic call sites stay clean (`let x: Uint<32> = ...`); explicit-strategy variants come naturally (`Uint<32, Hot>`).

The canonical-width aliases (`Uint5..Uint64`, `Int7..Int64`) become redundant once `Uint<N>` / `Int<N>` exist. They land for deletion in the SRC phase or stay as documentation-friendly shorthand — TBD per usage audit.

Cap: `N <= 64` implicitly via `Hot: UContainerFor<N>` (or `Warm`/`Precise` for narrower). Wider widths require Cold (bitpacking) and aren't in scope this round.

### D-10: bitfield macro stays as DSL sugar, generated bodies trivialise

`bitfield!` keeps its current DSL surface (declarative struct + slot list). What changes: the generated bodies stop doing inline `Bits::new(u64)` / `.bits()` arithmetic. Instead they delegate to const trait methods (`Slot::<P, LO, HI>::read(parent)`, `Narrow::<...>::narrow_to::<N>(...)`, etc.). The generated code shrinks substantially and stops being a per-N arithmetic-heavy expansion. The macro is purely a struct-and-method-stub generator.

### D-11: `Transparent` and `NumericPrimitive` realigned

`NumericPrimitive` returns to its clean marker-only role. No methods. Identifies the bottom-out primitives that arvo wrappers transmute to. Lives in `arvo-transparent`.

`Transparent` becomes `pub const unsafe trait`. The default `raw()` method body uses `transmute_copy` and is now const-callable. This eliminates the macro's emitted-twice surface (currently emits both a trait impl AND a separate inherent `pub const fn raw(self)` per wrapper). Const trait collapses to one impl.

Free fn `arvo::raw<W, T>(w: W) -> T where W: [const] Transparent<Inner = T>` keeps its prefix-style ergonomics, callable in const context.

### D-12: Const-trait sweep across existing trait families

The following traits become `pub const trait` in their contract crates:

- `UContainerFor<N>` — type-only, no methods. Const trait so `<S as UContainerFor<N>>::T` resolves in const fn bodies.
- `IContainerFor<N>` — same.
- `HasBitWidth` — exposes `WIDTH: u8` const; trait const so `T::WIDTH` works in const fn bodies under generic-T const fn.
- `BitAccess`, `BitSequence`, `BitLogic` — all methods are pure bit ops, inherently const.
- `BitPrim`, `UBitContainer`, `IBitContainer` — primitive bridge family.
- `Abs`, `Recip`, `Sqrt`, `TotalOrd`, `FromConstant` — pure numeric ops.
- `Hasher<N>`, `HasherExt<N>` (arvo-hash) — `hash_const` becomes natural fallout of const trait.
- `Predicate` family (#122) — pure boolean predicates.

Traits that stay non-const: anything with side effects or runtime-only behaviour (none currently in this list, but the rule is the principle).

## Resolved questions

### Q-A resolved: typed meta-bits spike succeeded — IBits/FBits/Width become real Bits<7, Hot> wrappers

Spike on rustc 1.96.0-nightly (commit fda6d37bb 2026-03-27) compiled cleanly with `feature(adt_const_params)` + `feature(const_trait_impl)` + `feature(const_param_ty_trait)` + `feature(generic_const_exprs)`:

```rust
pub const trait UContainerFor<const N: u8>: Strategy {
    type T: Copy + ConstParamTy_ + ...;
}

impl const UContainerFor<7> for Hot {
    type T = u8;
}

#[repr(transparent)]
pub struct Bits<const N: u8, S: Strategy>(pub <S as UContainerFor<N>>::T)
where S: UContainerFor<N>;

impl<const N: u8, S: Strategy> ConstParamTy_ for Bits<N, S>
where 
    S: UContainerFor<N>,
    <S as UContainerFor<N>>::T: ConstParamTy_,
{}

#[derive(ConstParamTy, ...)]
#[repr(transparent)]
pub struct IBits(pub Bits<7, Hot>);

pub const fn ibits(n: u8) -> IBits { IBits(Bits::<7, Hot>(n)) }

pub struct Carrier<const I: IBits>;
pub type Seven = Carrier<{ ibits(7) }>;  // works
```

Decision: `IBits`, `FBits`, `Width` become real `Bits<7, Hot>` newtypes. The projection cascade resolves via const-trait machinery — non-const trait dispatch was the blocker, const trait fixes it. Meta_bits_wrapper macro shrinks because the inner type is now a real arvo primitive.

`feature(const_param_ty_trait)` joins the existing feature flags. Risk class: same incomplete-features bucket as `generic_const_exprs`. Issue: rust-lang/rust#95174 (S-tracking-needs-impl-or-find-a-bug, actively under impl).

`[const]` bound caveat: `[const] UContainerFor<N>` only allowed in const fn signatures and const-trait impls. Plain (non-const) impl blocks and struct definitions use `UContainerFor<N>` (non-const form) for the bound; const-callability is established at use sites that need it.

### Q-B resolved: initial split per D-4, doc CL phase audits for collapse

The six-crate split (`arvo-transparent`, `arvo-strategy`, `arvo-bits-contracts`, `arvo-mask-contracts`, `arvo-narrow-contracts`, `arvo-numeric-contracts`) is the starting point. During doc CL, the import-set audit identifies any pairs where 100% of consumers need both — those merge. Mergers happen by mechanical rule, not preserved-for-its-own-sake structure.

### Q-C resolved: option 2 — Narrow impls in arvo-bitmask, contracts in arvo-narrow-contracts

`Narrow<T>` trait declaration lives in `arvo-narrow-contracts` (uses `Mask<W>` from `arvo-mask-contracts` + `BitLogic` from `arvo-bits-contracts`). Concrete `Narrow<T>` impls (per source/target primitive pair) live in `arvo-bitmask` alongside the Mask concretes they bind to. Generic-over-Narrow consumer code depends only on the contract crate and doesn't pull in mask concretes.

### Q-D resolved: option 2 — std-parallel canonical aliases only (8 total)

Surviving canonical-width aliases (in the arvo facade):

```rust
pub type Uint8<S = Warm>  = Uint<8, S>;
pub type Uint16<S = Warm> = Uint<16, S>;
pub type Uint32<S = Warm> = Uint<32, S>;
pub type Uint64<S>        = Uint<64, S>;  // no Warm; needs Hot/Precise
pub type Int8<S = Warm>   = Int<8, S>;
pub type Int16<S = Warm>  = Int<16, S>;
pub type Int32<S = Warm>  = Int<32, S>;
pub type Int64<S>         = Int<64, S>;   // no Warm; needs Hot/Precise
```

All other width aliases (`Uint5`, `Uint6`, `Uint7`, `Uint13`, `Int7`, `Int13`) get deleted. Consumers wanting those widths write `Uint<5>`, `Int<13>` etc. directly via the generic form.

## Scope explicitly excluded

- `ConstParamTy on UFixed / IFixed` — separate round once a consumer needs it.
- `feature(implied_bounds)` adoption — separate round once that feature stabilises (tracked: rust-lang/rust#44491). Trilemma bound restatement remains a constraint until then.
- Cold strategy bitpacking + widths beyond 64 — separate round.
- Algorithm crate refactors (graph/sparse/spectral/comb) — only updates to use new Narrow/Cast surfaces; no algorithmic changes.
- New numeric op traits (matrix ops, etc.) — out of scope.
