## Semantic aliases are strategy-generic

Aliases fix the `<I, F>` configuration (that's the semantic meaning)
but are generic over the strategy with a sensible default:

```rust
pub type Ratio<S: Strategy = Warm> = UFixed<0, 8, S>;
pub type Factor<S: Strategy = Warm> = UFixed<2, 8, S>;
pub type Norm<S: Strategy = Warm> = IFixed<1, 14, S>;
pub type Coord<S: Strategy = Warm> = IFixed<16, 16, S>;
pub type Angle<S: Strategy = Warm> = UFixed<9, 7, S>;
pub type Count<S: Strategy = Warm> = UFixed<32, 0, S>;
pub type Average<S: Strategy = Warm> = UFixed<16, 16, S>;
pub type Deviation<S: Strategy = Warm> = UFixed<8, 24, S>;
pub type Bool<S: Strategy = Hot> = UFixed<1, 0, S>;
```

Usage:
```rust
Column<Angle>           // Warm by default
Column<Angle<Hot>>      // Hot for morsel-loop-heavy column
Column<Angle<Precise>>  // Precise for audit-critical output
```

Generic code over strategy:
```rust
fn process<S: Strategy>(angle: Angle<S>, ratio: Ratio<S>) -> Angle<S> {
    // both use S's semantics
}
```

The strategy propagates through the type system. If a WU reads
`Column<Angle<Hot>>` and writes `Column<Angle<Precise>>`, the
cross-strategy conversion rules from the prior topic apply at the
WU boundary.

## Open question resolutions

### Q1. Warm debug vs release

**Resolution: Warm is the same everywhere.** No debug/release split.

Warm's safety comes from the 2x container, not from runtime checks.
A single add/sub/mul of two values within their logical range cannot
overflow the 2x container — this is a mathematical guarantee, not a
runtime assertion. If you chain enough operations to overflow the 2x
container, that's a logic error in the consumer's code, and it wraps
silently (same as Hot, but requiring much more accumulation to hit).

If you want debug assertions on overflow, use Precise — it saturates
and you can add debug_assert checks on clamping. The strategies are
explicit: pick the one that matches your intent.

### Q2. IFixed signed containers

**Resolution: same widening rules, signed containers.**

IFixed<I, F, S> follows the same strategy mechanics as UFixed:
- Hot: minimum signed container (i8, i16, i32, i64)
- Warm: 2x signed container
- Cold: minimum signed, bitpacked
- Precise: 2x signed, saturating

Signed saturation clamps to `i_MIN..i_MAX` of the logical range
(not the container range). The 1 sign bit is always present in
IFixed. The container accounts for it: IFixed<7, 8, Hot> has
1 + 7 + 8 = 16 logical bits, stored in i16 (Hot) or i32 (Warm).

### Q3. Maximum logical width and u128

**Resolution: I + F <= 64. No u128 containers.**

The ColumnValue constraint is ≤16 bytes (R4). u128 is 16 bytes and
would consume the entire column value budget for a single number.
This leaves no room for composite types (structs with multiple
fields).

Practical limit: I + F <= 64 for UFixed, 1 + I + F <= 64 for IFixed.
The backing container for Hot is u64 max. For Warm, the 2x container
for a 64-bit logical value would need u128 — which we reject.

Therefore: Warm and Precise for I + F > 32 use u64 as the container
(same as Hot). The 2x widening benefit only applies when I + F <= 32.
For I + F > 32, Warm degrades to Hot's container width but retains
wrapping semantics. Precise for I + F > 32 uses u64 with saturating
ops (no extra container headroom for multiply intermediates — the
saturating clamp catches overflow).

| I + F | Hot | Warm | Cold | Precise |
|---|---|---|---|---|
| 1-8 | u8 | u16 | bitpacked | u16 |
| 9-16 | u16 | u32 | bitpacked | u32 |
| 17-32 | u32 | u64 | bitpacked | u64 |
| 33-64 | u64 | u64 (degrades) | bitpacked | u64 (saturating) |

This means Warm is most useful at I + F <= 32 where it gets the
full 2x headroom. At I + F > 32, Warm and Hot have the same
container — the only difference is the default intent (Warm says
"I expected safety" even though the container can't guarantee it
at this width).

### Q4. Mixed-strategy columns in one fiber

**Resolution: the holistic feasibility check uses physical width.**

The domain 14 feasibility check already accounts for
`size_of::<T>()` per column (R3, type-native stride). The strategy
marker determines the container, which determines `size_of`. A
`Column<Angle<Hot>>` contributes `size_of::<u16>() = 2` bytes to
the check. A `Column<Angle<Warm>>` contributes `size_of::<u32>() = 4`
bytes. The check sees the physical reality.

No special handling needed. The type system carries the strategy,
the strategy determines the container, `size_of` reports the
container, the feasibility check uses `size_of`. It just works.
