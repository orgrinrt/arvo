# arvo — Architecture Principles

## no_std, no alloc

Every crate is `#![no_std]` with no allocation. All collections are
const-sized. Consumers own allocation strategy; arvo never calls out.

## Strategy markers are load-bearing

`Hot` / `Warm` / `Cold` / `Precise` are not decoration. They drive:
- Container physical width (u8 / u16 / u32 / u64 / arbitrary).
- Codegen decisions (branch hints, fused-mul-add, prefetch).
- Tradeoff clarity — a `Mass<Hot>` is explicitly lossy in exchange for
  speed; a `Mass<Precise>` is explicitly exact in exchange for cost.

Default strategy is `Warm` unless called out. Every semantic alias
carries a default: `Angle<S: Strategy = Warm>`.

## Monomorphisation is the dispatch

No `dyn`. No `TypeId`. No `std::any`. Sub-crates are generic over
trait bounds (`BitSequence`, `BitAccess`, numeric trait contracts);
consumers pick concrete instantiations at use site.

The monomorphised function IS the type identity. Compiler proves
devirtualisation; we never pay for dispatch.

## Contracts before concretes

- `arvo` declares numeric primitives.
- `arvo-bits` declares bit-level contracts (traits only, no storage).
- `arvo-bitmask` picks Mask64/Mask256 concretes on top of the
  contracts.

Sub-crates depending on the contracts (not the concretes) mean
consumers can swap storage widths without touching algorithm code.

## Sizes are const

Every size lives at type level: `Mask64`, `Mask256`, `Seq<T, N>`.
No runtime resize, no runtime grow. If a consumer needs a bigger
mask, they pick the bigger type.

## Zero platform dep

arvo has no opinion about syscalls, threads, clocks, or filesystems.
Everything is pure compute on const-sized containers. Platform
concerns live in consumer crates (notably `hilavitkutin`).
