# arvo — Design

**arvo is a fixed-point arithmetic stack.** Standalone `#![no_std]`,
no alloc, const-generic sizing. Same LLVM optimisation discipline
whatever runs on top. Cross-project: used by hilavitkutin,
polka-dots, saalis, and loimu. The exclusive numeric substrate
for the clause-dev workspace.

Every numeric primitive that crosses any arvo API (parameter,
return, struct field, associated type) flows through
`UFixed<I, F, S>`, `IFixed<I, F, S>`, `FastFloat`, or `StrictFloat`.
Bare Rust primitives are not valid in this universe; the
`arvo-types-only` lint (#50) enforces their absence. See
`PRINCIPLES.md` for the full statement.

Name: arvo (Finnish: "value/worth/estimate").

## Primitive vocabulary

This section enumerates the concrete shape of the identity stated
at the top of this file; the `arvo-types-only` lint (#50) is the
operational enforcement.

arvo follows the clause-dev primitive-vocabulary discipline. Public
API positions that would carry `Option<T>` / `Result<T, E>` on
arvo's own traits carry `notko::Maybe<T>` / `notko::Outcome<T, E>`
instead. Concretely: `UNarrowFrom::u_try_narrow` and
`INarrowFrom::i_try_narrow` return `Outcome<Self::T, ()>`.

`impl TryFrom<UFixed<...>> for UFixed<...>` (and the signed
counterpart) keeps the std `Result<Self, Self::Error>` signature
because `TryFrom` is a std trait — the designated escape hatch. Those
impls adapt at the boundary via a one-line match on the `Outcome`
they receive from `u_try_narrow` / `i_try_narrow`. Same pattern for
`Iterator::next -> Option<T>` (iterator impls stay on std).

`&'static str` is debug-gated: `Strategy::NAME` carries
`#[cfg(debug_assertions)]` so release builds drop the strings
entirely. Strategy identity at runtime flows through `RANK: u8`;
`NAME` is for diagnostics and tests only. Three const-generic
accessors in `strategy.rs` (`ibits_u8`, `fbits_u8`,
`is_fractional`) retain raw `u8` / `usize` returns because they exist
solely to let `IBits` / `FBits` values appear inside anonymous
const-generic expressions where direct field access is disallowed on
the current nightly. Each carries an `allow-bare-numeric: tracked:
#71` annotation; a BACKLOG note captures the eventual collapse into
a `Bits` newtype accessor once nightly loosens const-expression
rules.

## Primitive newtypes

arvo wraps Rust primitives used in public APIs. All are
`repr(transparent)` — zero overhead after compilation.

### Const generic bit counts

```rust
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct IBits(pub Bits<7, Hot>);  // integer bit count

#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct FBits(pub Bits<7, Hot>);  // fractional bit count
```

`IBits` and `FBits` wrap `Bits<7, Hot>` (the L0 storage primitive at
7 bits, sufficient for `0..=127`), not bare `u8`. The single
bottom-out point for `u8` const-generics is `Bits<const N: u8, S>`
itself; meta-layer types (`IBits`, `FBits`, `Width`) lift to typed
fields above it. The `from_raw_u8` constructor on `Bits<7, Hot>`
masks input to 7 bits, ensuring two `IBits` values constructed from
`135` and `7` compare structurally equal under `ConstParamTy`
(critical for soundness on the meta layer).

Requires `adt_const_params` (nightly). Prevents accidentally
swapping integer and fractional bit counts at type definition sites.
Consumer code defines its own semantic aliases over `UFixed` /
`IFixed` and rarely touches `IBits` / `FBits` directly.

### Width

```rust
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct Width(pub Bits<7, Hot>);  // bit-width meta value (1..=128)
```

The bit-width meta-newtype. Used as the const-generic param type for
`Hasher<const N: Width>` (in arvo-hash), `Fnv1a<const N: Width>`,
and any other API that takes a logical bit-width as a typed const
value. The L0 storage primitive `Bits` itself keeps its `const N:
u8` parameter as the irreducible root of the const-generic tower
(workspace-rule exception #2); `Width` lifts above that root.

The name avoids collision with the `arvo_bits::HasBitWidth` trait
(L1 contract layer; "this type has a bit width").

### USize

```rust
#[cfg(target_pointer_width = "64")]
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct USize(pub Bits<64, Hot>);

#[cfg(target_pointer_width = "32")]
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct USize(pub Bits<32, Hot>);

impl Deref for USize {
    type Target = usize;
    fn deref(&self) -> &usize {
        // SAFETY: Bits is repr(transparent); cfg-gated width matches
        // target_pointer_width; the underlying container is layout-
        // equivalent to usize.
        unsafe { &*(&self.0 as *const Bits<_, Hot> as *const usize) }
    }
}
```

Index and count newtype. The field type is `Bits<{cfg-platform-width},
Hot>` (the L0 storage primitive at the platform pointer width); the
underlying transparent-repr lowers to `usize`. `Deref<Target = usize>`
is preserved via documented `unsafe` transmute so `array[*idx]`
ergonomics work unchanged across every consumer call site. The
unsafe boundary is workspace-rule-exception #4 (ergonomic helper
returning a bare primitive); soundness rests on `Bits`'s
`repr(transparent)` plus the cfg gating that matches container width
to platform pointer width.

Also derives `ConstParamTy` so `USize` can appear inside other
const-generic newtypes (see `Cap` below). See DEEPDIVE_newtype-deref
for the Deref rationale.

### Cap

```rust
#[derive(ConstParamTy, PartialEq, Eq)]
pub struct Cap(pub USize);
```

Const-generic capacity for fixed-size structures. Used wherever a
const generic sizes an array: `BitMatrix<const N: Cap>`,
`Csr<const ROWS: Cap, const NNZ: Cap>`, power iteration matrices.
The const parameter name carries the semantic distinction (N, ROWS,
NNZ); the type prevents mixing capacities with other integers.

### Bool

```rust
#[repr(transparent)]
pub struct Bool(bool);

impl Deref for Bool {
    type Target = bool;
    fn deref(&self) -> &bool { &self.0 }
}

impl Try for Bool {
    type Output = bool;
    type Residual = Infallible;

    fn branch(self) -> ControlFlow<Infallible, bool> {
        ControlFlow::Continue(self.0)
    }
    fn from_output(output: bool) -> Self { Bool(output) }
}

pub trait AsBool {
    fn as_bool(&self) -> bool;
}
impl AsBool for Bool {
    fn as_bool(&self) -> bool { self.0 }
}
```

Control flow boolean. Returned by predicates (`is_zero`, `bit`).
NOT a fixed-point type — `Bool` is for branching, `Bit` (defined
in arvo-bits) is for 1-bit data storage.

Three paths to extract the inner `bool`:

| Path | Syntax | Where it works |
|---|---|---|
| `?` (Try) | `if val.is_zero()? { ... }` | Functions returning Outcome/Just/Maybe/Result |
| `*` (Deref) | `if *val.is_zero() { ... }` | Everywhere |
| `.as_bool()` | `if val.is_zero().as_bool() { ... }` | Everywhere, explicit compatibility bridge |

In WU code: use `?` (comes free via `Outcome` return type).
In general code: use `*`. For library boundary compatibility:
`.as_bool()`. See DEEPDIVE_newtype-deref for full rationale on the
Deref choice.

## Bits — the storage primitive

`Bits<const N: u8, S: Strategy = Hot>` is the L0 storage primitive.
It moved to arvo from arvo-bits in round 202604500000 because every
meta-layer ConstParamTy newtype (`IBits`, `FBits`, `Width`) wraps
`Bits<7, Hot>`, and `UFixed` / `IFixed` use `Bits<{ufixed_bits(I, F)},
S>` for transparent storage. Hosting `Bits` in arvo-bits would
create a circular dependency with arvo's own primitive layer.

`Bits` is opaque to arithmetic by deliberate design (no `Add` /
`Sub` / `Mul` / `Div` / `Ord`). It is the storage carrier;
arithmetic types (`UFixed`, `IFixed`) wrap it and impl arithmetic at
their layer. Two `Bits<N>` values compare via `PartialEq` / `Hash`;
there is no which-is-bigger question on a bit pattern.

```rust
#[derive(ConstParamTy, PartialEq, Eq, Copy, Clone, Debug)]
#[repr(transparent)]
pub struct Bits<const N: u8, S: Strategy = Hot>(<S as UContainerFor<N>>::T)
where
    S: UContainerFor<N>;
```

The `const N: u8` parameter is the workspace-rule exception #2 root:
the single bottom-out point where a const-generic position takes a
bare primitive. Every higher-layer const-generic (`Hasher<const N:
Width>`, `Fnv1a<const N: Width>`) lifts above it. Compile-time
asserted: `N` must be in `1..=64`.

### Constructor surface

```rust
impl<const N: u8, S: Strategy> Bits<N, S>
where Hot: UContainerFor<N>, S: UContainerFor<N>
{
    /// Associated-type-typed constructor.
    pub const fn from_raw(raw: <S as UContainerFor<N>>::T) -> Self;

    /// Universal masking entry (Hot-container width). Masks input to N bits.
    pub const fn from_raw_u64(raw: u64) -> Self;

    /// Width-class siblings. Each masks within its width class.
    pub const fn from_raw_u32(raw: u32) -> Self;
    pub const fn from_raw_u16(raw: u16) -> Self;
    pub const fn from_raw_u8(raw: u8) -> Self;

    /// Existing accessor.
    pub const fn to_raw(self) -> <S as UContainerFor<N>>::T;

    /// Symmetric meta-layer accessor (reads `Width.0.0` etc.).
    pub const fn to_raw_u8(self) -> u8;
}
```

`from_raw_uN` masks to N bits before storing. This is critical for
`ConstParamTy` soundness: `Bits<7, Hot>::from_raw_u8(255)` masks to
`255 & 0b0111_1111 = 127`. Two `Width` values constructed via
`Width(Bits::<7, Hot>::from_raw_u8(7))` and `Width(Bits::<7,
Hot>::from_raw_u8(135))` (which masks to 7) compare structurally
equal at the type level. Without masking they would be unequal.

`Bits::new(u64)` and `Bits::bits(self) -> u64` are deprecated. They

### Trait surface

Implements `arvo_bits::HasBitWidth` (reports `N`), `BitAccess`
(per-bit read / write), `BitSequence` (scan / popcount), and
`BitLogic` (whole-word bitwise) via blanket impls in arvo-bits. The
Strategy parameter defaults to `Hot` because bit-level work typically
wants the minimum container and wrapping semantics.

### Bit-size aliases

Default-Hot aliases at the canonical widths:

```rust
pub type Bit<S: Strategy = Hot>    = Bits<1,  S>;
pub type Nibble<S: Strategy = Hot> = Bits<4,  S>;
pub type Byte<S: Strategy = Hot>   = Bits<8,  S>;
pub type Word<S: Strategy = Hot>   = Bits<16, S>;
pub type DWord<S: Strategy = Hot>  = Bits<32, S>;
pub type QWord<S: Strategy = Hot>  = Bits<64, S>;
```

`Nibble` = 4-bit value (half-byte). `DWord` = 32 bits (double word).
`QWord` = 64 bits (quad word, the maximum logical width). x86
naming convention where `Word` = 16 bits. Consumers that want
arithmetic on an N-bit value reach for `UFixed<{ ibits(N) }, {
fbits(0) }, S>`. Consumers that want opaque identity (bit logic,
scan, popcount, equality, hash) use `Bits<N>` or one of its named
aliases. Storage footprint is identical; the wrapper is transparent.

### Bitfield — packed layouts with named sub-ranges

Where `Bits<N>` is "N bits, no structure," the `bitfield!` macro
generates a `#[repr(transparent)]` struct over the same
container-dispatched storage, with named sub-ranges and per-field
accessor methods. Use it for packed-layout identities where the
consumer cares about semantic sub-ranges (flag bits, reserved
blocks, ID ranges).

Example:

```rust
use arvo::bitfield;

bitfield! {
    /// Interned string handle. 32 bits total.
    pub struct StrHandle: 32 {
        /// 1 = runtime-interned, 0 = compile-time-interned.
        origin: 1 at 31,
        /// Reserved for future use.
        reserved: 3 at 28,
        /// 28-bit interned identity.
        id: 28 at 0,
    }
}
```

Generates a `StrHandle` that is `#[repr(transparent)]` over
`Bits<32, Hot>` (= 4 bytes). Inherent methods:

```rust
impl StrHandle {
    pub const BITS: u8 = 32;
    pub const fn new() -> Self;
    pub const fn from_bits(raw: Bits<32, Hot>) -> Self;
    pub const fn to_bits(self) -> Bits<32, Hot>;

    // 1-bit sub-range → Bit<Hot> = Bits<1, Hot>
    pub const fn origin(self) -> Bit<Hot>;
    pub const fn with_origin(self, value: Bit<Hot>) -> Self;
    // 3-bit sub-range → Bits<3, Hot>
    pub const fn reserved(self) -> Bits<3, Hot>;
    pub const fn with_reserved(self, value: Bits<3, Hot>) -> Self;
    // 28-bit sub-range → Bits<28, Hot>
    pub const fn id(self) -> Bits<28, Hot>;
    pub const fn with_id(self, value: Bits<28, Hot>) -> Self;
}
```

Accessor types are `Bits<W, Hot>` for the sub-range width: the
exact-width arvo primitive, never a raw `u8` / `u16` / `u32`. Common
widths match the named aliases (`Bit<Hot>` = 1, `Nibble<Hot>` = 4,
`Byte<Hot>` = 8, `Word<Hot>` = 16, `DWord<Hot>` = 32, `QWord<Hot>`
= 64). Setters mask to the sub-range width before writing, so
garbage high-bits on callers never corrupt neighbouring fields.

The macro also generates per-sub-range pre-shifted
`<FIELD>_MASK: Bits<N, Hot>` constants for boundary bit-arithmetic
where consumers hold raw values before wrapping them into the
bitfield.

Primary consumer: arvo-hash's `ContentHash` alias (`Bits<28, Hot>`,
u32-backed, 4 bytes). hilavitkutin-str's `Str` handle uses `Bitfield`
for its 32-bit packed identity layout.

```rust
/// Construct an `IBits` value from a `u8` literal.
pub const fn ibits(n: u8) -> IBits {
    IBits(Bits::<7, Hot>::from_raw_u8(n))
}

/// Construct an `FBits` value from a `u8` literal.
pub const fn fbits(n: u8) -> FBits {
    FBits(Bits::<7, Hot>::from_raw_u8(n))
}

/// Construct a `Width` value from a `u8` literal.
pub const fn width(n: u8) -> Width {
    Width(Bits::<7, Hot>::from_raw_u8(n))
}

/// Read the inner u8 from an `IBits`. Anonymous-const-context helper.
pub const fn ibits_u8(i: IBits) -> u8 { i.0.to_raw_u8() }

/// Read the inner u8 from an `FBits`. Anonymous-const-context helper.
pub const fn fbits_u8(f: FBits) -> u8 { f.0.to_raw_u8() }

/// Read the inner u8 from a `Width`. Anonymous-const-context helper.
pub const fn width_u8(w: Width) -> u8 { w.0.to_raw_u8() }

/// Whether a `Width` value is `<= 64`. Used by Fnv1a's const-eval guard.
pub const fn width_le_64(n: Width) -> bool { width_u8(n) <= 64 }
```

The `u8` parameters / returns on these helpers are workspace-rule
exception #4 (ergonomic helper-fn parameters that construct arvo
types from literals; ergonomic accessors that return a bare primitive
into anonymous-const context where direct field access is disallowed
on current nightly). The helpers' bodies wrap / unwrap exactly once.

`width(n)` is named per existing convention (matches `ibits` /
`fbits`, no `_of` suffix).

## Type family marker traits

Five marker traits classify arvo's type families. Used by arvo-bits
for blanket impls and by consumers for generic constraints.

```rust
pub trait IntegerLike: Copy {}
pub trait FractionLike: Copy {}
pub trait BitPresentation: Copy {
    const LOGICAL_WIDTH: USize;
}
pub trait FloatLike: Copy {}
pub trait BoolLike: Copy {
    fn pack(self) -> Bit;
}
```

| Marker | Types | Semantic domain |
|---|---|---|
| `IntegerLike` | UFixed with F=0, IFixed with F=0 | whole numbers |
| `FractionLike` | UFixed with F>0, IFixed with F>0 | fractional values |
| `BitPresentation` | all UFixed, all IFixed | raw bits accessible |
| `FloatLike` | FastFloat, StrictFloat | IEEE float wrappers |
| `BoolLike` | Bool | truth values, packable to Bit |

Blanket impls:

```rust
impl<const I: IBits, S: Strategy> IntegerLike
    for UFixed<I, {FBits(0)}, S> {}

impl<const I: IBits, S: Strategy> IntegerLike
    for IFixed<I, {FBits(0)}, S> {}

impl<const I: IBits, const F: FBits, S: Strategy> BitPresentation
    for UFixed<I, F, S>
{
    const LOGICAL_WIDTH: USize = USize(I.0 as usize + F.0 as usize);
}

impl<const I: IBits, const F: FBits, S: Strategy> BitPresentation
    for IFixed<I, F, S>
{
    const LOGICAL_WIDTH: USize = USize(1 + I.0 as usize + F.0 as usize);
}

// FractionLike: UFixed/IFixed where F > FBits(0)
// FloatLike: FastFloat, StrictFloat
// BoolLike: Bool
```

`BoolLike::pack()` converts a truth value to a 1-bit data value
(`Bit` from arvo-bits). The reverse direction is `Bit::unpack() ->
Bool`. This formalises the boundary between control flow (Bool) and
data storage (Bit) as a trait contract.

`FractionLike` blanket impl requires a `where F > FBits(0)` const
generic constraint. Exact mechanism depends on `generic_const_exprs`
stability; design intent is clear, src CL resolves the mechanism.

## Fixed-point types

```rust
pub struct UFixed<const I: IBits, const F: FBits, S: Strategy> {
    bits: S::Container<I, F>,
}

pub struct IFixed<const I: IBits, const F: FBits, S: Strategy> {
    bits: S::SignedContainer<I, F>,
}
```

`I` = integer bits, `F` = fractional bits. `S` = strategy marker
(no struct-level default; the strategy default lives at the alias
layer, e.g. `pub type Uint8<S = Warm> = UFixed<{ ibits(8) }, {
fbits(0) }, S>`). `I.0 + F.0` is the logical bit width. The actual
storage width depends on the strategy.

The struct-level `S = Warm` default was dropped in round
202604500000 (foundations restructure). Two sources of Warm-default
(struct-level + alias-level) created drift risk; the alias layer is
now the single source.

Both `repr(transparent)` over the strategy-determined container type.
No runtime representation overhead. After compilation, every arvo
type resolves to the optimal integer instruction for the target.

## Strategy markers

The strategy is a type parameter that controls container width,
arithmetic semantics, bitpacking, cross-width operation rules, and
SIMD density. Four strategies:

### Hot

Optimised for L1 density and operation throughput.

**Container:** minimum byte-aligned standard width that fits `I + F`
bits. A 5-bit value stores in u8. A 12-bit value stores in u16.
No bitpacking — byte-aligned always.

**Arithmetic:** wrapping. Single instruction per op. May overflow
silently. LLVM vectorises freely. Div-by-zero returns the numerator
unchanged (wrapping has no identity for zero; silent garbage is
preferable to panic under the single-instruction contract).

**Cross-width ops:** shift-narrow the wider value to match the
narrower type's layout. Right-shift preserves integer bits, loses
fractional precision. Operate at the narrower width.

**SIMD:** most lanes per register. u8 in 128-bit NEON = 16 lanes.

**Use case:** inner morsel loops where range is known safe. Counters,
indices, flags, enum-like values.

### Warm (default)

Store big, operate fast. The development-friendly default.

**Bound:** `I + F ≤ 32`. Warm's value proposition is the 2x container
safety guarantee on single ops; no u128 is available (Q3), so widths
beyond 32 logical bits cannot be serviced without losing that
guarantee. Warm is a compile-time error at `I + F > 32` — pick Hot
(wrapping, accepts overflow risk) or Precise (saturating, carries
correctness) instead. This surfaces the trade-off at the type level
rather than hiding it.

**Container:** 2x the logical bit width, rounded to next standard
width. A 16-bit logical value stores in u32; a 32-bit logical value
stores in u64.

**Arithmetic:** wrapping. Single instruction. The 2x container means
a single add/sub/mul of two values within their logical range cannot
overflow the container. Chains of operations (accumulating many
values) can eventually overflow.

**Cross-width ops:** widen both operands to 2x the wider operand's
container width. Operate at that width. Single ops safe, chains may
overflow.

**Bitpacking:** no. Byte-aligned, rounded up.

**SIMD:** fewer lanes than Hot (wider containers), but still wrapping
ops so LLVM vectorises freely.

**Use case:** the default for most values up to 32 logical bits.
Correctness on single ops without saturating overhead.

### Cold

Store small, operate carefully. Optimised for storage density.

**Container:** minimum container that fits `I + F` bits. Same as
Hot for byte-multiples. For sub-byte values: bitpacked (5-bit value
stores as 5 bits, not u8).

**Arithmetic:** widen to 2x before operating, narrow back after.
Every operation has load-widen-op-narrow-store overhead.

**Cross-width ops:** widen both to the wider operand's container
width. Operate at that width. Narrow back on store.

**Bitpacking:** yes. Stores exactly the logical bits. Pays
shift/mask cost on every access. Acceptable because Cold values
ride L2/L3, not L1.

**SIMD:** narrow storage but ops widen, so effective lane count
matches the widened width.

**Use case:** columns stored in bulk but processed rarely. Archival
data, audit trails, metadata columns, lookup tables.

### Precise

Store exactly, operate exactly. Correctness above all.

**Container:** 2x the logical width (same as Warm).

**Arithmetic:** saturating. Overflow clamps to max/min rather than
wrapping. Saturating add/sub have SIMD intrinsics on most platforms.
Multiply uses the widened container so the intermediate result is
exact, then saturate on store if the result exceeds the logical range.

**Cross-width ops:** widen both to 2x the wider operand's container.
Operate. Saturate on store. No precision loss at any point.

**Bitpacking:** yes. Stores exactly the logical bits. The
representation IS the precision.

**SIMD:** fewest lanes (widest containers + saturating ops). But
saturating add/sub are SIMD-supported.

**Use case:** financial calculations, audit-critical values, values
that flow into user-visible reports where silent data loss is
unacceptable.

## Container width table

| I + F | Hot | Warm | Cold | Precise |
|---|---|---|---|---|
| 1-8 | u8 | u16 | bitpacked | u16 |
| 9-16 | u16 | u32 | bitpacked | u32 |
| 17-32 | u32 | u64 | bitpacked | u64 |
| 33-64 | u64 | *unavailable* | bitpacked | u64 (saturating) |

No u128 containers. I + F <= 64 for UFixed, 1 + I + F <= 64 for
IFixed. Warm is a compile-time error at I + F > 32 (no u128 container
means Warm cannot offer its 2x safety; rather than degrade silently,
Warm is forbidden at those widths — pick Hot or Precise).

IFixed follows the same widening rules with signed containers (i8,
i16, i32, i64). Signed saturation clamps to the logical range
(i_MIN..i_MAX), not the container range.

## Strategy comparison

| Aspect | Hot | Warm (default) | Cold | Precise |
|---|---|---|---|---|
| Container | min aligned | 2x logical | min (bitpacked) | 2x logical |
| Bitpacking | no | no | yes | yes |
| Add/Sub | wrapping | wrapping (safe 1-op) | widen-op-narrow | saturating |
| Multiply | wrapping | wrapping (safe 1-op) | widen-op-narrow | widening + saturate |
| Div-by-zero | numerator | numerator | numerator | max |
| Chain safety | overflows | overflows on long chains | overflows on long chains | clamps always |
| Cross-width | shift-narrow wider | widen to 2x wider | widen to wider | widen to 2x wider |
| SIMD lanes | most | moderate | most (stored) / moderate (ops) | fewest |
| L1 density | best | 2x worse | best | 2x worse |
| Access cost | byte load | byte load | shift/mask | shift/mask |
| Precision loss | yes (wrapping + narrow) | no (single ops) | no (single ops) | never |

## Column layout derivation

The column layout mode (ByteAligned / BitPacked) is derived from
the strategy, not set independently:

| Strategy | Layout mode |
|---|---|
| Hot | ByteAligned (minimum aligned container) |
| Warm | ByteAligned (2x container, no packing) |
| Cold | BitPacked (exact logical bits) |
| Precise | BitPacked (exact logical bits) |

The strategy marker IS the layout decision.

## Cross-width operation detail

When `UFixed<I1, F1, S1>` operates with `UFixed<I2, F2, S2>`:

1. **Strategy resolution:** if S1 != S2, the result strategy is the
   more conservative: Precise > Cold > Warm > Hot. The less
   conservative operand is converted before the operation.

2. **Width alignment:** fractional bits are aligned by shifting the
   value with fewer F bits left. Integer bits are aligned by widening.

3. **Result type:** `I = max(I1, I2)`, `F = max(F1, F2)`. Strategy
   is the resolved strategy from step 1.

## UContainerFor / IContainerFor

`UContainerFor<const BITS: u8>` (and the signed sibling
`IContainerFor`) is the sealed dispatch trait that maps `(strategy,
bit-count)` to the concrete container type. Each strategy marker
implements `UContainerFor<N>` for the bit-counts it supports:

- `Hot`: `1..=64` (full range).
- `Warm`: `1..=32` (Warm cap; above 32 is unimplemented and surfaces
  a compile-time error).
- `Cold`: `1..=64`.
- `Precise`: `1..=64`.

The `BITS: u8` parameter is workspace-rule exception #2 (storage-
primitive const-generic at the L0 root); meta-layer types (`Width`,
`IBits`, `FBits`) lift above this root.

The trait carries `#[diagnostic::on_unimplemented]` so the
unimplemented-bound error points the consumer at strategy choice in
human-readable terms instead of an obscure trait-bound failure:

```rust
#[diagnostic::on_unimplemented(
    message = "strategy `{Self}` does not provide a container for {BITS}-bit width",
    note = "Warm caps at I+F<=32; for wider widths, choose Hot or Precise explicitly: `Uint64<Hot>`, `Uint64<Precise>`, etc.",
)]
pub trait UContainerFor<const BITS: u8>: Strategy { type T: ...; }
```

The same diagnostic attribute applies to `IContainerFor`. The Warm-
default-Uint64 case (alias `Uint64<S = Warm>` with no explicit S)
fires `UContainerFor<64> not implemented for Warm`; the diagnostic
message names the strategy and the path to a fix.

### ConstParamTy on strategy markers

`Hot`, `Warm`, `Cold`, `Precise` each derive `ConstParamTy` so they
can appear in const-generic position when needed (e.g. inside the
const-generic param of a meta-newtype that encodes both bit-width
and strategy). The markers are unit structs; the derive is mechanical
and follows arvo's existing tuple-struct derive precedent.

## Conversion between strategies

Converting between strategies for the same `<I, F>`:

- **Hot -> Warm**: zero-extend into wider container. Free.
- **Warm -> Hot**: truncate to minimum container. Wraps if value
  exceeds container.
- **Hot/Warm -> Cold/Precise**: bitpack from byte-aligned to exact
  bits. Shift/mask cost.
- **Cold/Precise -> Hot/Warm**: unpack from bitpacked to byte-aligned.
  Shift/mask cost.

Converting between different `<I, F>` configurations is a computation:
shift fractional bits, clamp or wrap integer bits. The strategy
determines the clamping/wrapping semantics.

## Float wrappers

`FastFloat` — wraps f32/f64 with LLVM fast-math intrinsics enabled.
Trades IEEE 754 compliance for speed (reassociation, reciprocal
approximation, no NaN propagation). MathPeephole LLVM pass fires on
FastFloat operations via hilavitkutin-build.

`StrictFloat` — wraps f32/f64 with strict IEEE 754 semantics. No
fast-math. For validation, comparison, and contexts where bit-exact
reproducibility matters.

`StrictFloat::sqrt` is currently a 7-ULP Newton-Raphson approximation
in no_std (the substrate has no libm dependency). The bit-exact
contract still stands in design intent; the approximation is a
substrate limitation tracked in BACKLOG. Consumer code that needs
correctly-rounded sqrt should either link libm at its own layer or
wait for the substrate BACKLOG item to ship.

`Float` — type alias. Resolves to `FastFloat` when `cfg(arvo_fast_math)`
is active (the FastMath pragma from hilavitkutin-build), `StrictFloat`
otherwise.

Float types are actual f32/f64 wrappers. They are NOT fixed-point.
They do not have strategy markers. Completely separate type family.
LLVM fast-math flags apply to float operations only; fixed-point uses
integer arithmetic.

## Composable traits

No monolithic `Numeric` trait. Small single-method traits composed
via bounds:

- `TotalOrd` — total ordering (NaN-aware for floats)
- `Sqrt` — square root (hardware or approximation via cfg)
- `Recip` — reciprocal (hardware or Newton-Raphson via cfg)
- `Abs` — absolute value
- `FromConstant` — `fn from_constant(n: USize) -> Self`. Compile-time
  literal conversion; input is `USize` (never a bare primitive).
  Float impls bridge internally via a private `FromU8Ieee` helper
  for the lossless 0..=255 range. For fixed-point types, `n` is
  placed at the integer-bit position (shifted left by `F`).
- `Pred<A>` / `Pred2<A, B>` / `Pred3<A, B, C>` — named predicate
  bounds for consumer callbacks. Blanket-impl'd for
  `Fn(&A) -> Bool` / `Fn(&A, &B) -> Bool` /
  `Fn(&A, &B, &C) -> Bool`; custom callable types implement
  `test(&self, ...)`.

Consumer writes `T: Add + Mul + Sqrt` not `T: Numeric`. Only the
needed operations appear in bounds. The mathematical sub-crates
(arvo-graph, arvo-sparse, arvo-spectral, arvo-comb) are generic
over these trait bounds, not concrete arvo types.

## L0 non-members

arvo ships raw `UFixed<I, F, S>` / `IFixed<I, F, S>` and the four
strategy markers. It ships no semantic type aliases and no
platform/layout newtypes.

**No semantic domain aliases.** `Ratio`, `Factor`, `Norm`, `Coord`,
`Angle`, `Count`, `Average`, `Deviation`, and similar are consumer
concerns, not substrate. Each downstream crate chooses the `<I, F>`
configuration that fits its domain and defines the alias it wants.

Example (downstream, not here):
```rust
// in some consumer crate
pub type Angle<S = Warm> = arvo::UFixed<{IBits(9)}, {FBits(7)}, S>;
pub type Coord<S = Warm> = arvo::IFixed<{IBits(16)}, {FBits(16)}, S>;
```

Strategy propagates through the type system: cross-strategy
conversion rules apply at boundaries. Generic code over strategy
remains idiomatic:
```rust
fn process<S: Strategy>(
    a: UFixed<{IBits(8)}, {FBits(0)}, S>,
    b: UFixed<{IBits(0)}, {FBits(8)}, S>,
) -> UFixed<{IBits(8)}, {FBits(8)}, S> { ... }
```

**Deferred (not in arvo L0):**
- `ByteCount`, `Alignment` — layout / physical-storage concerns. These
  belong at the storage boundary (ColumnValue layer in the engine
  that consumes arvo), not in L0 substrate.
- `CoreCount`, `Nanoseconds` — platform-environment concerns. arvo
  has zero platform dep. These belong adjacent to `ThreadPool` /
  `Clock` trait definitions in the consuming layer.

The decision is about layer fit, not necessity. The types remain
designed; they land when the consuming layer first needs them.

**`Bool` stays in arvo.** `Bool` is a control-flow newtype over
`bool`, not a fixed-point alias — it has no strategy marker and is
used by predicate methods (`is_zero`, `bit`). The 1-bit data type
`Bit` for column storage lives in arvo-bits.

## Hardware-aware backend selection

arvo detects hardware capabilities in its own build.rs via
`CARGO_CFG_TARGET_FEATURE` (the standard cargo env var, not custom
flags from hilavitkutin-build). If `target_feature` includes `sqrt`,
arvo uses the hardware sqrt intrinsic; otherwise software
approximation.

The only cfg arvo receives from hilavitkutin-build is `arvo_fast_math`
(when the FastMath pragma is active), which gates fast-math float
semantics in arvo's float wrappers.

The consumer never sees any of this. The API is identical regardless
of backend.

## arvo-types-only lint

The `arvo-types-only` lint (within this repo) requires arvo types in
data-facing struct fields across all arvo sub-crates. Raw `u32`,
`f64`, etc. are rejected in favour of semantic arvo types. After
compilation, arvo types are `repr(transparent)` — zero overhead.

## Concrete vs generic

The algorithm sub-crates (arvo-graph, arvo-sparse, arvo-comb,
arvo-spectral) are generic over composable numeric trait bounds and
do not import `UFixed` / `IFixed` directly. Concrete type choice is
the consuming application's responsibility — not arvo's.

## Bit-width reporting

Any `ColumnValue`-style consumer layer that needs to know the logical
bit width reads `LOGICAL_WIDTH` via the `BitPresentation` marker
(`I.0 + F.0` for UFixed; `1 + I.0 + F.0` for IFixed). Physical
storage width depends on the strategy:
- BitPacked strategies (Cold, Precise): physical = LOGICAL_WIDTH
- ByteAligned strategies (Hot, Warm): physical = container size in bits
