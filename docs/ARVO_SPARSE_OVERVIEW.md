# arvo-sparse — Design

Sparse matrix representations and algorithms on const-generic
fixed-size structures. `#![no_std]`, no alloc. Operates on
arvo-bitmask types for small matrices and CSR for larger ones.

## Capacity

Ships against `BitMatrix64` / `Mask64` (up to 64 rows or columns of
adjacency). `BitMatrix256` / `Mask256` sibling algorithms are tracked
in `BACKLOG.md.tmpl`; the algorithmic shape is identical, only the
bitmask backing changes. `Csr<ROWS, NNZ, W>` is keyed on arvo `Cap`
const generics and has no arvo-bitmask dependency, so it scales to
any ROWS/NNZ the caller picks.

`const N: Cap` / `const ROWS: Cap` / `const NNZ: Cap` carry arvo's
capacity newtype on every public signature. Array storage unwraps
via `cap_size(c: Cap) -> usize`, re-exported from arvo-bitmask.
Array return types in signatures below are shown as `[X; N]` for
readability; the compiled form is `[X; cap_size(N)]` with the
implied `where [(); cap_size(N)]:` bound.

Generic over numeric trait bounds for matrix values. Does not import
UFixed or IFixed directly. CSR is generic over the value type.
Consumer instantiates with concrete arvo types; this crate is
library code.

Note: "row" and "column" in this document refer to matrix
rows/columns, not data records or data-storage columns.

## CSR

```rust
pub struct Csr<const ROWS: Cap, const NNZ: Cap, W: Copy> {
    row_ptr: [USize; ROWS],
    col_idx: [NodeId; NNZ],
    values: [W; NNZ],
}
```

Compressed sparse row. Const-generic fixed-size arrays — no heap.
`ROWS` and `NNZ` are arvo's `Cap` const-generic capacity newtype.
Row pointers and column indices use `NodeId` from arvo-bitmask.

Generic over the value type `W`. No numeric trait needed on W for
the structure itself — only for algorithms that compute on the
values.

## Sparse matrix representations

**Bitmask matrix** (<=64 rows/cols): `[Mask64; N]` from arvo-bitmask.
One-instruction operations. Used when the access matrix fits in
64 nodes (typical for small-to-medium DAGs).

**CSR** (>64 rows/cols): const-generic fixed-size arrays as above.
For >256 nodes where bitmask operations would require multi-word
masks.

The API is uniform: algorithms accept a trait that both
representations implement. Const-generic sizing means no heap
allocation regardless of representation.

## RCM reordering

Reverse Cuthill-McKee bandwidth minimization. O(n + nnz).

```rust
fn rcm_reorder<const N: Cap>(
    adjacency: &BitMatrix64<N>,
) -> [NodeId; N]  // permutation: new position -> old NodeId
```

Common applications:

**Row reordering (execution order):** reorders matrix rows so that
operations accessing similar column sets become adjacent. Better
grouping of subsequent batched work than naive ordering; most
impactful for wide DAGs with fan-out.

**Column reordering (memory layout):** reorders matrix columns so
that co-accessed entries have adjacent offsets. The hardware
prefetcher benefits from spatial locality.

Design intent: run unconditionally during planning when the consumer
is a pipeline scheduler (polka-dots T3 decision #67: "RCM always").
Sub-microsecond at typical pipeline scale.

## Block diagonal detection

```rust
fn block_diagonal<const N: Cap>(
    adjacency: &BitMatrix64<N>,
) -> (USize, [USize; N])  // (block_count, block_id_per_node)
```

Permute the access matrix to block diagonal form. Each block is an
independent sub-problem with no shared columns between blocks.

Build bipartite graph (WUs on one side, stores on other), find
connected components (delegates to arvo-graph or internal bitmask
DFS). Each component is a block. Permute matrix rows and columns
to group blocks along the diagonal.

Common application: align blocks with phases from waist detection
(arvo-graph). Also enables dead-column elimination via
Dulmage-Mendelsohn. Design intent for pipeline schedulers: run
unconditionally during planning (polka-dots T3 decision #68: "block
diagonal always").

## Dulmage-Mendelsohn decomposition

```rust
pub struct DulmageMendelsohn<const N: Cap> {
    pub horizontal: Mask64,  // dead-end rows (output never read)
    pub vertical: Mask64,    // read-only columns (never written)
    pub square: Mask64,      // core matched row-column pairs
}

fn dulmage_mendelsohn<const N: Cap>(
    adjacency: &BitMatrix64<N>,
) -> DulmageMendelsohn<N>
```

Structural decomposition of a bipartite graph into fine and coarse
components. Refines block diagonal detection:

- **Horizontal components:** rows whose outputs are never read by
  any other row. Dead-end — their outputs can be eliminated unless a
  final consumer reads them.
- **Vertical components:** columns that are only inputs (read-only
  throughout the computation).
- **Square components:** matched row-column pairs where every output
  is consumed. The core of the computation.

Dead-component elimination reduces the working set in the consumer's
downstream grouping, which can directly improve cache behaviour.
