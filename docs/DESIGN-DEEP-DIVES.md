# arvo — Deep Dive: Deref on Primitive Newtypes

## Context

arvo wraps several Rust primitives in newtypes to enforce semantic
meaning and satisfy the arvo-types-only lint:

- `Bool(bool)` — control flow boolean, returned by predicates
- `USize(Bits<{cfg-platform-width}, Hot>)` — index and count type
- `IBits(Bits<7, Hot>)` — integer bit count for const generics
- `FBits(Bits<7, Hot>)` — fractional bit count for const generics
- `Width(Bits<7, Hot>)` — bit-width meta value for const generics

These newtypes are `repr(transparent)`. After compilation they are
identical to their inner type's underlying container. They exist
purely for compile-time type safety: you cannot accidentally pass a
`USize` where a `Bool` is expected, and you cannot swap an `IBits`
with an `FBits` even though both wrap the same `Bits<7, Hot>` shape.

The four ConstParamTy newtypes (`USize`, `IBits`, `FBits`, `Width`)
moved to wrapping `Bits` over wrapping a bare primitive in round
202604500000. The `Bits` storage primitive itself bottoms out at a
`u8` const-generic (workspace-rule exception #2 root); the
meta-newtypes lift above that root.

The question is how to access the inner value when you need it.

## The problem

Rust's `if` requires `bool`. Not a type that wraps `bool`, not a
type that can be converted to `bool` — exactly `bool`. Similarly,
array indexing requires `usize`. These are hard-wired into the
language with no trait-based extension point.

If `is_zero()` returns `Bool` (our newtype), `if val.is_zero()`
does not compile. If `trailing_zeros()` returns `USize` (our
newtype), `array[val.trailing_zeros()]` does not compile.

Three options exist to bridge this gap.

## Option 1: Explicit conversion method

Define a trait or inherent method for extraction:

```rust
pub trait AsBool {
    fn as_bool(&self) -> bool;
}

impl AsBool for Bool {
    fn as_bool(&self) -> bool { self.0 }
}
```

Usage: `if val.is_zero().as_bool() { ... }`

Advantages: explicit, no magic, follows Rust `As*` naming convention.
Disadvantages: 10 extra characters at every call site. In hot loop
code operating on arvo types, predicates and index operations appear
frequently. The verbosity adds up and obscures the actual logic.

## Option 2: Try trait (? operator)

`Bool` implements `Try` with an uninhabited residual:

```rust
impl Try for Bool {
    type Output = bool;
    type Residual = Infallible;
    fn branch(self) -> ControlFlow<Infallible, bool> {
        ControlFlow::Continue(self.0)
    }
    fn from_output(output: bool) -> Self { Bool(output) }
}
```

Usage: `if val.is_zero()? { ... }`

The `?` always takes the `Continue` path (the `Break` path is
uninhabited and unreachable). Zero runtime cost. One character.

Limitation: the enclosing function must return a type that implements
`FromResidual<Infallible>`. In hilavitkutin, all WU execute methods
return `Outcome<T>` which satisfies this. Outside WU code, the
function's return type must be one of hilavitkutin's result family
(`Outcome`, `Just`, `Maybe`, `Result`) or the `?` approach is not
available.

This is the preferred path in WU code, where it comes for free.

## Option 3: Deref

`Bool` implements `Deref<Target = bool>`:

```rust
impl Deref for Bool {
    type Target = bool;
    fn deref(&self) -> &bool { &self.0 }
}
```

Usage: `if *val.is_zero() { ... }`

One character (`*`). Works everywhere, no return type requirements.

## Why Deref is unconventional here

The Rust API guidelines recommend `Deref` for smart pointer types
(`Box`, `Rc`, `Arc`, `MutexGuard`), not for newtypes. The reasoning
behind this convention stems from three concrete concerns:

### 1. Method resolution opacity

When you call `val.method()`, Rust's auto-deref chain tries methods
on the outer type first, then on `*val`, then on `**val`, etc. If
`Bool` has a method `then()` and `bool` also has `then()`, the
`Bool` version wins. If `Bool` does not have `then()`, `bool`'s
version is found via auto-deref. Reading the code, you cannot tell
which type's method is being called without checking both.

### 2. Reference coercion at call boundaries

Deref coercion means `&Bool` silently becomes `&bool` when passed
to a function expecting `&bool`. The type boundary evaporates at
reference sites without any visible marker in the code.

### 3. Trait bound inconsistency

`bool` may implement traits that `Bool` does not. Deref does not
make the outer type satisfy the inner type's trait bounds. But deref
coercion does make `&Bool` usable where `&bool` is expected. This
means the inner type's capabilities sometimes leak through and
sometimes do not, depending on whether the context uses values,
references, or trait bounds.

## Why these concerns are acceptable for arvo

### Method resolution is bounded

`bool` has a small, well-known API: `then()`, `then_some()`, and
the boolean operators. `usize` has more methods but they are all
standard numeric/conversion methods with no surprising behaviour.
There is no realistic scenario where auto-deref to the inner type
produces unexpected method dispatch. If arvo adds a method with the
same name as a method on the inner type, it intentionally shadows
it.

### Reference coercion is correct behaviour

These newtypes are `repr(transparent)` wrappers. At the machine
level, `&Bool` and `&bool` are the same pointer to the same byte.
The deref coercion makes the language-level behaviour match the
machine-level reality. A function accepting `&bool` receiving a
`&Bool` is not a type safety violation — it is the intended
transparency.

### Trait bound inconsistency is a learnable quirk

The inconsistency ("this works with `&Bool` but not with `Bool`
directly for some trait bounds") is real but not a safety hazard.
It manifests as a compile error, not a runtime bug. Developers
learn the pattern once: explicit `*` when you need the inner value,
auto-deref when working with references. This is the same pattern
Rust developers already use with `String`/`&str` via Deref.

### The alternative is worse

Without Deref, every predicate check and index operation requires
either `.as_bool()` / `.as_usize()` or the `?` operator (which
constrains the function's return type). In performance-critical
code with many predicate checks and array accesses, the explicit
conversion noise obscures the actual algorithm. The cure is worse
than the disease.

## The three-path design

arvo provides three bridges from newtypes to primitives, each
suited to different contexts:

| Path | Syntax | Where it works |
|---|---|---|
| `?` (Try) | `if val.is_zero()? { ... }` | Functions returning Outcome/Just/Maybe/Result |
| `*` (Deref) | `if *val.is_zero() { ... }` | Everywhere |
| `.as_bool()` (AsBool) | `if val.is_zero().as_bool() { ... }` | Everywhere, explicit, compatible |

In WU code (the hot path): use `?`. It is the most ergonomic and
comes for free because `execute()` returns `Outcome`.

In general arvo-consuming code: use `*`. One character, no return
type constraints.

For library public APIs that must expose Rust-native types for
compatibility with non-arvo consumers: use `.as_bool()` or
`.as_usize()` via the `AsBool` / `AsUSize` traits. This is the
explicit escape hatch at the boundary between arvo's type system
and external Rust code.

## Which newtypes get Deref

| Type | Deref target | Rationale |
|---|---|---|
| `Bool` | `bool` | Predicate results used in `if`/`while` |
| `USize` | `usize` | Indices used in array access |
| `IBits` | no | Compile-time const generic, no runtime extraction needed |
| `FBits` | no | Compile-time const generic, no runtime extraction needed |

`IBits` and `FBits` are used exclusively as const generic parameters.
Their values are never accessed at runtime. They do not need Deref.

## DerefMut

None of these types implement `DerefMut`. Mutation goes through
arvo's own APIs, not through the inner primitive. If you need to
increment a `USize`, use arvo's `Add` implementation, not `*val += 1`.
This preserves the type boundary for writes while relaxing it for
reads.

## Summary

Using `Deref` on newtypes is unconventional. The convention exists
to prevent confusion in large public APIs consumed by developers
who may not understand the auto-deref mechanism. arvo is an internal
library with expert users, transparent `repr(transparent)` wrappers,
and bounded inner-type APIs. The concrete downsides (method
resolution opacity, reference coercion, trait bound inconsistency)
are minor for this use case. The ergonomic gain (single `*` instead
of `.as_bool()` at every predicate site) is significant. The design
is a deliberate choice, not an oversight.
