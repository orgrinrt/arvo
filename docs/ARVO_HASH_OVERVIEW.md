# arvo-hash — Design

L2 hash-domain types and algorithm contracts. Ships a domain alias
over the arvo L0 storage primitive, generic trait contracts for
hash algorithms, and one concrete algorithm: `Fnv1a<const N: Width>`.

`#![no_std]`, no alloc. Depends on arvo (L0) and arvo-bits (L1).

## Why this crate exists

Hashes (content hashes, name hashes, pass-key identity) are numeric
algorithms on opaque bit-patterns. The stack needs one home for the
runtime carriers. arvo provides the generic `Bits` container; arvo-
hash provides the domain vocabulary (`ContentHash`), the algo
contracts (`Hasher`, `HasherExt`) consumers parameterise over, and
the canonical first algorithm (`Fnv1a`).

Growth surface (additional concrete algorithms, cryptographic hashes
as a sibling crate, semantic aliases beyond ContentHash) is tracked
in BACKLOG.

## Types

### ContentHash

A type alias for `arvo::Bits<28, Hot>`. Matches hilavitkutin-str's
low-28-bit FNV content-hash identity. Consumers that want a named
type at call sites (hilavitkutin-persistence StringTableEntry,
clause resolver pass-key identity) bind `ContentHash` instead of
the raw `Bits<28, Hot>`.

```rust
pub type ContentHash = arvo::Bits<28, Hot>;
```

A direct type alias rather than a newtype: the semantic domain is
fully captured by the bit width; no additional invariants warrant
a distinct type. The `Bits` const-generic stays `u8` (root exception
#2); only `Hasher` / `Fnv1a` lift to `Width`.

## Algorithm contracts

### Hasher (canonical, streaming)

```rust
use arvo::{Width, Bits, Hot};
use arvo::strategy::{UContainerFor, width_u8};

pub trait Hasher<const N: Width>
where Hot: UContainerFor<{ width_u8(N) }>
{
    fn update(&mut self, bytes: &[u8]);
    fn finalize(self) -> Bits<{ width_u8(N) }, Hot>;
}
```

Streaming hash: feed bytes incrementally, finalise to an N-bit
opaque pattern. The const-generic `N: Width` is the meta-newtype
encoding the bit width; the inner `Bits` is sized via `width_u8(N)`
into the L0 storage primitive's `u8` const-generic. Consumers
implementing a hash algorithm implement this trait.

The `&[u8]` parameter is workspace-rule exception #4 (boundary
input from raw bytes; canonical hash input shape).

### HasherExt (oneshot convenience, blanket impl)

```rust
pub trait HasherExt<const N: Width>: Hasher<N> + Sized
where Hot: UContainerFor<{ width_u8(N) }>
{
    /// One-shot: update with the complete input, then finalize.
    fn hash(mut self, bytes: &[u8]) -> Bits<{ width_u8(N) }, Hot> {
        self.update(bytes);
        self.finalize()
    }
}
impl<H, const N: Width> HasherExt<N> for H
where
    H: Hasher<N> + Sized,
    Hot: UContainerFor<{ width_u8(N) }>,
{}
```

`HasherExt`'s default method provides oneshot ergonomics over any
`Hasher` implementor. The blanket impl means every `Hasher`
automatically exposes a `.hash(bytes)` one-shot form; implementors
do not write their own oneshot path. Mirrors the round-202604201048
Encoder / Decoder + EncoderExt / DecoderExt pattern: streaming is
canonical, oneshot is a derived convenience.

## Concrete algorithm: Fnv1a

```rust
use arvo::{Width, Bits, Hot, width};
use arvo::strategy::{UContainerFor, width_u8, width_le_64};

/// Streaming FNV-1a-64 hasher with N-bit output.
///
/// `Fnv1a<{ width(N) }>` produces an N-bit hash. Bound: `N <= 64`.
/// For wider widths see the `Fnv1a128` BACKLOG entry.
pub struct Fnv1a<const N: Width>
where
    Hot: UContainerFor<{ width_u8(N) }>,
    [(); 1 / (width_le_64(N) as usize)]: ;

impl<const N: Width> Fnv1a<N>
where
    Hot: UContainerFor<{ width_u8(N) }>,
    [(); 1 / (width_le_64(N) as usize)]: ,
{
    /// Compile-time hash construction.
    ///
    /// Used by hilavitkutin app-builder for compile-time access-set
    /// hash-membership validation. Const-callable in const-generic
    /// position on the consumer side.
    pub const fn hash_const(bytes: &[u8]) -> Bits<{ width_u8(N) }, Hot> {
        let raw = fnv1a_64(bytes);
        Bits::<{ width_u8(N) }, Hot>::from_raw_u64(raw)
    }
}

impl<const N: Width> Hasher<N> for Fnv1a<N>
where
    Hot: UContainerFor<{ width_u8(N) }>,
    [(); 1 / (width_le_64(N) as usize)]: ,
{
    fn update(&mut self, bytes: &[u8]) { /* SRC */ }
    fn finalize(self) -> Bits<{ width_u8(N) }, Hot> { /* SRC */ }
}
```

The `[(); 1 / (width_le_64(N) as usize)]:` const-eval guard divides
by zero (compile error) when `N > 64`. The `#[diagnostic::on_-
unimplemented]` attribute on `UContainerFor` (defined in arvo)
produces a human-readable strategy-pointing error if the bound is
violated. An out-of-bound use such as `Fnv1a::<{ width(96) }>::hash_-
const(b"x")` triggers the const-eval failure attached to the where-
clause site; the diagnostic message states the `<= 64` constraint
and points at the BACKLOG `Fnv1a128` entry.

`Fnv1a::hash_const` is the entry point for compile-time access-set
hash-membership validation downstream (hilavitkutin app-builder
formalisation, round 202604520000). Streaming `Hasher` impl covers
the runtime path.

## Usage

Downstream code parameterises over the trait; swapping algorithm is
a type-parameter change, not a call-site rewrite.

Streaming:

```rust
fn intern_chunks<H>(mut hasher: H, chunks: &[&[u8]]) -> ContentHash
where
    H: Hasher<{ width(28) }>,
{
    for chunk in chunks { hasher.update(chunk); }
    hasher.finalize()
}
```

One-shot:

```rust
fn intern<H>(hasher: H, bytes: &[u8]) -> ContentHash
where
    H: HasherExt<{ width(28) }>,
{
    hasher.hash(bytes)
}
```

To swap algorithm, change the type the caller supplies. No site
edits; the trait signatures stay put.

## Dependencies

- arvo (L0): provides `Bits<const N: u8, S>` (the underlying
  storage primitive), `Width` / `IBits` / `FBits` ConstParamTy
  newtypes, the `width(n)` / `width_u8(W)` / `width_le_64(W)`
  helpers, the strategy markers (`Hot`, `Warm`, `Cold`, `Precise`),
  and `UContainerFor` / `IContainerFor`.
- arvo-bits (L1): provides `HasBitWidth` / `BitAccess` /
  `BitSequence` / `BitLogic` trait bounds where the algorithm needs
  bit-level access on its outputs.

## What this crate does NOT contain

- Cryptographic hashes (SHA, BLAKE); out of scope entirely. A
  sibling arvo-crypto crate covers cryptographic needs if they
  surface.
- Hash widths beyond 64 bits in `Fnv1a`; tracked as `Fnv1a128` in
  BACKLOG.
- Other non-cryptographic algorithms (xxHash, CRC32); tracked in
  BACKLOG.
- Semantic aliases beyond ContentHash; consumers that need
  NameHash, KeyHash, or similar start with their own local alias
  over ContentHash or `Bits<N, Hot>` and promote to arvo-hash once
  patterns stabilise.
- rkyv Archive impls; consumers define Archive at their boundary.
- Owned byte buffers; all hashing takes `&[u8]` input.

## Design rule

`#![no_std]`, no alloc. Trait-first: the contract is generic over
`Width`, the concrete algorithm (`Fnv1a`) is one implementor, more
land as additional implementors. Consumer code parameterises over
the trait, not the algorithm.
