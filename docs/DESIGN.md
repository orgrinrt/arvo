# arvo — Architecture Design

> Numeric primitives + analysis algorithms with Hot/Warm/Cold/Precise
> strategy markers. `#![no_std]`, no alloc. Zero dependency on
> platform. Extracted from polka-dots/hilavitkutin consolidation work
> so all consumers (hilavitkutin, loimu, saalis, clause, polka-dots)
> share one substrate.

## Principles

1. **Strategy-parameterised primitives** — numeric containers declared
   at type level as Hot/Warm/Cold/Precise, driving storage width +
   codegen choices without runtime dispatch.
2. **no_std, no alloc** — entire substrate is const-sized. Consumers
   pre-allocate; arvo never calls into an allocator.
3. **Trait contracts, not concrete types** — sub-crates generic over
   numeric trait bounds so downstream code picks its own concrete
   instantiations.
4. **Monomorphisation is the dispatch** — no dyn, no TypeId. The
   monomorphised function IS the type identity.

## Consumers

arvo is designed to be consumed by:

- **hilavitkutin** — pipeline execution engine (primary consumer;
  uses arvo-graph for plan-stage DAG analysis, arvo-bitmask for
  AccessMask, arvo-sparse for RCM reordering, arvo-spectral for
  trunk formation, arvo-comb for fiber grouping).
- **clause** — language toolchain (uses arvo-graph for pass DAG,
  arvo-bitmask for artifact read/write sets).
- **loimu, polka-dots, saalis** — miscellaneous numeric needs plus
  whatever hilavitkutin transitively exposes.
