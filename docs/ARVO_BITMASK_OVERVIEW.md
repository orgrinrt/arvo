# arvo-bitmask — Design

Fixed-width bitmask types and bit-matrix adjacency structures.
Cross-cutting foundation for arvo-graph, arvo-sparse, arvo-comb,
and any consumer that needs set-level adjacency over bit storage.
`#![no_std]`, no alloc, const-generic sizing.

Depends on arvo-bits for bit-level contracts (BitWidth, BitAccess,
BitSequence). Does NOT depend on arvo directly — bitmasks are not
numeric types. They do bitwise operations (OR, AND, popcount), not
arithmetic (Add, Mul, Sqrt). The type families are fundamentally
different.

## NodeId

```rust
#[repr(transparent)]
pub struct NodeId(pub USize);
```

Node index newtype. Wraps `USize` for type safety — prevents mixing
node indices with bit positions, column indices, or other counts.
Used by `BitMatrix` and all arvo-graph algorithms. Zero cost after
compilation.

## Bitmask types

`Mask<W>` — const-generic fixed-width bitmask, generic over a single
bit-bearing word type `W`. `W` must implement the arvo-bits
`BitSequence + BitAccess` contracts. The bitmask width is derived
from `W::WIDTH`.

Concrete aliases we ship:

- `Mask64` = `Mask<QWord<Hot>>`. Payload is `QWord<Hot>` =
  `UFixed<{IBits::SIXTYFOUR}, {FBits::ZERO}, Hot>`, backed by `u64`.
  Covers up to 64 elements. One word, single-instruction set-ops.
- `Mask256` = a fixed `[QWord<Hot>; 4]` backing. Covers up to 256
  elements. Four 64-bit words with unrolled loop-free set-ops.

The `Mask<W>` surface stays public and generic so consumers can
substitute a narrower width (e.g. `DWord<Hot>` for a 32-bit
bitmask) by implementing `BitSequence` + `BitAccess` for their own
type — but `Mask64` / `Mask256` are the two we ship and test.

All operations are single-instruction bitwise on the backing type.
No branches, no loops for Mask64. Mask256 uses a fixed `[QWord<Hot>;
4]` array with unrolled operations.

## Set operations

All operations are `const fn` where possible:

```rust
fn union(self, other: Self) -> Self        // bitwise OR
fn intersection(self, other: Self) -> Self // bitwise AND
fn difference(self, other: Self) -> Self   // self & !other
fn complement(self) -> Self                // bitwise NOT
fn is_empty(self) -> Bool                  // == 0
fn intersects(self, other: Self) -> Bool   // (self & other) != 0
fn contains(self, pos: USize) -> Bool      // test bit at position
fn insert(&mut self, pos: USize)           // set bit at position
fn remove(&mut self, pos: USize)           // clear bit at position
fn count(self) -> USize                    // popcount via BitSequence::count_ones
```

## Bit scanning

Iterate over set bits efficiently, built on arvo-bits contracts:

```rust
fn lowest_set(self) -> USize    // BitSequence::trailing_zeros
fn highest_set(self) -> USize   // BitSequence::leading_zeros then subtract
fn iter_set_bits(self) -> impl Iterator<Item = USize>
```

`iter_set_bits` implemented via `trailing_zeros` + clear lowest bit
loop. Zero allocation, compiles to `rbit`+`clz` on aarch64, `tzcnt`
on x86.

## Bit-matrix

`BitMatrix64<const N: Cap>` — adjacency matrix for up to N nodes
with N ≤ 64, stored as `[Mask64; cap_size(N)]`.
`BitMatrix256<const N: Cap>` — the 256-wide variant, stored as
`[Mask256; cap_size(N)]`. Row `i` has bit `j` set if edge `i -> j`
exists. `Cap` is arvo's const-generic capacity newtype;
`cap_size(c: Cap) -> usize` is a local `const fn` that unwraps the
newtype for array sizing (inline `.0.0` is rejected by nightly
`generic_const_exprs`).

```rust
fn edge(i: NodeId, j: NodeId) -> Bool     // BitAccess::bit, one instruction
fn set_edge(i: NodeId, j: NodeId)         // BitAccess::with_bit_set
fn successors(i: NodeId) -> Mask          // row i (all outgoing edges)
fn predecessors(j: NodeId) -> Mask        // scan column j across all rows
fn transitive_closure(&mut self)          // Warshall's algorithm on bitmasks
```

Two concrete types rather than one generic `BitMatrix<W, N>` — the
`Mask64` and `Mask256` backings differ enough at the storage layer
that abstracting over them via a trait fights `generic_const_exprs`
on current nightly (same trade-off as arvo-sparse's `_64` / `_256`
algorithm pair).

## Dirty propagation

```rust
fn propagate_dirty_64<const N: Cap>(dag: &BitMatrix64<N>, dirty: &mut Mask64)
fn propagate_dirty_256<const N: Cap>(dag: &BitMatrix256<N>, dirty: &mut Mask256)
```

OR sequence through DAG adjacency. For each set bit in `dirty`, OR
in all its successors. Propagates "changed" status through the
dependency graph. One AND per node. For 50 nodes on Mask64: 50 AND
operations, ~50ns.

Generic primitive — any consumer that needs to propagate a change
flag through DAG adjacency can use it.

## Consumers

- **arvo-graph**: bit-matrix DAG representation for topo / rank / waist
- **arvo-sparse**: sparse matrix bitmask operations (RCM, D-M, block)
- **arvo-comb**: constraint bitmasks for feasibility predicates

Downstream consumers (pipeline engines, compilers) use these through
the algorithm crates above, or directly for access-set / dirty-bit
tracking.
