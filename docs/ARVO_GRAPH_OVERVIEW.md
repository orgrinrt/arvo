# arvo-graph — Design

DAG and graph algorithms on const-generic fixed-size structures.
`#![no_std]`, no alloc. All algorithms operate on arvo-bitmask
`BitMatrix64` and `Mask64` types. Node indices use `NodeId` from
arvo-bitmask.

## Capacity

Algorithms ship against `BitMatrix64` — up to 64 nodes per adjacency
row, covering the common case per polka-dots research. `BitMatrix256`
variants (256 nodes) share the algorithmic shape; only the bitmask
backing changes. They are tracked in `BACKLOG.md.tmpl` and can be
added by a follow-up round without a design change.

`const N: Cap` carries arvo's capacity newtype on every public
signature. Array storage inside bodies unwraps via `cap_size(c: Cap)
-> usize` (canonical home: arvo-tensor; arvo-bitmask re-exports, and
arvo-graph reaches it via `arvo_bitmask::cap_size`). Array return
types in signatures below are shown as `[X; N]` for readability; the
compiled form is `[X; cap_size(N)]` with the implied `where [();
cap_size(N)]:` bound.

Generic over numeric trait bounds for weight types. Does not import
UFixed or IFixed directly. Consumer instantiates with concrete arvo
types; this crate is library code.

## Weight type contract

```rust
fn upward_rank<const N: Cap, W>(
    dag: &BitMatrix64<N>,
    weights: &[W; N],
) -> [W; N]
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant
```

The algorithm needs addition (accumulate path weights), ordering
(find maximum), copy (store in arrays), and a zero value (initialise).
It does not need multiplication, division, or square root.

If W is `UFixed<{IBits(16)}, {FBits(0)}, Hot>`, rank accumulation
wraps on overflow. If the same type with `Precise`, it saturates.
The algorithm is identical. The consumer chooses the safety level by
choosing W.

## Topological sort

Kahn's algorithm on BitMatrix adjacency:

```rust
fn topo_sort<const N: Cap>(
    dag: &BitMatrix64<N>,
) -> (USize, [NodeId; N])  // (valid_count, topo_order)
```

Fixed `[NodeId; N]` queue + `[USize; N]` in-degree array. Zero heap.
Returns topo-ordered node IDs and the count of successfully sorted
nodes. Cycle detection built-in: `valid_count < N` means a cycle
was detected. `valid_count` tells you exactly how many nodes were
sorted before the cycle was hit.

## Node renumbering

```rust
fn renumber<const N: Cap>(
    topo_order: &[NodeId; N],
) -> [NodeId; N]
```

After topo sort, renumber all node IDs so they match reverse
topological order. Upward rank traversal then accesses the rank
array sequentially — the hardware prefetcher sees a sequential
scan. Same principle as column co-location: make access patterns
sequential for the prefetcher.

## Rank computation

```rust
fn upward_rank<const N: Cap, W>(
    dag: &BitMatrix64<N>,
    weights: &[W; N],
) -> [W; N]
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant

fn downward_rank<const N: Cap, W>(
    dag: &BitMatrix64<N>,
    weights: &[W; N],
) -> [W; N]
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant
```

`rank(u) = cost(u) + max(rank(successors(u)))`. Bottom-up in reverse
topo order. Cost per node is a generic weight type W satisfying the
trait bounds above. Identifies the critical path: the node with the
highest rank is the trunk root.

Upward rank: longest path from any node to any leaf.
Downward rank: longest path from any root to any node.
Both computed in one pass each over the topo-ordered nodes.

## Connected components

```rust
fn components<const N: Cap>(
    dag: &BitMatrix64<N>,
) -> [USize; N]
```

DFS on BitMatrix adjacency using a `Mask` as the visited set. Stack
is a fixed `[NodeId; N]` array. Returns component ID (as USize) per
node. Two nodes are in the same component if their access masks
intersect (directly or transitively).

Common application: feeding phase identification by pairing
component membership with block-diagonal detection results.

## Longest/shortest weighted path

```rust
fn longest_path<const N: Cap, W>(
    dag: &BitMatrix64<N>,
    weights: &[W; N],
    topo_order: &[NodeId; N],
) -> (W, Mask, [NodeId; N])  // (path_weight, has_predecessor, predecessors)
where
    W: Add<Output = W> + TotalOrd + Copy + FromConstant
```

Weighted path computation on DAG (no cycles). Uses topo order for
single-pass O(V+E). Returns the path weight, a validity Mask, and
the predecessor array. Roots have their bit unset in the Mask. Only
read `predecessors[i]` when the Mask bit is set.

## Critical path analysis

The critical path is the longest weighted path from any root to any
leaf. Computed via upward rank — the path following the highest-ranked
successor at each step. Identifies the serial bottleneck in a DAG.

The weight type is consumer-chosen; a common pattern is summing
`size_of` of the data each node touches, which approximates
per-element compute cost in a pipeline setting.

## Waist detection

```rust
fn waist_detect<const N: Cap>(
    dag: &BitMatrix64<N>,
    topo_order: &[NodeId; N],
) -> Mask
```

Walk topo order. At each level (set of nodes at the same topo depth),
count how many independent paths are alive. Where the count drops to
a local minimum = waist = phase boundary. Returns a Mask with bits
set at waist positions in the topo order.

Different from connected components: connected components find fully
independent sub-problems. Waist detection finds narrow bottleneck
points WITHIN a connected DAG where all paths converge through a few
nodes before fanning out again.

Common application: phase decomposition — the waist points become
natural phase boundaries in a staged pipeline.

## Spanning tree decomposition

```rust
pub struct SpanningTree<const N: Cap> {
    pub on_trunk: Mask,               // which nodes are on the trunk
    pub trunk_next: [NodeId; N],      // next node on trunk (valid where on_trunk is set)
    pub branch_roots: Mask,           // nodes that start branches
    pub bridges: Mask,                // fan-in nodes (mandatory fiber breaks)
}

fn spanning_tree<const N: Cap, W>(
    dag: &BitMatrix64<N>,
    ranks: &[W; N],
) -> SpanningTree<N>
where
    W: TotalOrd + Copy
```

Decompose a DAG into a spanning tree with trunk (critical path) and
branches (side paths joining at fan-in points).

1. Trunk = path following highest-ranked successor from root to leaf
2. At fan-out: trunk follows heaviest child, others become branches
3. At fan-in: record bridge (node reading from multiple trunks)
4. Recurse: each branch is a sub-DAG

Natural barriers (fan-in points) are mandatory grouping-break points
for any consumer that batches along trunks.
