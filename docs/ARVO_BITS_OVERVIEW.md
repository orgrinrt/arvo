# arvo-bits — Design

Bit-level primitive crate. Sits between arvo (numeric primitives) and
arvo-bitmask (fixed-width mask types). Ships two kinds of surface:

- **Contracts** — traits for bit-width introspection (`HasBitWidth`),
  individual bit manipulation (`BitAccess`), bit-sequence scanning and
  popcount (`BitSequence`), and whole-word bitwise logic (`BitLogic`),
  with concrete blanket impls so all arvo L0 primitives are
  bit-accessible.

Post round 202604500000, arvo-bits is a contracts-only crate. The
`Bits<const N: u8, S>` storage primitive, the bit-size aliases
(`Bit`, `Nibble`, `Byte`, `Word`, `DWord`, `QWord`), and the
`Bitfield` primitive (with its `bitfield!` macro) all live in arvo
L0. Consumers that previously imported `arvo_bits::Bits`,
`arvo_bits::Byte`, `arvo_bits::bitfield!`, etc., update to `arvo::*`.
Mask concretes (`Mask64`, `Mask256`, `BitMatrix`) still belong in L2
arvo-bitmask.

`#![no_std]`, no alloc.

Depends on arvo. No other arvo-family dependencies.

## Why this crate exists

arvo defines numeric types (UFixed, IFixed, Float). arvo-bitmask
defines bit-level types (Mask64, Mask256, BitMatrix). These are
fundamentally different type families: numerics do Add/Mul/Sqrt,
bitmasks do OR/AND/popcount. The original structure had arvo-bitmask
depending directly on arvo, but bitmasks are not numeric — the
dependency existed only for the arvo-types-only lint, not for any
numeric operation.

arvo-bits is the contract layer. It defines what it means for a type
to be "a sequence of bits" without requiring numeric semantics. Both
arvo numerics and arvo-bitmask types implement these contracts, but
the contract itself is independent of either.

## Const generic bit-count types

arvo defines `IBits`, `FBits`, and `Width` as ConstParamTy newtypes
wrapping `Bits<7, Hot>` (the L0 storage primitive at 7 bits). These
appear in trait signatures here. See arvo's DESIGN for the full
definitions.

Requires `adt_const_params` (nightly). These ensure consumers cannot
accidentally swap integer and fractional bit counts at type
definition sites. Consumer code typically defines its own semantic
aliases over `UFixed` / `IFixed` and rarely touches `IBits` / `FBits`
directly.

## Traits

### HasBitWidth

How many logical bits this type occupies:

```rust
pub trait HasBitWidth {
    const WIDTH: USize;
}
```

For arvo types: `WIDTH = I + F` (UFixed) or `WIDTH = 1 + I + F`
(IFixed). The width is the logical bit count, not the container
size. Container size depends on the strategy marker and may be
larger. `USize` is arvo's newtype for `usize`.

### BitAccess

Individual bit manipulation:

```rust
pub trait BitAccess: HasBitWidth + Copy {
    fn bit(self, idx: USize) -> Bool;
    fn with_bit_set(self, idx: USize) -> Self;
    fn with_bit_cleared(self, idx: USize) -> Self;
    fn with_bit_toggled(self, idx: USize) -> Self;
}
```

All methods take `self` and return `Self` (functional style, no
mutation). `idx` is from LSB (bit 0 = least significant). Operates
on the raw bit representation of the backing container, not the
numeric value. `Bool` is arvo's boolean newtype (see below).

### BitSequence

Scanning and counting, mapping directly to hardware intrinsics:

```rust
pub trait BitSequence: HasBitWidth + Copy {
    fn trailing_zeros(self) -> USize;
    fn leading_zeros(self) -> USize;
    fn count_ones(self) -> USize;
    fn count_zeros(self) -> USize;
    fn is_zero(self) -> Bool;
}
```

Hardware mapping:
- `trailing_zeros` -> `cttz` (x86: `tzcnt`, aarch64: `rbit` + `clz`)
- `leading_zeros` -> `ctlz` (x86: `lzcnt`, aarch64: `clz`)
- `count_ones` -> `popcnt` (x86: `popcnt`, aarch64: hardware popcount)

These are the primitives that arvo-bitmask's `iter_set_bits`,
`lowest_set`, `highest_set` are built on.

### BitLogic

Whole-word bitwise logic. Single-instruction on the backing
primitive, loop-free:

```rust
pub trait BitLogic: HasBitWidth + Copy {
    fn bitor(self, other: Self) -> Self;
    fn bitand(self, other: Self) -> Self;
    fn bitnot(self) -> Self;
    fn bitxor(self, other: Self) -> Self;
    fn clear_lowest_set_bit(self) -> Self
    where
        Self: BitAccess + BitSequence;
}
```

`BitLogic` covers the four bitwise operations that map directly to
hardware instructions (`or` / `and` / `not` / `xor` on x86; identical
on aarch64). `BitSequence` covers SCAN / POPCOUNT; `BitAccess` covers
SINGLE-BIT read/write; `BitLogic` covers WHOLE-WORD logic. The three
are independent contracts that together describe a bit-bearing
container.

Concrete impls live on each `UFixed<I, F, Hot>` container family
(`Bit`, `Nibble`, `Byte`, `Word`, `DWord`, `QWord`) — each delegates
once via `UFixed::to_raw` / `from_raw` to the backing primitive's
core operator. Other strategies (`Warm`, `Precise`, `Cold`) do not
impl `BitLogic`: their wider containers make whole-word ops
meaningless at the logical-width level. `BitLogic` is a Hot-only
surface.

The `clear_lowest_set_bit` method has a default implementation on
`BitLogic` that uses `BitSequence::trailing_zeros` +
`BitAccess::with_bit_cleared`. Concrete types override with
`self.to_raw() & (self.to_raw() - 1)` (one instruction via x86 `BLSR`)
for the fast path. Exposed primarily for set-bit iterators that scan
and clear the lowest set bit at each step.

In addition to the trait methods, arvo-bits ships `impl core::ops::BitOr
/ BitAnd / BitNot / BitXor for UFixed<I, F, Hot>` (blanket on
`T: BitLogic` if orphan rules permit; otherwise per-type). Consumers can
use `a | b`, `a & b`, `!a`, `a ^ b` syntax. Implementing core TRAITS
(contracts) on arvo's own types is distinct from using core CONCRETE
types in signatures — the #115 "freedom from core" direction targets
the latter, not the former.

## Bool and Bit

Two types, same physical width, different semantic domains:

**`Bit`** = `UFixed<{IBits(1)}, {FBits(0)}, Hot>` — a 1-bit data
value. Stored in columns. Participates in bit operations. Backed
by u8 (Hot strategy, minimum container).

**`Bool`** = `Bool(bool)` — a newtype over Rust's `bool`. Returned
by predicates (`is_zero`, `bit`). Used in control flow.

`Bit` is data. `Bool` is control flow. The boundary between them is
explicit conversion.

### Bool in control flow (Try)

`Bool` implements `Try` with `Residual = Infallible`:

```rust
pub struct Bool(bool);

impl Try for Bool {
    type Output = bool;
    type Residual = Infallible;

    fn branch(self) -> ControlFlow<Infallible, bool> {
        ControlFlow::Continue(self.0)
    }
    fn from_output(output: bool) -> Self { Bool(output) }
}
```

This enables `?` to extract the inner `bool` for use in `if`:

```rust
fn execute(&self, ctx: Context<...>) -> Outcome {
    if val.is_zero()? { ... }
    Outcome::done()
}
```

The `?` always takes the `Continue` path (Infallible residual is
uninhabited). Zero runtime cost. Works in any function returning a
type that impls `FromResidual<Infallible>` — which includes
hilavitkutin-api's `Outcome`, `Just`, `Maybe`, and `Result`.

### AsBool trait

Cold-path bridge and compatibility escape hatch:

```rust
pub trait AsBool {
    fn as_bool(&self) -> bool;
}

impl AsBool for Bool { ... }
```

For code paths outside WU execution, or for libraries that need to
expose Rust-native `bool` on their public API. Day-one the internal
codebase uses `?`; `AsBool` exists for boundary compatibility.

## Type family marker traits

arvo defines marker traits for its type families. arvo-bits uses
these as reference points (for documentation of what types are
covered), not as blanket-impl bounds:

- `IntegerLike` — F=0, pure integer semantics
- `FractionLike` — F>0, has fractional part
- `BitPresentation` — raw bits are accessible (all UFixed/IFixed)
- `FloatLike` — FastFloat/StrictFloat

## Implementation strategy

`HasBitWidth` / `BitAccess` / `BitSequence` are implemented as **concrete
impls on `UFixed<I, F, S>` and `IFixed<I, F, S>`**, not as blanket
impls over a `T: BitPresentation` bound. Reason: the L0
`BitPresentation` marker exposes only `LOGICAL_WIDTH` — there is no
method for arvo-bits to reach the underlying container. Keeping the
blanket-impl language would require extending L0 (closed round).
Concrete impls over UFixed/IFixed from arvo-bits stay within the
orphan-rule boundary (trait is local to arvo-bits, types are
foreign — local-trait-foreign-type impls are permitted) and
delegate bit-level ops through `UFixed::to_raw()` / `from_raw()` and
the equivalents on `IFixed`.

A sealed `BitPrim` trait (internal to arvo-bits) is implemented for
`u8` / `u16` / `u32` / `u64`, carrying the bit-manipulation surface:
`count_ones`, `trailing_zeros`, `leading_zeros`, `bit`,
`with_bit_set`, `with_bit_cleared`, `with_bit_toggled`. An
`IBitPrim` sibling covers `i8` / `i16` / `i32` / `i64`. The concrete
`impl BitAccess for UFixed<I, F, S>` delegates via
`UFixed::to_raw()` to the primitive's `BitPrim` method, then
`UFixed::from_raw(...)` wraps the result.

`HasBitWidth::WIDTH` for UFixed = `I.0 + F.0`, for IFixed = `1 + I.0 +
F.0`. Both IntegerLike (F=0) and FractionLike (F>0) types get bit
access by this route — the F>0 case enables using any UFixed as a
packed flag field (`UFixed<{IBits::ZERO}, {FBits(8)}, Hot>` treated
as 8 individual flag bits).


## What this crate does NOT contain

- Bitmask types (Mask64, Mask256, BitMatrix); live in arvo-bitmask.
- Numeric operations (Add, Mul, Sqrt); live in arvo.
- Set operations (union, intersection); live in arvo-bitmask.
- Float types; live in arvo.
- Fallible-return types (Outcome / Just / Maybe); consumer concern.
- Arithmetic fielded types (UFixed, IFixed); live in L0 arvo.
- Opaque-bit storage primitive (Bits) and packed-layout primitive
  (Bitfield); live in L0 arvo since round 202604500000. arvo-bits
  is contracts-only post-migration.

## Consumers

- **arvo-bitmask**: builds Mask types on HasBitWidth + BitAccess +
  BitSequence + BitLogic contracts. Mask64's `count()` delegates to
  `BitSequence::count_ones`; `union` / `intersection` / `difference` /
  `complement` delegate to `BitLogic::{bitor, bitand, bitxor, bitnot}`
  (or equivalently via `core::ops`). Bit scanning uses `trailing_zeros`
  / `leading_zeros`. No reach-through to raw `u64`.
- Other arvo algorithm crates (graph, sparse, comb, spectral) depend
  on arvo-bitmask, not arvo-bits, for adjacency and set work.
