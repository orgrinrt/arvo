# arvo-bits â€” Design

Bit-level contract crate. Sits between arvo (numeric primitives) and
arvo-bitmask (fixed-width mask types). Defines traits for bit-width
introspection, individual bit manipulation, and bit-sequence scanning.
Provides blanket impls so all arvo fixed-point types are bit-accessible.
`#![no_std]`, no alloc.

Depends on arvo. No other arvo-family dependencies.

## Why this crate exists

arvo defines numeric types (UFixed, IFixed, Float). arvo-bitmask
defines bit-level types (Mask64, Mask256, BitMatrix). These are
fundamentally different type families: numerics do Add/Mul/Sqrt,
bitmasks do OR/AND/popcount. The original structure had arvo-bitmask
depending directly on arvo, but bitmasks are not numeric â€” the
dependency existed only for the arvo-types-only lint, not for any
numeric operation.

arvo-bits is the contract layer. It defines what it means for a type
to be "a sequence of bits" without requiring numeric semantics. Both
arvo numerics and arvo-bitmask types implement these contracts, but
the contract itself is independent of either.

## Const generic bit-count types

arvo uses typed const generics for fixed-point bit counts. These are
defined in arvo and re-exported here for use in trait signatures:

```rust
#[derive(ConstParamTy, PartialEq, Eq)]
pub struct IBits(pub u8);  // integer bit count

#[derive(ConstParamTy, PartialEq, Eq)]
pub struct FBits(pub u8);  // fractional bit count
```

Requires `adt_const_params` (nightly). These ensure you cannot
accidentally swap integer and fractional bit counts at type
definition sites. Consumer code uses semantic aliases (Angle, Count,
etc.) and never sees IBits/FBits directly.

## Traits

### BitWidth

How many logical bits this type occupies:

```rust
pub trait BitWidth {
    const WIDTH: USize;
}
```

For arvo types: `WIDTH = I + F` (UFixed) or `WIDTH = 1 + I + F`
(IFixed). The width is the logical bit count, not the container
size. Container size depends on the strategy marker and may be
larger. `USize` is arvo's newtype for `usize`.

### BitAccess

Individual bit manipulation:

```rust
pub trait BitAccess: BitWidth + Copy {
    fn bit(self, idx: USize) -> Bool;
    fn with_bit_set(self, idx: USize) -> Self;
    fn with_bit_cleared(self, idx: USize) -> Self;
    fn with_bit_toggled(self, idx: USize) -> Self;
}
```

All methods take `self` and return `Self` (functional style, no
mutation). `idx` is from LSB (bit 0 = least significant). Operates
on the raw bit representation of the backing container, not the
numeric value. `Bool` is arvo's boolean newtype (see below).

### BitSequence

Scanning and counting, mapping directly to hardware intrinsics:

```rust
pub trait BitSequence: BitWidth + Copy {
    fn trailing_zeros(self) -> USize;
    fn leading_zeros(self) -> USize;
    fn count_ones(self) -> USize;
    fn count_zeros(self) -> USize;
    fn is_zero(self) -> Bool;
}
```

Hardware mapping:
- `trailing_zeros` -> `cttz` (x86: `tzcnt`, aarch64: `rbit` + `clz`)
- `leading_zeros` -> `ctlz` (x86: `lzcnt`, aarch64: `clz`)
- `count_ones` -> `popcnt` (x86: `popcnt`, aarch64: hardware popcount)

These are the primitives that arvo-bitmask's `iter_set_bits`,
`lowest_set`, `highest_set` are built on.

## Bool and Bit

Two types, same physical width, different semantic domains:

**`Bit`** = `UFixed<{IBits(1)}, {FBits(0)}, Hot>` â€” a 1-bit data
value. Stored in columns. Participates in bit operations. Backed
by u8 (Hot strategy, minimum container).

**`Bool`** = `Bool(bool)` â€” a newtype over Rust's `bool`. Returned
by predicates (`is_zero`, `bit`). Used in control flow.

`Bit` is data. `Bool` is control flow. The boundary between them is
explicit conversion.

### Bool in control flow (Try)

`Bool` implements `Try` with `Residual = Infallible`:

```rust
pub struct Bool(bool);

impl Try for Bool {
    type Output = bool;
    type Residual = Infallible;

    fn branch(self) -> ControlFlow<Infallible, bool> {
        ControlFlow::Continue(self.0)
    }
    fn from_output(output: bool) -> Self { Bool(output) }
}
```

This enables `?` to extract the inner `bool` for use in `if`:

```rust
fn execute(&self, ctx: Context<...>) -> Outcome {
    if val.is_zero()? { ... }
    Outcome::done()
}
```

The `?` always takes the `Continue` path (Infallible residual is
uninhabited). Zero runtime cost. Works in any function returning a
type that impls `FromResidual<Infallible>` â€” which includes
hilavitkutin-api's `Outcome`, `Just`, `Maybe`, and `Result`.

### AsBool trait

Cold-path bridge and compatibility escape hatch:

```rust
pub trait AsBool {
    fn as_bool(&self) -> bool;
}

impl AsBool for Bool { ... }
```

For code paths outside WU execution, or for libraries that need to
expose Rust-native `bool` on their public API. Day-one the internal
codebase uses `?`; `AsBool` exists for boundary compatibility.

## Type family marker traits

arvo defines marker traits for its type families. arvo-bits uses
these as bounds in blanket impls:

- `IntegerLike` â€” F=0, pure integer semantics
- `FractionLike` â€” F>0, has fractional part
- `BitPresentation` â€” raw bits are accessible (all UFixed/IFixed)
- `FloatLike` â€” FastFloat/StrictFloat

These are defined and blanket-impl'd in arvo. arvo-bits consumes
them:

```rust
impl<T: BitPresentation> BitAccess for T { ... }
impl<T: BitPresentation> BitSequence for T { ... }
impl<T: BitPresentation> BitWidth for T {
    const WIDTH: USize = T::LOGICAL_WIDTH;
}
```

`BitPresentation` provides `LOGICAL_WIDTH` (the I+F or 1+I+F bit
count). `BitWidth::WIDTH` delegates to it. Both `IntegerLike` (F=0)
and `FractionLike` (F>0) types get bit access â€” the F>0 case
enables using any UFixed as a packed flag field (`UFixed<{IBits(0)},
{FBits(8)}, Hot>` treated as 8 individual flag bits).

## Semantic aliases

Bit-size type aliases. Default strategy is Hot (bit-level work
typically wants minimum container and wrapping ops):

```rust
pub type Bit<S: Strategy = Hot>   = UFixed<{IBits(1)},  {FBits(0)}, S>;
pub type Nibble<S: Strategy = Hot> = UFixed<{IBits(4)},  {FBits(0)}, S>;
pub type Byte<S: Strategy = Hot>  = UFixed<{IBits(8)},  {FBits(0)}, S>;
pub type Word<S: Strategy = Hot>  = UFixed<{IBits(16)}, {FBits(0)}, S>;
pub type DWord<S: Strategy = Hot> = UFixed<{IBits(32)}, {FBits(0)}, S>;
pub type QWord<S: Strategy = Hot> = UFixed<{IBits(64)}, {FBits(0)}, S>;
```

All integer-width (F=0). `Nibble` = 4-bit value (half-byte).
`DWord` = 32 bits (double word). `QWord` = 64 bits (quad word,
our maximum logical width). These are x86 naming convention where
word = 16 bits.

No SIMD-register-width alias: a 128-bit type would exceed our
I+F <= 64 constraint. SIMD register width is a strategy concern
(how many lanes of your type fit in a register), not a type alias.

## What this crate does NOT contain

- Bitmask types (Mask64, Mask256, BitMatrix) â€” arvo-bitmask
- Numeric operations (Add, Mul, Sqrt) â€” arvo
- Set operations (union, intersection) â€” arvo-bitmask
- Float types â€” arvo, completely separate from bit-level
- Outcome/Just/Maybe/Result types â€” hilavitkutin-api

## Consumers

- **arvo-bitmask**: builds Mask types on BitSequence/BitAccess
  contracts. Mask64's `count()` delegates to `BitSequence::count_ones`.
  Bit scanning uses `trailing_zeros`/`leading_zeros`.
- **hilavitkutin**: indirect via arvo-bitmask. Does not depend on
  arvo-bits directly.
- **arvo sub-crates** (graph, sparse, spectral, comb): indirect via
  arvo-bitmask. Do not depend on arvo-bits directly.
